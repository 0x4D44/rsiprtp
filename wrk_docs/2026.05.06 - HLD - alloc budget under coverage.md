# HLD — Alloc budget under coverage instrumentation

**Date:** 2026-05-06
**Author:** Claude (with Arthur)
**Status:** Draft for review
**Related:**
- `crates/rsiprtp/tests/allocations_sip_parse.rs` — the test under repair
- `wrk_journals/2026.05.06 - JRN - code review LLVM_PROFILE_FILE skip.md` — review that prompted this follow-up
- `tools/full_test/src/main.rs` — the 7-stage test bar that runs `cargo llvm-cov` as stage 7

## Context

`crates/rsiprtp/tests/allocations_sip_parse.rs` locks in the per-INVITE
allocation count for four fixtures. It uses `stats_alloc::INSTRUMENTED_SYSTEM`
as the global allocator (scoped to this test binary only) and a single
`#[test]` function so cargo's parallel scheduler cannot poison the
process-wide counters.

Today, lines 191-201 silently skip the assertion when `LLVM_PROFILE_FILE`
is present in the environment:

```rust
if std::env::var_os("LLVM_PROFILE_FILE").is_some() {
    eprintln!("skipping allocation budget test under llvm-cov instrumentation");
    return;
}
```

This was a defensible patch when the alternative was a budget that fired
spuriously under `cargo llvm-cov`. But it has turned a regression oracle
into a no-op in the most-trafficked code path:

`cargo run --release -p full_test` (the 7-stage test bar used in CI and
local validation) runs `cargo llvm-cov` as **stage 7**, which sets
`LLVM_PROFILE_FILE` for every spawned test process, which means the
assertion is **always** skipped under the test bar. So the regression
check is effectively dead in the path that matters most. (`cargo test
--workspace` outside the bar still exercises it — but that path is
exercised less often, especially in CI, and gives no signal at the
moment of the bar where regressions are caught.)

The reviewer's recommended fix is a two-tier assertion: capture an
under-coverage baseline per fixture, add parallel `*_BUDGET_UNDER_COVERAGE`
constants, and replace the skip with mode-aware assertions so the oracle
fires under both modes.

## Goals

- Make the alloc-budget assertion fire under both `cargo test` and
  `cargo llvm-cov` runs.
- Preserve the no-coverage budget tightness (current 1.15x headroom on
  measured + 4 floor).
- Keep the test a single `#[test]` (process-wide counters require it).
- Leave a refresh procedure in the docstring that future maintainers
  cannot misread.
- Make the under-coverage path visibly *do* something on success — no
  silent skip semantics.

## Non-goals

- Achieve identical budgets in both modes. Coverage instrumentation
  fundamentally changes the allocation profile (~5x); the under-coverage
  budget will be looser than the no-coverage budget by construction.
- Detect coverage-tool-version regressions in the parser. If
  `cargo-llvm-cov` ships a new instrumentation strategy that changes
  alloc counts, the under-coverage budget may need re-baselining. We
  document this in the risk register and accept it.
- Catch *bytes-allocated* regressions. We continue to assert only on
  `Stats::allocations` (count). Bytes are too brittle to lock in.
- Replace the existing test architecture (single `#[test]`, fixture
  array, `Region::change()` measurement). Those decisions were made
  with care and remain correct.

## Design

### 1. Mode detection — what signals "running under coverage"?

Three candidate signals were considered.

**Option A — `LLVM_PROFILE_FILE` env var presence (current approach).**
Set by `cargo-llvm-cov` for every test process it spawns; also set by any
other LLVM coverage runner (e.g. `cargo-pgo`, raw `RUSTFLAGS=-Cinstrument-coverage`).
Reads at runtime. Simple, no Cargo.toml or feature wiring.

Drawbacks:
- Stale env-var leakage: a developer who manually exports
  `LLVM_PROFILE_FILE` (e.g. left over from a prior coverage run) will
  silently fall into the looser budget. Probability is low; the
  worst case is "oracle slightly less tight in one shell session".
- Coupling to one specific coverage tool's environment contract. If
  cargo-llvm-cov ever stops setting it (unlikely), detection breaks.

**Option B — `#[cfg(coverage)]` / `cfg(coverage_nightly)`.** A compile-time
flag set by `cargo-llvm-cov` *only when invoked without
`--no-cfg-coverage`*. Reading `tools/full_test/src/main.rs:409` shows our
test bar invokes `cargo llvm-cov ... --no-cfg-coverage --json --
--test-threads=1` — meaning **the `cfg(coverage)` flag is explicitly
disabled in our pipeline**. So `cfg(coverage)` would be permanently
false in the path that matters and is therefore unusable as-is.

We could change the test bar to drop `--no-cfg-coverage`, but that flag
is there for a reason: it lets test code that has its own
`#[cfg(coverage)]` branches stay in-scope of coverage reporting. We
won't touch the test bar for this.

**Option C — runtime detection via `RUSTFLAGS` introspection or a
side-channel allocation probe.** Both are uglier than (A) and bring no
real reliability win. Rejected.

**Decision: Option A.** Keep the runtime `LLVM_PROFILE_FILE` check.
It's the only signal that's reliably set by our actual pipeline. The
stale-leakage risk is real but low-impact; we mitigate it with a
visible diagnostic line so a developer staring at "wait, why is this
loose budget" output understands what mode they're in. We do not add
defensive secondary checks (e.g. also sniffing `CARGO_LLVM_COV`) —
that's belt-and-braces over-engineering for a 9-line patch that already
works in the only pipeline we run.

We rename the helper:

```rust
fn under_coverage_instrumentation() -> bool {
    std::env::var_os("LLVM_PROFILE_FILE").is_some()
}
```

so the call site at the assertion reads as a question, not as a magic
env-var grovel.

### 2. Budget structure — two parallel constants per fixture

Naming: keep the existing `*_BUDGET` constants for the no-coverage mode;
add parallel `*_BUDGET_UNDER_COVERAGE` constants. Rationale: the
existing names are referenced in error messages (the "update {}" hint at
line 213); renaming them would break the ergonomics of refresh
diagnostics. Adding parallel constants is purely additive.

```rust
const TYPICAL_INVITE_ALLOC_BUDGET: usize = 49;
const TYPICAL_INVITE_ALLOC_BUDGET_UNDER_COVERAGE: usize = <to be measured>;

const BODY_INVITE_ALLOC_BUDGET: usize = 17;
const BODY_INVITE_ALLOC_BUDGET_UNDER_COVERAGE: usize = <to be measured>;

const COMPACT_INVITE_ALLOC_BUDGET: usize = 49;
const COMPACT_INVITE_ALLOC_BUDGET_UNDER_COVERAGE: usize = <to be measured>;

const LONGREQ_INVITE_ALLOC_BUDGET: usize = 258;
const LONGREQ_INVITE_ALLOC_BUDGET_UNDER_COVERAGE: usize = <to be measured>;
```

`BudgetCase` grows two fields:

```rust
struct BudgetCase {
    name: &'static str,
    fixture_path: &'static str,
    bytes: &'static [u8],
    budget: usize,
    budget_under_coverage: usize,
    constant_name: &'static str,
    constant_name_under_coverage: &'static str,
}
```

The assertion picks one or the other at the call site:

```rust
let under_cov = under_coverage_instrumentation();
for case in BUDGET_CASES {
    let stats = count_parse_allocs(case.bytes);
    let (budget, constant) = if under_cov {
        (case.budget_under_coverage, case.constant_name_under_coverage)
    } else {
        (case.budget, case.constant_name)
    };
    if stats.allocations > budget { /* push failure mentioning `constant` */ }
}
```

**Why per-fixture, not a global multiplier (`measured * 1.15`)?** Two
reasons. First, we already have a per-fixture formula in the no-coverage
budget (the "+4 floor or 1.15x, whichever is bigger"); using a flat
`measured * 1.15` for the under-coverage path would split the formula
between modes, which is harder to reason about. Second, our exploratory
expectation is that coverage-induced allocations don't scale linearly
with parser allocations — they scale with **basic-block coverage**.
Fixtures that hit cold paths (longreq's elaborate header parsing) will
take a different multiplier than fixtures that hit hot paths (a small
INVITE going through the same basic blocks four parsers have already
warmed up). A per-fixture baseline absorbs that variance directly.

We use the same headroom formula: `max(measured + 4, ceil(measured * 1.15))`,
applied to under-coverage measurements. Same docstring rationale (small
floor for tiny fixtures, 1.15x to absorb harmless refactors). Document
this in the budget block so future maintainers don't re-derive it.

### 3. Baseline capture — reproducibility and variance

The existing `discover_baselines` test prints per-fixture allocation
counts. To capture under-coverage baselines, we run it under
`cargo llvm-cov`:

```text
cargo llvm-cov test -p rsiprtp --test allocations_sip_parse \
    --no-cfg-coverage -- --ignored --nocapture discover_baselines
```

Note the `--no-cfg-coverage` flag mirrors the test-bar invocation at
`tools/full_test/src/main.rs:409` so we measure the *exact* configuration
that runs in CI, not a slightly different llvm-cov mode. Mismatching this
is a real risk: see the risk register.

**Variance.** Coverage instrumentation is *almost* deterministic — the
emitted instrumentation calls are pinned by the compiler, but counter
updates may go through global state that's affected by test-binary
process startup (lazy statics, thread-local init for libtest's harness,
etc.). The S1 implementation stage measures allocation counts across
**N=5 consecutive runs** and records:

- per-fixture min/max/median across runs,
- the spread (max - min).

If spread ≤ 1 across all fixtures, treat the median as "the measurement"
and apply the standard headroom formula. If spread > 1 on any fixture,
add `spread` to the headroom: `max(median + 4 + spread, ceil(median *
1.15))`. Document the observed spread alongside the constants:

```rust
/// Measured 2026-05-06 under coverage (median of N=5, spread=0):
/// 213 allocations. Budget 245 = ceil(213 * 1.15).
const TYPICAL_INVITE_ALLOC_BUDGET_UNDER_COVERAGE: usize = 245;
```

The "spread=0" annotation tells a future maintainer that this number
is reproducible to the allocation; if they see drift later, they know
to suspect cargo-llvm-cov / llvm-tools-preview version drift, not
parser drift.

### 4. Refresh procedure — two commands, one block

Update the module-level "Refreshing the budget" docstring section to
explain that **both** baselines must be refreshed when the parser is
intentionally re-tuned:

```text
## Refreshing the budgets

Both budgets must be refreshed together. Run:

  # No-coverage baseline:
  cargo test -p rsiprtp --test allocations_sip_parse \
      -- --ignored --nocapture discover_baselines

  # Under-coverage baseline (mirrors full_test stage 7 invocation):
  cargo llvm-cov test -p rsiprtp --test allocations_sip_parse \
      --no-cfg-coverage -- --ignored --nocapture discover_baselines

For each fixture, update both the *_BUDGET and *_BUDGET_UNDER_COVERAGE
constant. The under-coverage value is expected to be ~5x the no-coverage
value; if it isn't, suspect a coverage-tool change (see the risk
register in 2026.05.06 - HLD - alloc budget under coverage.md).
```

Two commands, same single `#[ignore]`d test, deterministic ordering.

A maintainer who only updates one of the two will get a sharp failure
on the next test bar run — either the no-coverage budget will stay too
loose (mild) or the under-coverage budget will fire spuriously (also
mild). Either way, "you forgot to refresh the other one" is the obvious
diagnosis from the failure message, which already names the
out-of-budget constant.

| Stage | Action | Output |
|-------|--------|--------|
| **S1** | Run `discover_baselines` under coverage 5 times; record per-fixture min/median/max and spread. | A short note in the implementation journal with the four numbers and the chosen budgets per the headroom formula. |
| **S2** | Add the four `*_BUDGET_UNDER_COVERAGE` constants with measured values + provenance comments. Extend `BudgetCase` with `budget_under_coverage` and `constant_name_under_coverage` fields. Wire the four entries in `BUDGET_CASES`. | Compiles. No assertion change yet. |
| **S3** | Replace the early-return skip with the two-tier assertion. Add the `under_coverage_instrumentation()` helper. Print a single diagnostic line via `eprintln!` showing which mode is active and which budget set is in effect. | Test fires under both modes. |
| **S4** | Update the module-level docstring: "Refreshing the budget" → "Refreshing the budgets" with the two-command recipe and the "expect ~5x ratio" diagnostic hint. Update the in-line comment block above the budget constants to note both modes. | Docstring carries the durable record. |
| **S5** | Verify both pipelines: `cargo test -p rsiprtp --test allocations_sip_parse` (no coverage) and `cargo run --release -p full_test` (full bar including stage 7 coverage). Both must pass. | Two green runs, captured in the journal. |

S1 is the only stage that requires fresh measurement work; S2-S4 are
straightforward edits; S5 is verification. A single agent should be
able to complete S1-S5 in one session.

| # | Risk | Likelihood | Impact | Mitigation |

A bust paired with a pinned toolchain change is recognizably "tool drift, not parser drift". Refresh under-coverage baseline; do not touch no-coverage baseline. |

| R3 | Optimization-level mismatch: coverage runs in debug, normal `cargo test` runs in debug. Both should be allocator-mode-symmetric, but a future change to `tools/full_test` that flips coverage to release would shift the under-coverage baseline. | Low (no plan to flip). | Under-coverage budget busts on next test bar. | The under-coverage budget comment names the test-bar invocation it was measured against (cf. S4 docstring). Anyone changing `tools/full_test` invocation should re-baseline; reviewer should catch it. |
| R4 | Stale `LLVM_PROFILE_FILE` in a developer's shell selects the looser budget when running `cargo test` outside the test bar. | Low (uncommon). | Oracle is loose, not silent. | R5 | The test bar starts running with `--cfg coverage` (drops `--no-cfg-coverage`). Cargo cfg-gates start firing in test code; alloc counts under coverage shift. | Low (no plan to change). | Under-coverage budget busts. | Re-baseline. Comment names `--no-cfg-coverage` as part of the measurement command, so the diff is easy to spot. |
| R6 | Variance between runs is higher than S1 measures. (We measure N=5; production sees N=many.) | Low (instrumentation is mostly deterministic). | Spurious budget bust on a particular run; CI flake. | If observed in practice, widen the spread term in the headroom formula. The fixture failure message names the constant, so the fix is obvious. |
| R7 | `LLVM_PROFILE_FILE` semantics change in a future cargo-llvm-cov release (e.g. only set during specific subcommands). | Low (it's been stable since cargo-llvm-cov's introduction). | Detection silently fails. | Detection failing means `under_coverage_instrumentation()` returns false under coverage; we'd see the **tight** budget under coverage, which fires loudly. R1 is the realistic worry. R7 is the most amusing one because it
fails-loud rather than fails-silent, which is exactly the property we
wanted from this whole redesign.

## Acceptance criteria

The implementation is complete when **all** of the following hold:

1. `cargo test -p rsiprtp --test allocations_sip_parse` runs and asserts
   the four no-coverage budgets (no skip path, no early return).
2. `cargo run --release -p full_test` runs to completion with stage 7
   green, and the under-coverage budget assertion fires (verified by
   inspection: deliberately tighten one under-coverage budget by 1, run
   the bar, see it fail, revert).
3. The module-level docstring explicitly names both refresh commands.
4. Each `*_BUDGET_UNDER_COVERAGE` constant carries a comment with: date,
   median value, spread across N=5, and the formula applied.
5. No new `#[ignore]` tests are added; `discover_baselines` continues to
   service both refresh paths.

## Why we are doing this

The current skip behaves correctly under both modes (no false positives
under coverage, real assertion under no-coverage) but is **dead in the
path that matters**: the test bar runs coverage, coverage sets
`LLVM_PROFILE_FILE`, the assertion skips. The reviewer flagged this as
a tracked follow-up; this HLD is that follow-up.

The 18-month-from-now reader of this code should understand that:

- "Under-coverage" budgets exist not because we tolerate coverage
  instrumentation as a parser cost, but because we cannot get a clean
  allocation signal *during* coverage and refuse to accept "the oracle
  doesn't run during the only pipeline that runs nightly" as the
  trade-off.
- The two baselines must be refreshed together, and the refresh recipe
  in the docstring is the authoritative source for how. Not this HLD.
  Not the journal. The docstring.

Two budgets is twice the maintenance cost of one. We accept that cost
because the alternative — silently disabled oracle — is worse.
