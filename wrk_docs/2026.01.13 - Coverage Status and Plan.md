## Multi-Stage Plan to Reach >98% Branch Coverage

### Stage 0: Stabilize Coverage Runs (Baseline)
**Goal:** Repeatable branch coverage runs on Windows and CI-friendly commands.

- Keep a canonical command line in docs/scripts:
  - `cargo +nightly llvm-cov --branch --workspace --exclude gabby -- --test-threads=1`
- Document nightly requirement and the reason for `--test-threads=1` (avoid socket/port contention on Windows).
- Optionally add a wrapper script under `wrk_docs` or `scripts/` for consistent local runs.
- Validate that ignored tests remain ignored; note which external dependencies (baresip, asterisk) affect coverage.

**Exit criteria:** Running the command above is stable and produces a consistent baseline report.

### Stage 1: Resolver and Transaction Manager Branches (Highest ROI)
**Goal:** Remove the largest branch deficits first.

- Resolver (`mdsiprtp-transport/src/resolver.rs`):
  - Add tests covering NAPTR -> SRV -> A/AAAA fallback chains.
  - Exercise error branches: empty record sets, malformed responses, timeouts, invalid protocols, invalid ports.
  - Cover preference logic: priority vs weight, tie-breakers, zero-weight entries, transport filtering.
- Transaction manager (`mdsiprtp-transaction/src/manager.rs`):
  - Add tests that hit:
    - No-matching-transaction responses (client/server).
    - Transport error branches during different states.
    - ACK handling paths for 2xx/4xx/5xx, retransmit variations.
    - Timer handling branches (A/B/C/D/E/F/G/H/I/J/K) in both invite/non-invite flows.

**Exit criteria:** Branch coverage for resolver and manager both >= 90%.

### Stage 2: ICE/TURN/STUN Branches
**Goal:** Improve branch coverage in ICE agent and TURN/STUN handling without relying on external servers.

- ICE agent (`mdsiprtp-ice/src/agent.rs`):
  - Role conflict and tiebreaker branches.
  - Candidate pair nomination and state transitions (Frozen/Waiting/In-Progress/Succeeded/Failed).
  - Edge cases around checklist pruning, retries, and timeouts.
- TURN/STUN (`mdsiprtp-ice/src/turn.rs`, `stun.rs`):
  - Error branches for malformed responses, auth failures, mismatched transaction IDs.
  - Permission and channel bind failures.
  - Retransmit and timeout handling.

**Exit criteria:** ICE agent branch coverage >= 85%, TURN/STUN >= 90%.

### Stage 3: Media and RTP Branches
**Goal:** Close remaining media codec and jitter buffer branches.

- G711 codec (`mdsiprtp-media/src/codec/g711.rs`):
  - Saturation and edge-value conversion branches.
  - Invalid input size branches (odd-length buffers, empty input).
- Jitter/mixer paths (`mdsiprtp-media/src/jitter.rs`, `mixer.rs`):
  - Underflow/overflow paths, late/early packet handling.
  - Reordering and loss concealment branches.
- RTP packet/session (`mdsiprtp-rtp/src/packet.rs`, `session.rs`):
  - Edge branches for timestamp wrap, SSRC change, and congestion state transitions.

**Exit criteria:** Media and RTP modules > 90% branch coverage each.

### Stage 4: Transport TLS/UDP Branches
**Goal:** Cover error-handling branches in TLS/UDP paths.

- TLS (`mdsiprtp-transport/src/tls.rs`):
  - Handshake failures, invalid certs, empty chain, unsupported signatures.
  - Read/write error branches with forced socket closure.
- UDP (`mdsiprtp-transport/src/udp.rs`):
  - Send/recv error branches, invalid addresses, zero-length payload handling.
  - Timeout and retry logic (if present).

**Exit criteria:** TLS >= 90% branch coverage; UDP >= 85%.

### Stage 5: Global Sweep and Gating
**Goal:** Push overall branch coverage > 98% and keep it there.

- Run `llvm-cov` with branch mode after each stage; track totals and deltas.
- Create a short list of remaining missed branches (top 20) and add focused tests.
- Consider a CI gate (soft threshold initially, e.g. 95%, then ratchet to 98%).

**Exit criteria:** Overall branch coverage > 98% (llvm-cov branch mode), stable across two consecutive runs.

## Risks and Assumptions

- Some branches are only reachable with OS/network failures; may require test-only hooks or dependency injection.
- External dependencies (baresip/asterisk) remain optional; coverage runs should not depend on them.
- Branch coverage is nightly-only; stable toolchains will not reproduce these numbers.
