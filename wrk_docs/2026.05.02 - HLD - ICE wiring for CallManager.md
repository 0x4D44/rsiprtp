# 2026.05.02 - HLD - ICE wiring for CallManager

Companion to `2026.05.01 - Brief - ICE wiring for CallManager.md`.

## Goal

Make ICE end-to-end usable for a call placed by a `CallManager` user, so
that an app behind a NAT can negotiate connectivity with its peer
without reaching into `ice::` directly.

## Shape of the change (and where it deviates from the brief)

The brief proposed wiring ICE *into* `CallManager` â€” adding
`IceManagerConfig` to `ManagerConfig`, new `ManagerEvent` variants for
gathering / connected / failed, and a per-call `IceMediaCoordinator`
held inside `Call`.

Reading the actual code changes that picture. Two facts dominate:

1. `CallManager` is **Sans-IO**: it holds no sockets, performs no I/O,
   and exposes events the app drains (`crates/rsiprtp/src/session/manager.rs:54`).
2. `RtpSession` is **Sans-IO** too: it produces packet bytes; the
   application owns the `UdpSocket` (see `examples/basic_call.rs:102`
   binding the socket directly).

Threading an async ICE agent through a Sans-IO state machine is the
wrong join. It forces `CallManager` to grow an async surface, hides
socket ownership, and makes the boundary fuzzy.

**Therefore: ICE lives next to `CallManager`, not inside it.** The
wiring is a thin async helper (`IceSession`) the app drives in parallel
with the existing manager â€” exactly the way the app already drives
`UdpSocket` and `TransactionManager` separately. `CallManager` itself
gains nothing new.

This is smaller, keeps the Sans-IO boundary clean, and is enough to
solve the actual problem (NAT traversal for outbound and inbound
calls). If real usage later proves a tighter coupling pulls its weight,
that is a separate change.

## Deliverables

1. **`crates/rsiprtp/src/ice/agent.rs` â€” small additions.**
   - `IceAgent::socket_for(addr) -> Option<Arc<UdpSocket>>` so the
     selected pair's socket can be extracted (currently the `sockets`
     `HashMap` is private and only `close()` touches it from outside).
   - Lower the default `gather_timeout_ms` from `5000` to `1500`. A
     5-second pre-INVITE block is a poor default; STUN RTT is well
     under 200ms in practice.
2. `crates/rsiprtp/src/session/ice_session.rs` â€” `IceSession` async
   helper. Drives the existing `IceAgent`, calls `socket_for` after
   checks succeed, exposes the socket + peer address for RTP.
3. `crates/rsiprtp/src/sdp/ice_attrs.rs` â€” read/write helpers for
   `a=candidate:`, `a=ice-ufrag:`, `a=ice-pwd:`, `a=rtcp-mux` against
   the existing generic `Attribute` API. Also patches the `c=` line
   and `m=` port to match the chosen *default candidate* (see
   "Concrete snags" below). No changes to `MediaDescription`.
4. **rtcp-mux negotiation in `sdp::negotiation`.** Today `create_answer`
   ignores `a=rtcp-mux` entirely. Under ICE we require it: emit it on
   offers we build, mirror it on answers we build, reject offers that
   omit it from a media line that also carries ICE attributes. (We
   don't add a separate-RTCP code path â€” the existing implementation
   de-facto muxes anyway.)
5. **Inbound deferred-answer entry point on `CallManager`.** Add
   `accept_inbound_invite(dialog, &offer_sdp) -> Option<CallId>` that
   does the `handle_incoming_invite` work *minus* answer construction.
   Add `build_answer_for(call_id, default_candidate, ice_local) ->
   Option<SessionDescription>` that the app calls after `IceSession`
   has gathered. The existing `handle_incoming_invite` stays for the
   non-ICE path.
6. `crates/rsiprtp/examples/ice_call.rs` â€” runnable end-to-end demo
   between two `CallManager` instances on loopback with ICE enabled.
7. `crates/rsiprtp/tests/ice_basic.rs` â€” integration test mirroring
   the example without the boilerplate.
8. README and `lib.rs` doc updates: ICE moves from "out of scope"
   to "supported (host + srflx, no trickle, no restart, no IPv6
   dual-stack guarantees, no consent-freshness keepalives)".

Out of scope (matches the brief, plus the additional carve-outs found
during HLD review): trickle ICE, ICE restart, TURN credential refresh,
DTLS-SRTP, ICE-lite peer mode, RFC 7675 consent freshness, IPv6
dual-stack interop testing.

## Public API sketch

```rust
// session/ice_session.rs

pub struct IceSession {
    agent: IceAgent,
    socket: Arc<UdpSocket>,            // bound during gather()
    local: IceLocalParams,             // ufrag, pwd, candidates
    selected_peer: Option<SocketAddr>, // set after run_checks() succeeds
}

pub struct IceLocalParams {
    pub ufrag: String,
    pub pwd: String,
    pub candidates: Vec<Candidate>,
}

#[derive(Debug, Clone)]
pub struct IceConfig {
    pub stun_servers: Vec<StunServer>, // empty => host candidates only
    pub gather_timeout: Duration,
    pub check_timeout: Duration,
}

impl IceSession {
    /// Construct, bind a single UDP socket in the configured port
    /// range, and gather host + srflx candidates. Single component
    /// only â€” RTP and RTCP share the socket (rtcp-mux required).
    pub async fn gather(
        config: IceConfig,
        role: IceRole,
        rtp_port_range: (u16, u16),
    ) -> Result<Self, IceError>;

    pub fn local(&self) -> &IceLocalParams;

    /// Apply candidates and credentials from the peer's SDP and run
    /// connectivity checks. Returns the nominated peer `SocketAddr`.
    pub async fn run_checks(
        &mut self,
        remote: IceRemoteParams,
    ) -> Result<SocketAddr, IceError>;

    /// After `run_checks` succeeds: the socket the app should send
    /// RTP from, and the address it should send to.
    pub fn rtp_socket(&self) -> Arc<UdpSocket>;
    pub fn peer_addr(&self) -> Option<SocketAddr>;
}

pub struct IceRemoteParams {
    pub ufrag: String,
    pub pwd: String,
    pub candidates: Vec<Candidate>,
}
```

```rust
// sdp/ice_attrs.rs â€” round-trip helpers, no new SDP types.

pub fn read_ice_credentials(m: &MediaDescription) -> Option<(String, String)>;
pub fn read_candidates(m: &MediaDescription) -> Vec<Candidate>;
pub fn read_rtcp_mux(m: &MediaDescription) -> bool;

pub fn write_ice_credentials(m: &mut MediaDescription, ufrag: &str, pwd: &str);
pub fn write_candidates(m: &mut MediaDescription, cands: &[Candidate]);
pub fn write_rtcp_mux(m: &mut MediaDescription);

/// Patch `c=` and `m=` port to the default candidate's address,
/// per RFC 8839. The default candidate is the lowest-priority host
/// candidate (so the line stays usable for non-ICE peers).
pub fn apply_default_candidate(
    sdp: &mut SessionDescription,
    media_index: usize,
    default: &Candidate,
);
```

These touch only `MediaDescription.attributes: Vec<Attribute>` and the
`c=` / `m=` fields. The existing `Candidate::to_sdp()` /
`Candidate::from_sdp()` in `crates/rsiprtp/src/ice/candidate.rs:158`
already handle the wire format â€” we just push
`Attribute { name: "candidate", value: Some(c.to_sdp()) }` and read it
back.

## Concrete snags surfaced during HLD review

Four points the first draft glossed over. Each one is small but real;
none change the overall shape.

### 1. Socket extraction from `IceAgent`

`IceAgent` already binds one `UdpSocket` per local interface during
host-candidate gather and stores them in a private `HashMap`. There is
no public accessor today. After `selected_pair()` returns, only one of
those sockets corresponds to the winning local candidate, and that is
the socket we want to hand to RTP.

We add `IceAgent::socket_for(addr: SocketAddr) -> Option<Arc<UdpSocket>>`
that returns a clone of the `Arc` for the matching key. `IceSession`
calls it once `run_checks` succeeds and stashes the result. No new
ownership semantics â€” the socket is shared via `Arc`, which is exactly
what the agent already does internally for STUN.

### 2. Inbound deferred-answer flow

`handle_incoming_invite` is synchronous and returns the answer SDP in
its result tuple. With ICE, gather is async â€” the answer cannot be
built until candidates are ready. Rather than make the existing
function async (it would force every caller to await even non-ICE
flows), we split the inbound path:

- `accept_inbound_invite(dialog, &offer)` â€” does dialog/codec
  bookkeeping and returns a `CallId`. Synchronous. No SDP answer yet.
- `build_answer_for(call_id, default_candidate, ice_local)` â€” builds
  the answer SDP including ICE attributes once gather has finished.
  Synchronous; the async work is in `IceSession`, not here.

Non-ICE callers keep using `handle_incoming_invite` unchanged.

### 3. SDP `c=` line and `m=` port under ICE

RFC 8839 requires the `c=` connection address and `m=` port to be the
*default candidate's* address â€” typically a host candidate. Our SDP
is built first (codecs, port, `c=` line), then we layer ICE
attributes on top. The natural seam is one helper call:

```rust
sdp::ice_attrs::apply_default_candidate(&mut sdp, media_idx, &host_cand);
sdp::ice_attrs::write_ice_credentials(audio, &ice.local().ufrag, &ice.local().pwd);
sdp::ice_attrs::write_candidates(audio, &ice.local().candidates);
sdp::ice_attrs::write_rtcp_mux(audio);
```

`IceSession::default_candidate() -> &Candidate` exposes the choice
deterministically (lowest-priority host, falling back to first if no
host candidates were gathered).

### 4. rtcp-mux is currently unimplemented

Searching the codebase, `rtcp-mux` only appears in test scaffolding â€”
`create_answer` ignores it, and there is no separate-RTCP send path
either way. The existing stack effectively muxes by accident (one
socket, RTCP rarely emitted). Under ICE we make this explicit: emit
`a=rtcp-mux` on every offer/answer that carries ICE attributes, and
reject offers that combine ICE with no `rtcp-mux` (we will not grow a
two-component path). This is one new flag through `create_answer` and
the corresponding offer builder, plus the `read_rtcp_mux`/`write_rtcp_mux`
helpers above.

### Smaller items folded in elsewhere

- Default `gather_timeout_ms` lowered from 5000 â†’ 1500 (deliverable
  #1).
- `MediaSession.remote_addr` is unused under ICE; the app sends to
  `ice.peer_addr()` instead. Documented as a comment on
  `set_negotiated_media`, no code change.
- ICE failure â†’ call termination is the app's responsibility. The
  example shows the pattern: `if ice.run_checks().await.is_err() {
  manager.terminate_call(&call_id); }`. No new manager API.
- Port allocation: `CallManager::allocate_rtp_port` is bypassed for
  ICE calls. The `local_port` argument to `set_negotiated_media`
  becomes the port of the IceSession's chosen default candidate. We
  do *not* coordinate the two allocators â€” apps that mix ICE and
  non-ICE calls accept that ports come from the same range
  unsynchronised, and that is fine.

## Caller flow

The app flow for an outbound ICE call, end to end:

```
1. ice = IceSession::gather(cfg, Controlling, port_range).await?;
2. call_id = manager.create_call("sip:bob@...".to_string());
3. offer_sdp = build_offer(local_addr, codecs);
   sdp::ice_attrs::write_ice_credentials(audio, &ice.local().ufrag, &ice.local().pwd);
   sdp::ice_attrs::write_candidates(audio, &ice.local().candidates);
   sdp::ice_attrs::write_rtcp_mux(audio);
4. send INVITE with offer_sdp via existing transaction layer.
5. on 200 OK with answer_sdp:
     manager.handle_invite_success(...);
     remote = parse_remote_ice_params(&answer_sdp);
     peer = ice.run_checks(remote).await?;
6. start RTP loop on ice.rtp_socket() sending to peer; feed received
   bytes into RtpSession + JitterBuffer as today.
```

Inbound is the mirror: gather *before* generating the 200 OK answer so
the answer carries our credentials and candidates; run checks after the
ACK lands.

## Design choices and rejections

- **Single ICE component (rtcp-mux required).** Pairs are per-component
  (RFC 8445); two components doubles the work for nothing useful. If a
  peer offers without `rtcp-mux`, we reject the offer rather than
  growing a second-component path. Brief's open decision #3 â€” settled.
- **STUN-only by default; TURN is opt-in and we don't ship a TURN
  config helper in this change.** The `TurnClient` exists already; if
  the user wants relay, they can add servers to `IceConfig.stun_servers`
  manually once we surface that. No reason to scope-creep this work.
- **`IceConfig.stun_servers` defaults to empty.** Avoid the surprise
  of phoning home to Google STUN. Brief's open decision (and risk
  register) â€” settled.
- **Regular nomination, controlling derived from offer/answer role.**
  Offerer is controlling. No collision-resolution code in this change â€”
  if the upstream agent already handles it, fine; if not, it's a
  one-line follow-up.
- **One `UdpSocket` per call.** ICE binds it, RTP sends from it. No
  socket handoff, no `from_std` dance â€” `IceSession` simply hands out
  `Arc<UdpSocket>`. The "socket handoff is fiddly on Windows" risk in
  the brief is a non-issue once ICE owns the socket from the start.
- **No trickle.** Gather completes before the offer is built. Local
  network gather is sub-100ms; STUN gather is bounded by
  `gather_timeout` (default 1.5s, lower than the brief's 5s â€” a typical
  STUN RTT is well under 200ms).
- **No SDP type changes.** Brief proposed adding typed
  `IceCandidate`/`ice_ufrag`/`ice_pwd` fields to `MediaDescription`.
  That is more surface than the problem requires; the generic
  `Attribute` API plus three read/three write helpers is enough and
  keeps the existing parser/builder unchanged.

## Testing

- **Unit tests** in `sdp/ice_attrs.rs`: round-trip a `MediaDescription`
  with candidates + credentials + rtcp-mux through writeâ†’parseâ†’read.
  Use one captured offer (from baresip or Linphone) as a ground-truth
  fixture.
- **Unit tests** in `session/ice_session.rs` exercising `gather` with
  no STUN servers (host-only, deterministic) and `run_checks` against a
  second `IceSession` on loopback.
- **Integration test** `tests/ice_basic.rs`: two `CallManager`
  instances + two `IceSession`s on `127.0.0.1`, exchange offer/answer
  through a fake transport, run checks, send a few RTP packets, assert
  they arrive.

The integration test is the load-bearing one â€” it is the oracle that
encodes "ICE wiring works end to end" in a form a machine can check.
