### HLD — SDP parser round-trip oracle (+ SIP fuzz target backfill)

**Date:** 2026-05-05
**Author:** Claude (with Arthur)
**Status:** Draft for review
**Related:** `2026.05.04 - HLD - SIP parser round-trip oracle.md`,
`fuzz/fuzz_targets/sdp_session.rs`

## Context

We have one SDP fuzz target today: `fuzz/fuzz_targets/sdp_session.rs`
calls `SessionDescription::parse` and discards the result. That's a
pure crash/UB target — it catches panics and memory bugs but no
semantic bugs.

The SIP side recently landed a round-trip oracle (M11): parse the
wire form, serialize, parse again, and assert the second parse is a
fixed point of the parse/serialize cycle. The same shape applies
directly to SDP — `SessionDescription` is another text grammar
with parse and serialize sides we own, and RFC 4566 has plenty of
corners (whitespace, missing session name, lossy bandwidth, unknown
media types) where the two sides could disagree.

This is the highest-ROI extension of the SDP fuzz coverage: ~30 LOC
of oracle, ~15 LOC of fuzz target, an extra static-fixture driver.

### Why this HLD also covers a SIP backfill

While auditing the SIP round-trip oracle work to mirror it for SDP,
two deliverables from the SIP HLD's implementation plan turned out
to have never landed:

- `fuzz/fuzz_targets/sip_message_roundtrip.rs` does not exist.
- The matching `[[bin]]` entry in `fuzz/Cargo.toml` does not exist.

Only the oracle module (`tests/parser_roundtrip_oracle/mod.rs`)
and the static-fixture driver (`tests/parser_roundtrip.rs`)
shipped. The static driver alone covers ~16 hand-curated fixtures;
the unshipped fuzz target was the part that would explore beyond
that corpus, which is exactly the point of the M11 work. Shipping
the SDP fuzz target without backfilling the SIP one would leave
the SIP side under-tested in the same way that motivated this
whole HLD for SDP.

This HLD therefore folds the SIP fuzz-target backfill into the same
change. The SDP and SIP fuzz targets are mechanically near-identical
(~10 LOC each, share the same `#[path]` pattern), so the marginal
cost is one file and one `[[bin]]` entry.

## Goals

- Catch bugs where SDP parse and serialize disagree (any
  non-fixed-point in the parse → serialize cycle).
- Run as both a static fixture suite and a libfuzzer target — for
  *both* SDP and SIP. Backfill the SIP fuzz target that the M11
  HLD specified but never delivered.
- Land alongside the existing `sdp_session` crash target without
  touching it.

## Non-goals

- Catch bugs both parse and serialize *agree* on (silently dropped
  attributes, character escaping that's wrong but symmetric).
  Differential or grammar-conformance work, not this oracle.
- Replace `sdp_session.rs`. It still has value as a cheap crash
  target; round-trip is layered on top.
- Fix the known-lossy normalizations (see "What we already absorb
  on the first round-trip"). They're behaviors, not bugs the
  oracle is asked to find — and changing them is out of scope for
  a test-only change.

## What's already in place

- `SessionDescription` derives `PartialEq` (and so do `Origin`,
  `Connection`, `Timing`, `MediaDescription`, `Attribute`,
  `MediaType`, `Direction`).
- `impl std::fmt::Display for SessionDescription` (in
  `crates/rsiprtp/src/sdp/builder.rs:275`) emits the session in
  RFC-4566 line order. `to_string()` is the serialize side.
- `SessionDescription::parse(&str)` is the parse side
  (`crates/rsiprtp/src/sdp/parser.rs:32`).
- `fuzz/fuzz_targets/sdp_session.rs` already takes raw bytes and
  feeds the UTF-8 view to `SessionDescription::parse`.

So the primitives exist. What's missing is corpus coverage and a
round-trip target.

## Design

### The invariant

Same shape as the SIP oracle — fixed point after one normalization:

```rust
let s1 = SessionDescription::parse(input)?;        // OK is required
let s2 = SessionDescription::parse(&s1.to_string())?; // must succeed
let s3 = SessionDescription::parse(&s2.to_string())?; // must succeed
assert_eq!(s2, s3);                                // AST-equal
assert_eq!(s2.to_string(), s3.to_string());        // bytes-equal
```

If the first parse fails, round-trip is undefined; the oracle
returns silently. Fuzz inputs are mostly malformed — this is the
right discard rule.

### What we already absorb on the first round-trip

Listing these explicitly so reviewers know what is *expected*
non-idempotency on the s1 → s2 step (and therefore can't be
asserted at s1 == s2):

- **Whitespace collapse.** `Origin`, `Connection`, `Timing`, and
  `MediaDescription` parse via `split_whitespace`, so any run of
  spaces/tabs in the input collapses to single spaces on serialize.
- **Per-line trim.** Each line is `trim()`-ed before parsing, so
  leading/trailing whitespace and stray `\r` characters are
  silently dropped.
- **Defaulted session name.** Missing `s=` becomes `s=-` (RFC 4566
  requires `s=`; `parser.rs:109`).
- **Version reparse.** `v=` is parsed as `u8` and serialized as
  `Display` of `u8` — `v=00` becomes `v=0`. `v=` outside `0..=255`
  is rejected at parse, so the oracle does not see it.
- **`MediaType::Other` collapse.** Unknown media type strings (e.g.
  `m=foo …`) round-trip through `MediaType::Other` and serialize
  as the literal `other` (`builder.rs:333`). `m=foo` → `m=other`
  on the first round-trip; fixed at s2.
- **`num_ports` is silently dropped on serialize.** Input
  `m=audio 49170/2 RTP/AVP 0` parses into `MediaDescription { port:
  49170, num_ports: Some(2), … }`, but `write_media`
  (`builder.rs:336–347`) emits only `port`. Reparse → `num_ports:
  None`. **This is an actual semantic bug** (port-pair RTP/RTCP is
  lost), not just whitespace normalization. The oracle still holds
  the fixed point at s2, but flagging it because anyone reading the
  list of "expected" normalizations should know one of them is a
  latent defect worth fixing separately.
- **Lossy bandwidth value.** `b=AS:abc` parses with
  `bw.parse().unwrap_or(0)` (`parser.rs:69`) and serializes as
  `b=AS:0`. Lossy, but a fixed point at s2.
- **Lossy timing values.** `t=` start/stop parse via
  `parse().unwrap_or(0)` (`parser.rs:215–218`). Same lossy-parse
  shape as bandwidth.
- **Lossy origin tail.** `Origin::parse` reads exactly six
  whitespace-separated parts and ignores anything past index 5
  (`parser.rs:149`). Trailing junk is dropped on the first
  serialize; fixed at s2.
- **Multiple `c=` per media → last wins.** RFC 4566 §5.7 allows
  multiple connection lines per media for multicast; our parser
  overwrites (`'c' => m.connection = Some(…)`). Lossy.
- **Multiple `b=` with the same key per media → last wins.** The
  bandwidth `HashMap::insert` overwrites on key collision.
- **Session-level `b=` is dropped entirely.** Only the media-level
  branch handles `b=` (`parser.rs:62–77`); the session-level match
  arm has no `'b'` case (`parser.rs:89–98`). `b=` *is* a known SDP
  type; we just don't store it at the session level.
- **Misordered fields after `m=` are dropped.** Once
  `current_media` is set, the loop only routes `c`/`b`/`a` into the
  media block; anything else (`v=`, `o=`, `s=`, …) hits the `_ =>
  {}` and is silently dropped.
- **Unknown SDP types.** Single-letter types the parser doesn't
  recognize (e.g. `u=`, `p=`, `e=`, `r=`, `z=`, `k=`, `i=` at media
  level) are dropped on parse. Lossy on first round-trip; fixed at
  s2.
- **CRLF vs LF on serialize.** `Display` uses `writeln!`, which
  emits `\n`. RFC 4566 §5 requires `\r\n`. The parser accepts
  either (`for line in sdp.lines()`), so the fixed point holds —
  but this is a real RFC-conformance issue worth filing
  separately. Not in scope here.

The fixed-point-at-s2 form catches non-idempotency bugs without
forcing any of the above corpus restrictions. **None of these
expected normalizations are fixed by this change** — calling them
out so reviewers don't mistake them for things the round-trip
oracle was meant to find.

### What the oracle is well-positioned to find

Bugs the SIP equivalent has historically caught — and the
corresponding SDP failure modes:

- A serialize side that emits a form the parse side rejects
  (round-trip dies at the second parse).
- A parse side that produces ASTs whose serialize → reparse
  diverges (`s2 != s3`).
- A non-deterministic field ordering on serialize (would surface
  as bytes-inequality between b2 and b3 within the same process).

### What the oracle will *not* catch

Symmetric loss — anything both parse and serialize agree to drop.
Examples present in our code today: unknown SDP types like `r=`
(repeats), `z=` (timezones), `k=` (encryption keys), and the
`a=fmtp` / `a=rtpmap` rich-attribute round-trip beyond the
opaque `Attribute { name, value }` representation. These need a
grammar-conformance oracle, not this one. Out of scope.

### Oracle module

New file `crates/rsiprtp/tests/sdp_roundtrip_oracle/mod.rs`,
parallel to `parser_roundtrip_oracle/mod.rs`. One public function:

```rust
/// Assert the round-trip fixed-point invariant on `bytes`.
/// Decodes UTF-8 first; non-UTF-8 input → silent return (parse can't
/// see it). If first parse fails → silent return.
pub fn assert_roundtrip_fixed_point(bytes: &[u8]) { ... }
```

No skips, no configuration, no feature flags. ~30 LOC.

### Static-fixture driver

New file `crates/rsiprtp/tests/sdp_roundtrip.rs`, parallel to
`tests/parser_roundtrip.rs`. One `#[test]` per fixture under a new
`tests/fixtures/sdp/` directory.

Fixtures (~6 files, lifted from RFC 4566 examples and our own
inline tests in `crates/rsiprtp/src/sdp/tests/`):

- `minimal.sdp` — only `v=`, `o=`, `s=`, `t=`. No media.
- `audio_pcmu.sdp` — single audio m-line, PCMU, sendrecv.
- `audio_video.sdp` — two m-lines (audio + video), session-level
  `c=`, per-media attributes including `a=rtpmap` and `a=fmtp`.
- `bandwidth.sdp` — exercises `b=AS:` and `b=CT:` at session and
  media level.
- `rfc4566_example.sdp` — the canonical example from RFC 4566 §5
  (verbatim, in canonical CRLF form).
- `compact_whitespace.sdp` — multiple spaces in `o=` / `m=` lines
  to exercise the whitespace-collapse normalization.

Plus three sanity tests on the oracle itself, mirroring
`rt_oracle_skips_parse_failures`, `rt_oracle_holds_on_canonical_input`,
`rt_oracle_holds_after_normalization` from the SIP driver.

~50 LOC including fixtures-as-includes.

### Fuzz target

New file `fuzz/fuzz_targets/sdp_session_roundtrip.rs`:

```rust
#![no_main]
use libfuzzer_sys::fuzz_target;

#[path = "../../crates/rsiprtp/tests/sdp_roundtrip_oracle/mod.rs"]
mod oracle;

fuzz_target!(|data: &[u8]| {
    oracle::assert_roundtrip_fixed_point(data);
});
```

~10 LOC. Shares the oracle module via `#[path]` the same way the
SIP differential and round-trip targets share theirs. Add a `[[bin]]`
entry to `fuzz/Cargo.toml`.

### Skip / pin mechanism

**None at start.** Same reasoning as the SIP oracle: round-trip is
self-consistency on code we own — any failure is a real bug. If a
defensible non-fixed-point case turns up that we choose not to fix,
add a pinned skip then with the same pattern as
`parser_diff_oracle::Skip::Reason`.

### SIP round-trip fuzz target (backfill from M11)

The SIP M11 HLD specified, but did not deliver,
`fuzz/fuzz_targets/sip_message_roundtrip.rs` and the matching
`[[bin]]` entry. Backfilling it is mechanical:

```rust
// fuzz/fuzz_targets/sip_message_roundtrip.rs
#![no_main]
use libfuzzer_sys::fuzz_target;

#[path = "../../crates/rsiprtp/tests/parser_roundtrip_oracle/mod.rs"]
mod oracle;

fuzz_target!(|data: &[u8]| {
    oracle::assert_roundtrip_fixed_point(data);
});
```

The oracle module already exists and is already written to be
shareable via `#[path]` (see its docstring at
`tests/parser_roundtrip_oracle/mod.rs:10–16`). All this change
adds is the binary that calls into it.

Total cost: 1 new file (~10 LOC) + 1 `[[bin]]` block in
`fuzz/Cargo.toml`. No changes to the existing oracle module or
static-fixture driver.

### Bandwidth iteration is non-deterministic — fix on serialize

`MediaDescription.bandwidth` is `HashMap<String, u32>` and the
serializer iterates it directly (`builder.rs:355`). std Rust's
`HashMap::new()` calls `RandomState::new()`, which pulls a fresh
seed from a thread-local counter incremented on every call. Two
parses of the same input produce two `HashMap`s with **different
seeds**, so for ≥2 bandwidth entries the iteration order on
serialize can differ between s2 and s3. AST equality
(`s2 == s3`) still holds because `HashMap: PartialEq` is
content-equal, but bytes-equality would flake.

This change therefore includes a one-line determinism fix in the
serializer: collect entries, sort by key, then iterate. It's
behavior-preserving for any caller that doesn't depend on
iteration order (no caller can — the field is `HashMap`), and it
makes the oracle's bytes-equality assertion trustworthy.

```rust
// builder.rs, write_media — replaces `for (btype, bw) in &media.bandwidth`
let mut bw: Vec<_> = media.bandwidth.iter().collect();
bw.sort_by(|a, b| a.0.cmp(b.0));
for (btype, bw) in bw {
    writeln!(f, "b={}:{}", btype, bw)?;
}
```

Considered switching the field to `BTreeMap`. Rejected: changes a
public type and its `PartialEq` semantics (BTreeMap's equality is
order-sensitive in display but not in `==`, same as HashMap, but
the type signature change is a wider blast radius for what is
purely a serializer-side determinism issue).

## Trade-offs and acknowledged risks

**Risk:** the corpus is small. Mitigated by the fuzz target. We
can also seed it with `fuzz/triage/sdp_session-001/crash.bin` and
the inline-test SDP strings already in
`crates/rsiprtp/src/sdp/tests/rfc4566.rs`.

**Risk:** the oracle finds nothing. As with SIP, that's a *good*
outcome — the oracle's value then is regression-prevention. Cheap
to build and run; worth keeping either way.

**Trade-off rejected:** stricter "bytes-equal after first
round-trip" oracle. Forces every fixture to already be canonical
(CRLF, no whitespace runs, no compact forms, no missing `s=`),
which the RFC 4566 example corpus is not. Same call as the SIP
side.

**Trade-off rejected:** mixing this into `sdp_session.rs` as a
second assertion. Keeps the cheap crash target cheap and keeps two
independent invariants in two files. Same call as the SIP side.

**Trade-off rejected:** fixing the `\n` vs `\r\n` serialize, the
`MediaType::Other` collapse, and the unknown-type drop as part of
this work. All real bugs but each one is a behavior change with
its own review surface; the oracle doesn't need them fixed in
order to assert the fixed-point invariant.

## Out of scope

- Grammar-conformance oracle for SDP (catches symmetric loss).
- Differential oracle against another SDP parser.
- Round-tripping rich attributes (`a=rtpmap`, `a=fmtp`,
  `a=candidate`, etc.) at the typed level rather than as opaque
  `Attribute { name, value }`.
- Fixing `Display` to emit `\r\n`, preserving `MediaType::Other`'s
  original string, retaining `num_ports`, retaining unknown SDP
  types, or retaining session-level `b=` — all real improvements,
  all separate changes.
- Any change to `parser.rs`. The only `builder.rs` change is the
  bandwidth-sort determinism fix called out above.

## Implementation plan

**Phase A — SDP round-trip oracle**

1. Apply the bandwidth-sort determinism fix in
   `builder.rs::write_media` (~4 LOC).
2. Write `tests/sdp_roundtrip_oracle/mod.rs` (~30 LOC).
3. Add `tests/fixtures/sdp/` with 6 fixture files (lines vary).
4. Write `tests/sdp_roundtrip.rs` driver (~50 LOC), test names
   prefixed `rt_sdp_*` to be friendly under `cargo test` filters.
5. Write `fuzz/fuzz_targets/sdp_session_roundtrip.rs` (~10 LOC).
6. Add `[[bin]]` entry for `sdp_session_roundtrip` to
   `fuzz/Cargo.toml`.

**Phase B — SIP fuzz-target backfill (the M11 gap)**

7. Write `fuzz/fuzz_targets/sip_message_roundtrip.rs` (~10 LOC).
8. Add `[[bin]]` entry for `sip_message_roundtrip` to
   `fuzz/Cargo.toml`.

**Phase C — Validation**

9. `cargo test --workspace` — confirm both fixture suites pass.
10. `cargo fuzz build` — confirm both new targets compile.
11. `cargo fuzz run sdp_session_roundtrip -- -max_total_time=600`
    — 10-minute smoke fuzz on SDP. Triage any findings into a
    journal pin the same way the SIP M11 fuzz findings were
    handled.
12. `cargo fuzz run sip_message_roundtrip -- -max_total_time=600`
    — 10-minute smoke fuzz on SIP, primarily to catch any
    round-trip regressions that landed since the M11 static driver
    shipped without the fuzz coverage that was supposed to back it.

Seed corpus for step 11: copy the existing
`fuzz/triage/sdp_session-001/crash.bin` plus the SDP strings
embedded in `crates/rsiprtp/src/sdp/tests/rfc4566.rs` into a new
`fuzz/corpus/sdp_session_roundtrip/` directory before running.
Seed corpus for step 12: reuse the SIP fixtures under
`crates/rsiprtp/tests/fixtures/{handcrafted,mdsiprtp3,rfc4475}/`
into `fuzz/corpus/sip_message_roundtrip/`. Cheap acceleration;
not required for either.

Total estimated diff: ~105 LOC across 5 new files + 2 Cargo.toml
`[[bin]]` blocks + 6 short SDP fixtures + the 4-line `builder.rs`
determinism fix.
