## 1. Problem statement

The repository currently has **two cargo-fuzz crates**:

| crate                          | targets | special deps         | workspace                  |
| ------------------------------ | :-----: | -------------------- | -------------------------- |
| `fuzz/` (root)                 |   28    | `rsiprtp` only       | standalone, empty `[workspace]` |
| `crates/rsiprtp/fuzz/`         |    2    | `rsiprtp` + **`rsip = "0.4"`** | standalone, `members=["."]` |

The nested crate exists for one reason: its `sip_message_parse_diff`
target is a *differential* fuzzer that compares the in-tree SIP parser
against `rsip` 0.4 via the shared oracle module at
`crates/rsiprtp/tests/parser_diff_oracle/mod.rs` (pulled in via
`#[path = "../../tests/parser_diff_oracle/mod.rs"]`). The oracle code
references `rsip::*` directly, so the fuzz crate needs `rsip` as a
regular `[dependency]`.

Pre-M10, when `rsip` was a runtime dep of `crates/rsiprtp`, this layout
was free. After M10 dropped `rsip` from runtime deps, the only remaining
reason to keep two fuzz crates is **historical inertia**.

### Costs of the two-crate layout

1. **Two corpora** under `fuzz/corpus/` and `crates/rsiprtp/fuzz/corpus/`
   confuse seed-management. We just had to delete a 16,772-file stale
   leftover (`crates/rsiprtp/fuzz/corpus/sip_message_roundtrip/`) from a
   prior dedupe attempt — exactly the failure mode this layout invites.
2. **Two `Cargo.lock` files** to keep in sync (only `fuzz/Cargo.lock` is
   tracked; `crates/rsiprtp/fuzz/Cargo.lock` is gitignored — but they
   still both exist on disk and `cargo fuzz build` regenerates each).
3. **Two cwd contexts** for `cargo +nightly fuzz`. The wrapper auto-discovers
   them, but each new contributor has to learn that `--fuzz-dir` /
   working-directory matters.
4. **`sip_message_parse` is functionally redundant** with the root crate's
   `sip_message` + `sip_message_roundtrip` targets. It calls the same
   `SipMessage::parse` entry point, then walks accessors that
   `SipMessage::to_bytes()` already exercises — `sip_message_roundtrip`
   strictly subsumes its coverage.
5. **Cognitive overhead.** New fuzz targets need a "which crate?" decision
   that has nothing to do with the target's purpose.

### What's already proven to work from the root crate

The root `fuzz/fuzz_targets/sip_message_roundtrip.rs` already uses the
cross-tree pattern this consolidation needs:

```rust
#[path = "../../crates/rsiprtp/tests/parser_roundtrip_oracle/mod.rs"]
mod oracle;
```

So reaching from `fuzz/fuzz_targets/` into `crates/rsiprtp/tests/` via
`#[path]` is an established, working idiom in this repo. The diff target
needs the analogous pattern — only the relative path adjusts.

## 2. Recommendation

**Consolidate into a single root `fuzz/` crate. Retire `sip_message_parse`.
Migrate `sip_message_parse_diff` into the root crate.**

### Strongest argument FOR

The roundtrip oracle precedent already proves this works. We're not
inventing a layout — we're applying an existing one that is demonstrably
sound. The two-crate layout is a fossil from before the M10 rsip
dropoff; keeping it costs ongoing attention with zero offsetting benefit.
Consolidation is a one-time cleanup that immediately eliminates a class
of "stale corpus" bugs (we just had one) and the "which fuzz crate?"
decision for new contributors.

### Strongest argument AGAINST (must be honestly considered)

**Adding `rsip = "0.4"` to root `fuzz/Cargo.toml` lengthens fuzz-build
times for the other 27 targets** (rsip pulls in `nom` and ~20 transitive
crates). LibFuzzer-instrumented release builds are already ~3 minutes;
this could push them to ~3:30. Worse, **rsip is a *fuzz-only* test
oracle, not part of any rsiprtp surface** — having it in the same crate
graph as the RTP/SDP/ICE/codec fuzz targets blurs that boundary. A maximally
hygienic alternative is "keep the diff target separate so the root fuzz
crate stays oracle-free". The counter-argument: dev-only deps in the
fuzz crate's transitive closure are not architecturally meaningful — the
fuzz crate is a build target with `publish = false`, not a published
artifact, and `rsip` is no more "in the fuzz graph" than `libfuzzer-sys`
itself is.

**Other arguments AGAINST that I considered and discarded:**

- *"The nested layout matches the convention from when each crate was
  its own workspace member."* Defunct: the workspace was collapsed in
  the May 2026 single-crate refactor (`2026.05.01 - HLD - Collapse
  workspace into single rsiprtp crate.md`). The nested fuzz crate is a
  vestige of that older shape.
- *"Cargo features could keep `rsip` opt-in."* `cargo-fuzz` doesn't
  support per-target features cleanly (every target gets the same
  feature set), and adding feature plumbing would cost more than the
  ~30s of build time it saves.
- *"Other agents might have in-flight work in `crates/rsiprtp/fuzz/`."*
  Mitigated by atomic commit and a clear journal entry; risk register
  below.

### Net call

Consolidate. The repo gets simpler, the wrapper auto-discovery becomes a
no-op (which is fine — it's still correct), and the only cost is a small,
well-understood build-time hit on `cargo fuzz build`. **Cost is bounded
and predictable; the status-quo cost is ongoing and unbounded.**

## 3. Detailed design (consolidate path)

### 3.1 File moves

| from                                                                    | to                                                                  |
| ----------------------------------------------------------------------- | ------------------------------------------------------------------- |
| `crates/rsiprtp/fuzz/fuzz_targets/sip_message_parse_diff.rs`            | `fuzz/fuzz_targets/sip_message_parse_diff.rs`                       |
| `crates/rsiprtp/fuzz/fuzz_targets/sip_message_parse.rs`                 | **DELETE** (subsumed by `sip_message` + `sip_message_roundtrip`)    |
| `crates/rsiprtp/fuzz/Cargo.toml`                                        | **DELETE** (the whole nested crate goes away)                       |
| `crates/rsiprtp/fuzz/Cargo.lock` (gitignored)                           | gone with the crate                                                 |
| `crates/rsiprtp/fuzz/.gitignore`                                        | **DELETE**                                                          |
| `crates/rsiprtp/fuzz/M11_LAUNCH.md`                                     | move to `crates/rsiprtp/fuzz_M11_LAUNCH.md` *or* fold into root `fuzz/README.md` (see §3.6) |
| `crates/rsiprtp/fuzz/README.md`                                         | DELETE; pointer note in root `fuzz/README.md` (TBD — see §3.6)      |
| `crates/rsiprtp/fuzz/run_launcher*.ps1`, `fuzz_supervisor*.ps1`         | DELETE — runtime-artifact `.ps1` scripts; the inventory test reads `crates/rsiprtp/fuzz/*.ps1` so we must update it, see §3.5 |
| `crates/rsiprtp/fuzz/corpus/sip_message_parse/`                         | corpus migration: developer-machine concern, see §3.4               |
| `crates/rsiprtp/fuzz/artifacts/`, `target/`                             | DELETE locally (gitignored, no repo concern)                        |

The `oracle::*` module at `crates/rsiprtp/tests/parser_diff_oracle/mod.rs`
**stays put** — it is also consumed by `crates/rsiprtp/tests/parser_diff.rs`
(a real cargo test) which uses `#[path = "parser_diff_oracle/mod.rs"]`
relative to its own location. Per the brief's constraint, that consumer
must keep working unchanged.

### 3.2 `Cargo.toml` edits

In `fuzz/Cargo.toml`, add:

```toml
# rsip 0.4 is the differential-test oracle for the
# `sip_message_parse_diff` target. It is a fuzz-only dep of this crate
# (publish = false). The main crate carries `rsip` only as a
# `[dev-dependencies]` entry; M10 of the parser rewrite removed it
# from runtime deps. The fuzz target pulls in
# `crates/rsiprtp/tests/parser_diff_oracle/mod.rs` via `#[path]`, and
# that file references `rsip::SipMessage` directly.
rsip = "0.4"
```

…inside the existing `[dependencies]` block (regular dep, not
feature-gated — see §6 risk register for why feature-gating fails here).

And append:

```toml
[[bin]]
name = "sip_message_parse_diff"
path = "fuzz_targets/sip_message_parse_diff.rs"
test = false
doc = false
bench = false
```

The 28 existing `[[bin]]` stanzas stay unchanged. Net: 28 → 29 targets.

### 3.3 `#[path]` adjustment in the moved diff target

The oracle path was relative to `crates/rsiprtp/fuzz/fuzz_targets/`:

```rust
// before — at crates/rsiprtp/fuzz/fuzz_targets/sip_message_parse_diff.rs
#[path = "../../tests/parser_diff_oracle/mod.rs"]
mod oracle;
```

After moving to `fuzz/fuzz_targets/sip_message_parse_diff.rs`, it becomes
the **same shape** as the existing roundtrip target's path:

```rust
// after — at fuzz/fuzz_targets/sip_message_parse_diff.rs
#[path = "../../crates/rsiprtp/tests/parser_diff_oracle/mod.rs"]
mod oracle;
```

(Also update the comment block in the file to reflect the new location.)

### 3.4 Corpus migration

Both fuzz crates' `corpus/` directories are gitignored. The migration is
purely a developer-machine concern; the repo carries no corpus history.

**Recommended developer move:**

```powershell
# from repo root, before deleting the nested crate
if (Test-Path crates\rsiprtp\fuzz\corpus\sip_message_parse_diff) {
  New-Item -ItemType Directory -Path fuzz\corpus\sip_message_parse_diff -Force
  Copy-Item -Recurse -Force crates\rsiprtp\fuzz\corpus\sip_message_parse_diff\* fuzz\corpus\sip_message_parse_diff\
}
```

The seed corpus for `sip_message_parse_diff` is currently empty (no
seeds were added — libFuzzer grew it from scratch during the M11 8h
campaign). For machines without an existing corpus, libFuzzer will seed
from a clean directory and rebuild. This is acceptable: pinned
divergences are encoded in `tests/parser_diff.rs` regression tests, not
the corpus.

The `sip_message_parse` corpus is **abandoned** along with the target
itself. Developers can `rm -rf crates/rsiprtp/fuzz/corpus/sip_message_parse`
locally.

### 3.5 Wrapper updates

#### `fuzz_overnight.ps1`

The auto-discovery (lines 110–123) walks both `fuzz/fuzz_targets/` and
`crates/rsiprtp/fuzz/fuzz_targets/`. After the move, only the first is
populated. The block stays as-is — it's defensive code that's a no-op
when the second dir is missing.

The `sip-diff` profile (lines 56–59) currently lists:

```
sip-diff = (sip_message_parse, sip_message_parse_diff)
```

**Update to:**

```
sip-diff = (sip_message_parse_diff,)
```

(One target, single-element tuple. PowerShell accepts a 1-element array
literal as `@("name")`.)

#### `crates/rsiprtp/tests/fuzz_inventory.rs`

The test scans both target dirs (lines 96–100). The second scan becomes
a no-op when the dir is gone — the `list_target_names` helper handles
`read_dir` failure gracefully (`return vec![]`). **Optional cleanup:**
remove the second `targets.extend(...)` line for clarity.

The test also scans `crates/rsiprtp/fuzz/*.ps1` for wrapper references
(lines 53–61). After the consolidation, the supervisor `.ps1` scripts
under `crates/rsiprtp/fuzz/` are gone — the directory itself is gone —
and that scan becomes another no-op. The wrappers that survive are
`fuzz_overnight.ps1` at the repo root.

**Required cleanup:** the `collect_wrappers` `supervisor_dir` block
should be removed — once `crates/rsiprtp/fuzz/` is gone there are no
wrapper scripts at that path. Leaving it is harmless but stale.

### 3.6 Documentation moves

`crates/rsiprtp/fuzz/M11_LAUNCH.md` is launch-checklist documentation
for the differential campaign. It refers to paths under
`crates/rsiprtp/fuzz/...` that won't exist post-move.

**Two options, in preference order:**

1. **Fold into root `fuzz/README.md`** (preferred). Add an "M11
   differential campaign" section with the launch invocation, updated
   for cwd = repo root and target name `sip_message_parse_diff`. The
   wrapper-driven path (via `fuzz_overnight.ps1 -Profile sip-diff`) is
   the operational one anyway; the manual `cargo +nightly fuzz run`
   form in `M11_LAUNCH.md` is a fallback.
2. **Move to `wrk_docs/`** as a dated journal entry. Less discoverable.

`crates/rsiprtp/fuzz/README.md` documents the nested crate's two
targets. After consolidation, the root `fuzz/` crate has no `README.md`
at all; **this HLD recommends adding one** (separate scope) but does not
require it as a blocker.

### 3.7 What does NOT change

- `crates/rsiprtp/tests/parser_diff_oracle/mod.rs` — stays put, as
  required by the brief.
- `crates/rsiprtp/tests/parser_diff.rs` — stays put, no edit.
- `rsip` listing in `crates/rsiprtp/Cargo.toml` `[dev-dependencies]` —
  the oracle is still consumed by `parser_diff.rs` from `tests/`, so the
  dev-dep stays. (Per the brief: not in scope to retire.)
- The 27 non-SIP-parser targets in root `fuzz/` — untouched.
- `tools/full_test` runner — untouched.

## 4. Validation plan

Run, in order, from the repo root:

1. **Static checks**

   ```powershell
   cargo fmt --check
   cargo clippy --workspace -- -D warnings
   ```

   (cargo-fuzz crates are NOT workspace members — they need separate
   `cargo fmt --manifest-path fuzz/Cargo.toml --check` if we want fmt on
   them. Currently neither fuzz crate is under `cargo fmt`. Out of
   scope.)

2. **Fuzz target builds compile in the new location**

   ```powershell
   cargo +nightly fuzz build sip_message_parse_diff
   cargo +nightly fuzz build sip_message_roundtrip   # regression check
   cargo +nightly fuzz build sip_message             # regression check
   ```

   The second and third confirm the added `rsip` dep didn't break
   non-diff targets.

3. **Smoke run on the moved target**

   ```powershell
   cargo +nightly fuzz run sip_message_parse_diff -- -max_total_time=60
   ```

   Should run for ~60 seconds and exit cleanly (or surface a
   pre-existing pinned divergence — same as before the move). Inputs
   from local seed corpus (if migrated) should reach >0 execs/s within
   the first second.

4. **Existing test consumer of the oracle is unbroken**

   ```powershell
   cargo test -p rsiprtp --test parser_diff
   ```

   Must stay green. (This is the most load-bearing post-move check —
   if the `parser_diff.rs` consumer breaks, the oracle has been
   accidentally moved.)

5. **Inventory test reflects the new layout**

   ```powershell
   cargo test -p rsiprtp --test fuzz_inventory
   ```

   Must stay green. The cleanup in §3.5 should be done as part of the
   implementation step.

6. **Full-suite regression**

   ```powershell
   cargo run --release -p full_test
   ```

   Must stay green end-to-end (fmt, clippy, cargo-deny, build, tests,
   doc, coverage). Coverage may shift slightly due to the
   `sip_message_parse` retirement — `tests/fuzz_inventory.rs` is the
   only test that reads the fuzz tree and it's coverage-neutral.

7. **Wrapper sanity**

   ```powershell
   .\fuzz_overnight.ps1 -Profile sip-diff -BudgetSeconds 90 -SliceSeconds 60
   ```

   Should run `sip_message_parse_diff` once for ~60s, emit a `CLEAN`
   event, then `DONE`. The cwd auto-discovery should resolve to the
   repo root for that target.

## 5. Stages of work

Sequential. Each stage is small enough to commit atomically; no stage
leaves the repo in a broken state.

| stage | description                                                                                                | exits when                                                  |
| :---: | ---------------------------------------------------------------------------------------------------------- | ----------------------------------------------------------- |
|  S1   | Move `sip_message_parse_diff.rs` to `fuzz/fuzz_targets/`. Update its `#[path]`. Add the `[[bin]]` stanza in `fuzz/Cargo.toml`. Add `rsip = "0.4"` to `fuzz/Cargo.toml` `[dependencies]`. | `cargo +nightly fuzz build sip_message_parse_diff` from repo root succeeds. |
|  S2   | Smoke-run the moved target for 60s.                                                                        | Clean exit (or pre-existing pinned-divergence behavior reproduces). |
|  S3   | Update `fuzz_overnight.ps1`: `sip-diff` profile shrinks to one target.                                     | `.\fuzz_overnight.ps1 -Profile sip-diff -BudgetSeconds 90 -SliceSeconds 60` runs cleanly. |
|  S4   | Update `crates/rsiprtp/tests/fuzz_inventory.rs`: drop the second-dir scan, drop the supervisor-dir wrapper scan. | `cargo test -p rsiprtp --test fuzz_inventory` green.       |
|  S5   | Move/fold `M11_LAUNCH.md` content into `fuzz/README.md` (creating the README if needed). Adjust internal cross-references in any other doc that points into `crates/rsiprtp/fuzz/`. | `grep -r "crates/rsiprtp/fuzz" wrk_docs/ wrk_journals/` only hits historical entries (which we leave alone). |
|  S6   | Delete `crates/rsiprtp/fuzz/` (the entire nested crate, including `Cargo.toml`, `.gitignore`, leftover `*.ps1`/`*.log`/`*.pid`/`*.marker` runtime artifacts, `target/`, `corpus/`, `artifacts/`, `M11_LAUNCH.md`, `README.md`). | Directory is gone.                                          |
|  S7   | Final regression: `cargo run --release -p full_test`.                                                      | Stays green. (If it was green before S1, it's green now.)   |

S6 should be a single commit, even though it deletes many files; the
deletion is one logical unit ("retire nested fuzz crate"). Rationale:
keeping S6 atomic makes rollback (revert one commit) trivial.

## 6. Risk register

| # | risk                                                                                          | likelihood | impact | mitigation                                                                                  |
| - | --------------------------------------------------------------------------------------------- | ---------- | ------ | ------------------------------------------------------------------------------------------- |
| 1 | `rsip` dep added to root `fuzz/Cargo.toml` slows non-diff target builds                       | high       | low    | Build cost is one-time per machine (cached). Acceptable — see §2 strongest-argument-AGAINST. |
| 2 | The moved `#[path]` is wrong by one level                                                     | low        | low    | Compile error at `cargo fuzz build`; immediate signal. Existing `sip_message_roundtrip` path is the working precedent. |
| 3 | Other agents have in-flight edits to `crates/rsiprtp/fuzz/`                                   | low        | medium | Land the consolidation in a single PR / commit window. Coordinate via journal entry. The brief says "other agents possibly active" — the deletion is loud (entire dir disappears) and any conflict will surface as a merge conflict, not a silent loss. |
| 4 | Cargo-fuzz workspace boundary issue (root `fuzz/` has empty `[workspace]`)                    | very low   | low    | Empty `[workspace]` already works for 28 targets; adding `rsip` doesn't change resolver behavior. The nested crate's `[workspace] members=["."]` form is an alternate spelling of the same "exclude from parent workspace" pragma. |
| 5 | `fuzz_inventory` test fails after move because the diff target wasn't in `sip-diff` profile   | low        | low    | S3 is sequenced before deletion; the inventory test is in S4. |
| 6 | Lost corpus reduces fuzz effectiveness on next overnight run                                  | medium     | low    | Corpus is rebuildable in <1h of fuzzing for a Tier-1 oracle; pinned divergences are in `parser_diff.rs` regressions, not corpus. Document the migration script (§3.4) for developers who want continuity. |
| 7 | `sip_message_parse` retirement loses some coverage shape that `roundtrip` doesn't exercise    | low        | low    | Both targets use the same `SipMessage::parse` entry point. The roundtrip target additionally re-serializes and re-parses — strict superset of the parse-and-walk-accessors approach. |
| 8 | `rsip` feature-gating could halve build cost ⇒ should we?                                     | n/a        | n/a    | **Considered and rejected.** `cargo-fuzz` doesn't support per-target feature sets; gating affects all targets. The build-cost cost is tolerable; the gating-cost cost is "every contributor learns about an extra feature flag". Net loss. |
| 9 | `cargo run --release -p full_test` broke for an unrelated reason                              | low        | high   | Run S7 against current `main` *before* starting S1 to establish the baseline. If green before, green after. |
| 10 | Root `fuzz/Cargo.lock` churns significantly (rsip pulls 20+ transitive crates)                | high       | low    | Expected — review the lock-file diff in S1's commit, but it's a normal dep addition. |

## 7. Rollback plan

Each stage is a single commit; total expected: 6–7 commits.

- **Anything fails ≤ S5:** `git reset --hard HEAD~N` where N is the
  number of consolidation commits. The repo returns to two-fuzz-crate
  state. (Note Arthur's standing rule: destructive git ops require
  explicit OK — but the rollback is internal-developer-machine only,
  not pushed history.)
- **Anything fails after S6 (mass delete):** `git revert HEAD` of S6
  restores the nested crate verbatim (modulo `corpus/` which is
  gitignored and doesn't roll back — fine, libFuzzer will rebuild).
- **`full_test` fails at S7 for unrelated reason:** investigate
  separately; do *not* roll back the consolidation. The rollback
  criterion is "consolidation broke something", not "anything broke".

The HLD's atomicity choice in S6 is what makes this clean. If the
deletion were spread across files, rollback would require pinpoint
reverts; one commit makes it a single `git revert`.

## 8. Open questions for Arthur

1. **Should `fuzz/README.md` be created as part of this work?** The
   nested crate's `README.md` is going away; the root crate doesn't
   currently have one. I recommend folding the launch instructions for
   `sip_message_parse_diff` into a new root `fuzz/README.md`, but that
   adds scope. **Default if you don't say otherwise:** create the file
   in S5.
2. **Drop the `sip-diff` profile entirely, or shrink it to one target?**
   With `sip_message_parse` retired, the profile holds exactly one
   target (`sip_message_parse_diff`). One could argue it should fold
   into `sip` (the main parser-fuzzing profile). Counter-argument: the
   diff target needs `rsip` linked in and benefits from being scheduled
   on its own slice for triage clarity. **Default:** keep `sip-diff` as
   a one-target profile; treat it as a named slice.
3. **Should `M11_LAUNCH.md` migrate to `wrk_docs/` (as a dated journal /
   reference) or fold into `fuzz/README.md`?** Default: fold into the
   README. Alternative is `wrk_docs/2026.05.06 - Reference - M11
   differential fuzz launch.md`.
4. **Coordinate with other agents first?** The brief notes "other
   agents possibly active in the repo". Should I gate S6 on a journal
   entry + 24h coordination window, or land it directly? Default:
   journal entry + commit, no waiting window — the diff is loud enough
   that conflict detection is automatic.
