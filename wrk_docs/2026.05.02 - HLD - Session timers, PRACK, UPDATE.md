# 2026.05.02 - HLD - Session timers, PRACK, UPDATE

## Goal

Close the carrier-interop gap surfaced by the feature audit: handle
`Require: 100rel` (RFC 3262), allow refresh-without-reINVITE via UPDATE
(RFC 3311), and respond properly to `Session-Expires` so trunks don't
tear our calls down at the ~30-minute mark (RFC 4028).

These three are bundled because they share a single seam — provisional
and mid-dialog request handling — and because real carriers often
require all three together. Each one alone is small; together they are
still small.

## Shape of the change

Three clean additions, no rework of existing code:

1. **PRACK (RFC 3262).** UAS retransmits reliable provisional
   responses until a matching PRACK arrives. UAC sends PRACK when a
   provisional carries `Require: 100rel`. Both sides plumb six new
   headers (`Require`, `Supported`, `RSeq`, `RAck`, `Session-Expires`,
   `Min-SE`) through the existing builder/accessor pattern.
2. **Session timers (RFC 4028).** Calls negotiate an expiry, pick a
   refresher, and the chosen refresher fires UPDATE (or re-INVITE)
   before the deadline. Inbound `Session-Expires < Min-SE` returns 422.
3. **UPDATE (RFC 3311).** A new in-dialog non-INVITE request,
   reusing the existing `NonInvite` transactions. Used internally by
   the session-timer refresher; also accepted from the peer.

The Sans-IO discipline is preserved end to end. PRACK retransmit timing
lives in `InviteServerTransaction`. Session-timer deadlines live on
`Call`, surfaced through a new `CallManager::tick(now)` hook the app
already needs to drive transaction timers anyway.

## Scope decisions (what we are *not* doing)

The three RFCs each have legitimate-but-rare features that are not
needed for carrier interop. Cutting them keeps the change small.

- **No PRACK with SDP body.** RFC 3262 §6 allows offer/answer in PRACK
  for early-media re-negotiation. Carriers do not use this in practice;
  early media is delivered in the 18x SDP we already build. PRACK is a
  pure acknowledgement here.
- **No `Require: 100rel` on our outbound INVITEs.** We advertise
  `Supported: 100rel` so peers may opt in; we never *require* it. There
  is no use case where forcing the peer to be PRACK-capable helps us.
- **No UPDATE with SDP for hold or codec change.** Hold via re-INVITE
  already works. UPDATE-with-SDP is a more efficient path, but a path
  we don't need today. Session-timer refresh uses UPDATE *without* a
  body.
- **No 422 retry loop on outbound.** If our peer rejects our offered
  `Session-Expires` with `422 Session Interval Too Small`, we fail the
  call and surface `CallEndReason::Error`. A single reattempt with the
  peer's `Min-SE` is one more state transition for an edge case real
  carriers don't trigger against sensible defaults (1800/90).
- **No mixing PRACK with forking.** The dialog layer doesn't handle
  forked early dialogs cleanly today; PRACK on a forked early dialog is
  not a use case rsiprtp supports.

If any of these prove necessary later, they slot in on top — none of
the choices here block them.

## Deliverables

1. **`crates/rsiprtp/src/sip/headers.rs`** — six new header types as
   thin parsers, matching the existing `Via`/`Contact` pattern:
   `Require`, `Supported` (option-tag lists, comma-separated),
   `SessionExpires` (delta-seconds + optional `refresher=uac|uas`),
   `MinSe` (delta-seconds), `RSeq` (u32), `RAck`
   (`<rseq> <cseq> <method>`).
2. **`crates/rsiprtp/src/sip/message.rs`** — accessor methods on
   `SipRequest` and `SipResponse` for the same six headers, plus
   `SipRequestBuilder` / `SipResponseBuilder` setter methods. Reads
   route through `rsip::Header::Other` for the headers `rsip` doesn't
   model natively.
3. **`crates/rsiprtp/src/transaction/server/invite.rs`** — reliable
   provisional support:
   - New `send_provisional_reliable(response)` API parallel to the
     existing `send_provisional`. The transaction owns the RSeq
     counter and stamps `Require: 100rel` and `RSeq` onto the response
     itself — callers pass the bare 18x. Rejects `100 Trying`
     (debug-assert; reliable provisional is undefined for 100).
   - New timer `Timer::N` (RFC 3262 — retransmit at T1, doubling, cap
     T2, abandon at 64*T1 per outstanding reliable response).
   - New event input `Event::PrackReceived(u32)` that drops the buffer
     entry for the matching `RSeq`.
   - New event output `Event::PrackTimeout { rseq, final_sent }` when
     Timer N expires without a matching PRACK. `final_sent` lets the
     TU pick the right response per RFC 3262 §3 (504 if no final yet,
     BYE if 2xx already established a dialog, drop silently if a
     non-2xx final has been sent).
   - State stays in the existing `Proceeding` — reliable provisional
     retransmit is bookkeeping inside that state, not a new state.
   - Buffer is drained entirely on *any* final response leaving the
     transaction (2xx → `Terminated`, non-2xx → `Completed`,
     CANCEL→487 → `Completed`).
4. **`crates/rsiprtp/src/dialog/invite.rs`** — three new methods:
   - `build_prack(&SipResponse) -> SipRequest` (mirrors
     `build_ack`).
   - `build_update(session_expires: Option<u32>) -> SipRequest`.
   - `handle_update(&SipRequest) -> SipResponse` returning a 200 OK
     with the dialog's current `Session-Expires`.
5. **`crates/rsiprtp/src/session/call.rs`** — per-call session-timer
   state on `Call`:
   - `session_expires: Option<Duration>` — negotiated value.
   - `min_se: Duration` (default 90s).
   - `refresher: Option<Refresher>` (`Uac` or `Uas`).
   - `refresh_at: Option<Instant>` — set only when *we* are the
     refresher; UPDATE/re-INVITE goes out at this deadline.
   - `expiry_at: Option<Instant>` — set only when the *peer* is the
     refresher; if no refresh arrives by this deadline we BYE the
     call. Both deadlines slide forward on any successful in-dialog
     2xx for UPDATE or INVITE (refresh-driven, hold, re-negotiation —
     anything per RFC 4028 §7).
   - `update_unsupported: bool` — sticky flag set when peer responds
     501/405 to a UPDATE. Subsequent refreshes use re-INVITE.
6. **`crates/rsiprtp/src/session/manager.rs`** — wiring:
   - Outbound INVITE adds `Supported: timer, 100rel`, `Session-Expires`,
     `Min-SE`, and `Allow: ..., PRACK, UPDATE` from `CallConfig`. If
     `CallConfig.session_expires == ZERO`, neither `Supported: timer`
     nor `Session-Expires` is emitted — a single switch suppresses the
     whole RFC 4028 surface so peers don't push timers back at us.
   - 18x with `Require: 100rel` + `RSeq` triggers an internal
     `NonInvite` client transaction carrying our PRACK.
   - 200 OK to INVITE captures negotiated `Session-Expires` and
     `Refresher`. If we are refresher: `refresh_at = now + se/2`. If
     peer is refresher: `expiry_at = now + se`. Never both.
   - Inbound INVITE: if `Require: 100rel` is present, the server
     transaction uses `send_provisional_reliable`; if `Session-Expires
     < Min-SE`, we respond `422 Session Interval Too Small` with our
     `Min-SE`. We honour `Require: timer` implicitly because we always
     emit `Session-Expires` in 200 OK when session timers are enabled.
   - Inbound UPDATE: 200 OK echoing current `Session-Expires` and
     refreshing both deadlines on the call.
   - Any successful in-dialog 2xx (UPDATE or INVITE, inbound or
     outbound) slides both deadlines per RFC 4028 §7 — hold, transfer
     completion, refresh, all count.
   - New `tick(now: Instant)` method that handles two cases per
     `Established` call:
     - `refresh_at <= now` and we are refresher → send UPDATE (or
       re-INVITE if `update_unsupported`).
     - `expiry_at <= now` and peer is refresher → send BYE with
       `Reason: SIP;cause=200;text="Session timer expired"`,
       transition the call to `Terminating`.
     `tick` skips calls not in `Established` and skips refresh if a
     UAC transaction is already in flight on the dialog.
   - Refresh path: UPDATE has no body. The re-INVITE fallback path
     (when `update_unsupported`) MUST include the negotiated SDP
     rebuilt from `MediaSession` — re-INVITE without offer is
     malformed.
   - Existing `drain_events` is unchanged. No new `ManagerEvent`
     variants — refresh and expiry-BYE are internal.
7. **`crates/rsiprtp/src/core/config.rs`** — add to `CallConfig`:
   - `session_expires: Duration` (default 1800s).
   - `min_se: Duration` (default 90s).
   No env vars, no feature flags. `session_expires = Duration::ZERO`
   suppresses both `Supported: timer` and `Session-Expires` on
   outbound and disables tick-driven refresh — a single switch.
8. **`crates/rsiprtp/tests/session_timers.rs`** — integration test:
   two `CallManager`s on loopback, INVITE with `Session-Expires: 60`,
   advance virtual time, observe UPDATE on the wire, observe call
   stays up.
9. **`crates/rsiprtp/tests/prack.rs`** — integration test: server
   responds 180 with `Require: 100rel` + `RSeq: 1`, retransmit
   observed if PRACK is withheld, retransmit stops once PRACK is
   delivered.
10. **README and `lib.rs` doc updates.** PRACK / UPDATE / session
    timers move from undocumented gap to supported. README "Signaling"
    bullet list grows three entries.

## Concrete snags surfaced during HLD review

### 1. Where does PRACK retransmit timing live?

PRACK retransmit is functionally a non-INVITE-server-style timer
(start at T1, double until T2, abandon at 64*T1) but it lives *inside*
the INVITE server transaction. A 200 OK can land at any time and tear
it down. Splitting it into a separate transaction object would mean
two transactions racing for the same response, which is messier than a
single buffer.

The shape is: `InviteServerTransaction` gains a `Vec<(rseq, bytes,
next_interval)>` ordered by RSeq, plus an RSeq counter. `Timer::N`
fires per outstanding entry, restarts itself, and an entry is dropped
on `Event::PrackReceived(rseq)`. The buffer is drained entirely on
*any* final response leaving the transaction — 2xx (which exits to
`Terminated` after ACK), non-2xx (which goes to `Completed`), or
CANCEL→487 (also `Completed`). One rule, no special cases.

### 2. Refresher selection

RFC 4028 §7.4: the UAS picks `refresher=uac` or `uas` in the response,
constrained by `Supported: timer` on each side. We do the simple
thing: as UAS, always pick `uac` when the peer offered `timer`
support, else `uas`. As UAC we accept whatever the peer chose. This
matches the most common deployment (the caller refreshes) and avoids a
config knob.

### 2a. Non-refresher protection (the load-bearing case)

Session timers protect both sides of a call. The mechanism:

- The refresher (us or peer, exactly one) sends UPDATE/re-INVITE at
  half the negotiated interval.
- The non-refresher BYEs the call if no refresh has arrived by the
  full interval. This is the entire point — it catches the case
  where the peer goes silent (router crashes, NAT mapping drops,
  trunk fails over) and the local side is left holding a dead call.

The first draft of this HLD only handled the refresher path. That
covers half the value of session timers and leaves us vulnerable to
exactly the failure mode the feature is supposed to mitigate.

Implementation: each `Call` carries `refresh_at` (only when we
refresh) *or* `expiry_at` (only when peer refreshes), never both. Any
successful in-dialog 2xx for UPDATE or INVITE — refresh-driven, hold,
transfer-completion — slides whichever deadline applies forward by
the full interval. `tick` fires UPDATE on the first deadline, BYE on
the second.

### 3. UPDATE vs re-INVITE for refresh

UPDATE is preferred when both sides support it (RFC 4028 §7). We
detect peer support via `Allow:` on responses; if `UPDATE` is in
`Allow`, refresh uses UPDATE, else re-INVITE. Failure of UPDATE (501,
405) flips `update_unsupported` on the call and subsequent refreshes
use re-INVITE.

UPDATE refresh has no body. The re-INVITE fallback path *requires* a
body — a re-INVITE without offer is malformed per RFC 3261. We rebuild
the offer from the call's negotiated `MediaSession` so the SDP exactly
mirrors the established session (no codec re-negotiation, no port
changes). The peer's 200 OK answer is checked for SDP equivalence and
discarded otherwise; a mismatch on a refresh re-INVITE is treated the
same as a failed refresh and the call is BYEd.

### 4. `Allow` and `Supported` headers

We currently don't emit `Allow` on responses. Adding it is one line in
the response builder (`Allow: INVITE, ACK, BYE, CANCEL, OPTIONS,
PRACK, UPDATE`) and is needed so the peer knows we accept UPDATE for
refresh.

The companion `Supported: timer, 100rel` header on outbound INVITEs is
equally load-bearing: a strict RFC 4028 peer will *not* echo
`Session-Expires` in its 200 OK unless the offer carried
`Supported: timer`. Without it the entire session-timer feature
silently no-ops. Both option-tags go in the same comma-separated
`Supported:` value.

`session_expires == ZERO` suppresses both `Supported: timer` and
`Session-Expires` from outbound INVITEs — one switch, no
half-disabled state.

### 5. CSeq for PRACK

PRACK is a new transaction inside the dialog and gets the next CSeq
value, not the INVITE's CSeq. The `RAck` header carries the original
INVITE's CSeq. This is the one place implementers regularly trip — the
test in `tests/prack.rs` asserts both numbers explicitly.

### 6. `tick()` and `next_deadline()` on CallManager

CallManager has no tick today. The app pumps it by calling
`drain_events()` after handling network input. Adding `tick(now)` is a
new periodic obligation. The example and integration tests are
updated to show the pattern:

```rust
loop {
    tokio::select! {
        msg = transport.recv() => manager.handle_message(msg)?,
        _ = sleep_until(manager.next_deadline().unwrap_or(far_future)) => {}
    }
    manager.tick(Instant::now());
    for event in manager.drain_events() { /* ... */ }
}
```

This is the same shape ICE/transaction timers will eventually need —
session timers just get there first. We do not introduce a generic
timer-action vocabulary to the manager; one deadline-tracker per
concern, queried through one `next_deadline()`, is enough.

## Caller flow

Outbound call, end to end (from the app's perspective):

```
1. config.call_config.session_expires = Duration::from_secs(1800);
2. let call_id = manager.create_call("sip:bob@carrier".to_string());
   // INVITE goes out with Supported: timer, 100rel,
   // Session-Expires: 1800, Min-SE: 90, Allow: ... PRACK, UPDATE.
3. on incoming 183 with Require: 100rel + RSeq:
     manager processes internally:
       - builds PRACK via dialog.build_prack(...)
       - opens a NonInvite client transaction
4. on 200 OK to INVITE:
     manager parses Session-Expires + Refresher from the response.
     If refresher == uac: call.refresh_at = now + se/2.
     If refresher == uas: call.expiry_at  = now + se.
5. app loop calls manager.tick(now) on each wake.
6. when refresh_at elapses (we are refresher):
     manager.tick fires UPDATE (or re-INVITE if update_unsupported).
     On 200 OK, refresh_at slides forward by se/2.
   when expiry_at elapses (peer is refresher and went silent):
     manager.tick sends BYE with Reason: SIP;cause=200;text="Session
     timer expired", call moves to Terminating.
   when peer refreshes us (inbound UPDATE/re-INVITE, 200 OK):
     expiry_at slides forward by se. No tick action needed.
```

Inbound is the mirror: handle PRACK arrivals, build 200 OK to UPDATE,
respond 422 if `Session-Expires < Min-SE`, slide deadlines on every
successful in-dialog 2xx.

## Design choices and rejections

- **No PRACK with SDP.** Justified above.
- **No `Require: 100rel` on outbound.** Justified above.
- **No outbound 422 retry loop.** Justified above.
- **`tick()` over background task.** A spawned task pumping a
  `CallManager` would force an `Arc<Mutex<CallManager>>` and break
  Sans-IO. The app's event loop already wakes for transport input and
  has the right context to call `tick`.
- **Session timers always offered (when `session_expires > 0`).** No
  per-call opt-in API. Apps that don't want them set
  `session_expires = ZERO` once on the manager config; carriers that
  ignore the offer simply don't echo it. `ZERO` suppresses both
  `Supported: timer` *and* `Session-Expires` so peers don't push
  timers back at us — one switch, no half-disabled state.
- **Two deadlines per call, never both.** Refresher state is binary
  (us or peer). `refresh_at` and `expiry_at` are mutually exclusive
  per call. Keeps the tick logic obvious and avoids tracking a "did
  we already refresh this interval?" flag.
- **Refresher picked by UAS deterministically.** No knob.
- **PRACK retransmit lives in `InviteServerTransaction`, not a new
  transaction type.** A 200 OK racing a PRACK retransmit is one buffer
  to drain, not two transactions to reconcile.
- **`Allow` always emitted on responses.** One line. Avoids the
  refresher-negotiation footgun.
- **`Refresher` is a `core::config` type, not exported through
  `prelude`.** Apps don't pick it; we do. Keeps the prelude surface
  honest.

## Testing

- **Unit tests** in `sip/headers.rs`: round-trip each new header
  through parse → display → parse, with at least one malformed input
  per type.
- **Unit tests** in `transaction/server/invite.rs`:
  - Send 180 with `send_provisional_reliable`, advance time, assert
    retransmit; deliver `Event::PrackReceived(1)`, assert no further
    retransmit.
  - No PRACK in 64*T1 with no final sent → `PrackTimeout { final_sent:
    None }`. Repeat with non-2xx final sent → `final_sent: NonTwoXx`.
    Repeat with 2xx final sent → `final_sent: TwoXx`.
  - Final response (any kind) drains the entire PRACK buffer.
  - `send_provisional_reliable(100)` rejects (debug-assert).
- **Unit tests** in `dialog/invite.rs`: `build_prack` emits a request
  with the right RAck, CSeq is the dialog's next, To/From/Call-ID
  match the dialog. `handle_update` returns 200 with current
  `Session-Expires`.
- **Integration test** `tests/prack.rs`: two managers on loopback, UAS
  uses reliable provisional, capture PRACK on the wire, verify
  retransmit-then-stop behaviour.
- **Integration test** `tests/session_timers.rs`, three scenarios in
  one file:
  - *Refresher path*: two managers, INVITE with `Session-Expires: 60`
    where we are refresher, advance time past 30s, assert UPDATE on
    the wire, assert 200 OK, assert call stays up.
  - *Non-refresher BYE path* (load-bearing): INVITE with peer as
    refresher, advance time past 60s without delivering a refresh,
    assert BYE on the wire with the `Reason: ... Session timer
    expired` header, assert call ends.
  - *UPDATE-not-supported fallback*: peer's 200 OK omits UPDATE from
    `Allow`, advance to refresh deadline, assert refresh goes out as
    re-INVITE *with* SDP body matching the negotiated session.
- **Integration test** for 422: outbound INVITE with
  `Session-Expires: 30, Min-SE: 30`, peer rejects with
  `422 Session Interval Too Small, Min-SE: 1800`, assert call ends
  with `CallEndReason::Error`.
- **Integration test** for inbound refresh sliding `expiry_at`: peer
  is refresher, advance to 0.5*se, peer sends UPDATE, assert no BYE
  fires when we then advance past the original `expiry_at`.

The two end-to-end integration tests are the load-bearing oracles.
They encode "PRACK works on the wire" and "session timers refresh on
the wire" in a form a machine can verify.

## Risks

| Risk | Likelihood | Impact | Mitigation |
|------|------------|--------|------------|
| PRACK retransmit timer interacts badly with Timer G (UAS retransmitting a final response) once both are running | Low | Med — could double-send | Final response cancels the PRACK buffer entirely on entry to `Completed`; covered by a unit test |
| `rsip` doesn't model `Require`/`Supported`/`RSeq`/`RAck`/`Session-Expires`/`Min-SE` natively, forcing string round-trips through `rsip::Header::Other` | High | Low | This is already how the codebase handles non-standard headers; new accessors thin-wrap the same pattern |
| Session-timer refresh fires while a re-INVITE for hold/transfer is in flight | Med | Med — duplicate in-dialog request collision | `tick()` skips refresh if the dialog already has a pending UAC transaction. The refresh slips by one tick interval, which is fine |
| Carrier echoes our `Session-Expires` but omits `refresher`, leaving the role ambiguous | Med | Low | Default to UAC-as-refresher when the peer didn't pick — RFC 4028 §7.1 |
| `tick()` becomes the home for *every* future periodic concern and bloats | Low | Low | Document that `tick` only handles session-timer refresh; future deadlines (registration refresh, ICE consent freshness) get their own surface if needed |
| `CallEndReason::Error` for 422 hides the cause from the app | Low | Low | Log the response status and reason at warn level; app can still diagnose from logs. Not worth a new `CallEndReason` variant for an edge case |
| Always emitting `Allow` on responses changes wire output; existing snapshot/byte-compare tests may break | Med | Low — fix at landing | Audit response-output assertions in phase 1 before committing; the diff is mechanical (one new header line) |
| Re-INVITE refresh body diverges from established SDP and peer answers with a different codec set, breaking the call mid-stream | Low | High | Refresh re-INVITE rebuilds the offer verbatim from `MediaSession`; if the answer's codec list isn't a subset of what's already running, treat as failed refresh and BYE |
| Forgetting `Supported: timer` in the outbound INVITE silently disables the entire feature against strict peers | Was high in v1 | High | Fixed: `Supported: timer` is emitted alongside `100rel`; integration test asserts it on the wire |

## Out of scope (explicit)

- PRACK with SDP body for early-media re-negotiation (RFC 3262 §6).
- UPDATE with SDP body for hold/codec change.
- Outbound 422 retry with peer's `Min-SE`.
- PRACK on forked early dialogs.
