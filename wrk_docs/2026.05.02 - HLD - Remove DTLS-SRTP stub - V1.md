# HLD — Remove DTLS-SRTP stub

**Date:** 2026-05-02
**Author:** Arthur + Claude
**Status:** Proposed
**Related:** `wrk_docs/2026.05.01 - Brief - DTLS-SRTP decision.md`

## Goal

Delete `crates/rsiprtp/src/srtp/dtls.rs` (990 lines, no working handshake)
and every reference to it from the published surface, docs, and fuzz
harness. After this HLD lands, the README's "DTLS-SRTP framing types are
present" hedge goes away and the `srtp` module advertises only what it
actually does: SDES key exchange (RFC 4568) feeding `SrtpContext`.

## Decision: Option A, not A.5

The brief offered Option A (remove everything) and Option A.5 (keep
`Fingerprint` and `SrtpProfile` as "genuinely useful primitives" for
external users rolling their own DTLS).

**Pick A.** Justification:

- Grep across the repo confirms zero internal callers of `Fingerprint`,
  `SrtpProfile`, `DtlsRole`, or any other type in `srtp::dtls`. The only
  references outside `dtls.rs` itself are the `pub use` block in
  `srtp/mod.rs`, one fuzz target, one doc-comment in `lib.rs`, and one
  scope bullet in `README.md`. Nothing depends on these types.
- A.5 keeps types for the benefit of hypothetical external users. We have
  no such user. Per CLAUDE.md: "premature abstractions don't just waste
  effort — they actively resist the correct design when it arrives." When
  someone implements DTLS-SRTP for real, they'll want `Fingerprint` to be
  shaped to the chosen DTLS crate's certificate API, not the shape we
  guessed at in 2024.
- Re-adding ~80 lines of `Fingerprint` + `SrtpProfile` from git history is
  trivial if a future caller actually needs them.
- This is a development sandbox; no migration story to preserve.

A.5 buys nothing real and costs the same maintenance attention as A.

## Scope

### In scope (this HLD)

- Delete `crates/rsiprtp/src/srtp/dtls.rs` in its entirety.
- Drop the `pub mod dtls;` declaration and the `pub use dtls::{...};`
  re-export from `crates/rsiprtp/src/srtp/mod.rs`.
- Remove the `srtp_use_ext` fuzz target (its only call,
  `srtp::dtls::parse_use_srtp_extension`, is gone).
- Trim crate-level docs in `crates/rsiprtp/src/lib.rs` to drop the
  parenthetical "DTLS-SRTP framing types are present, but the DTLS
  handshake itself is not yet implemented."
- Trim the README scope bullet to a single honest line: WebRTC interop is
  out of scope; SRTP is reachable via SDES key exchange only.
- Note the removal in `CHANGELOG.md` under `[Unreleased] / Removed`.
- Leave `tests/baresip_integration/scenarios/security.rs` as-is. Its
  `test_dtls_srtp_simulation` only prints log lines and asserts an
  ordinary call established — it imports nothing from `srtp::dtls`. The
  comments are misleading but harmless and out of this HLD's scope.

### Explicitly out of scope

- Implementing real DTLS-SRTP (Option B in the brief). That is a separate
  brief + HLD + spike when there is a customer asking for it.
- Touching SDES (`srtp::sdes`), the SRTP context (`srtp::context`), or
  KDF (`srtp::kdf`). They are unrelated and fully working.
- Renaming or restructuring the `srtp` module. After deletion it becomes
  three submodules — `context`, `kdf`, `sdes` — and that's fine.
- Refactoring `security.rs` integration tests. Those are simulation
  placeholders; rewriting them is its own task.

## File-by-file changes

### 1. `crates/rsiprtp/src/srtp/dtls.rs`

**Delete the file.** All 990 lines, including the 30+ unit tests inside
`#[cfg(test)] mod tests`. Tests only exercised the dead surface; deleting
them with the surface is correct.

### 2. `crates/rsiprtp/src/srtp/mod.rs`

Before:

```rust
mod context;
pub mod dtls;
mod kdf;
mod sdes;

pub use context::{SrtcpContext, SrtpContext};
pub use dtls::{
    DtlsConfig, DtlsError, DtlsRole, DtlsSrtpKeys, DtlsState, Fingerprint, FingerprintHash,
    SrtpProfile,
};
pub use kdf::{CryptoSuite, SessionKeys};
pub use sdes::SdesAttribute;
```

After:

```rust
mod context;
mod kdf;
mod sdes;

pub use context::{SrtcpContext, SrtpContext};
pub use kdf::{CryptoSuite, SessionKeys};
pub use sdes::SdesAttribute;
```

Module-level doc comment is already SDES-focused (RFC 3711 + RFC 4568) and
needs no changes.

### 3. `crates/rsiprtp/src/lib.rs`

Replace this in the crate-level doc:

```text
- [`srtp`]: SRTP encryption (SDES key exchange; DTLS-SRTP framing types
  are present, but the DTLS handshake itself is not yet implemented)
```

with:

```text
- [`srtp`]: SRTP encryption with SDES key exchange (RFC 4568)
```

The earlier paragraph that says "there is no DTLS-SRTP handshake (only
SDES key exchange)" stays — it's now exactly correct rather than
half-correct.

### 4. `README.md`

Replace the WebRTC-interop bullet under **Scope** (currently:
"DTLS-SRTP is present as framing types only — the DTLS handshake itself
is not implemented…") with:

```markdown
- **WebRTC interop.** No DTLS-SRTP. SRTP is reachable via SDES key
  exchange (RFC 4568), which works against SIP carriers but not against
  browsers.
```

Also update the architecture table on line 120 — the inline tag
"SRTP-SDES" stays (already accurate); no other README change required.

### 5. `fuzz/fuzz_targets/srtp_use_ext.rs`

Delete the file. Then remove the corresponding `[[bin]]` entry from
`fuzz/Cargo.toml` (whichever section declares this target). Verify with
`cargo check --manifest-path fuzz/Cargo.toml` that no other fuzz target
references `srtp::dtls`.

### 6. `crates/rsiprtp/CHANGELOG.md`

Under `## [Unreleased]` → `### Removed`, add:

```markdown
- **DTLS-SRTP stub** (`rsiprtp::srtp::dtls`). The module never contained a
  DTLS handshake — only fingerprint parsing, role enums, and a
  use-srtp-extension codec. It is removed rather than completed; SRTP key
  exchange is via SDES only. If DTLS-SRTP support arrives later it will be
  designed against an actual DTLS crate, not retrofitted onto these types.
```

The previous 0.2.0 entry already noted the `dtls` *feature flag* removal;
this entry covers the *types* themselves, which 0.2.0 left in place.

### 7. `crates/rsiprtp/Cargo.toml` — version bump

Patch bump only — no SemVer minor needed because we are pre-1.0 and
`### Removed` under `[Unreleased]` already covers the surface change.
Bump `version = "0.2.0"` → `"0.2.1"`.

### 8. `crates/rsiprtp/public-api.txt`

This file is generated by `cargo public-api`. Regenerate after the build
passes:

```bash
cargo public-api -p rsiprtp > crates/rsiprtp/public-api.txt
```

Expect ~50 fewer entries (everything currently in lines 14604–14*** under
`pub mod rsiprtp::srtp::dtls`).

## Public API impact

Removed from `rsiprtp::srtp`:

| Type / fn | Note |
|---|---|
| `DtlsConfig` | Dead struct, no consumer. |
| `DtlsError` | Carried `OpenSsl` variant for a feature that no longer exists. |
| `DtlsRole` | Useful only with a real handshake. |
| `DtlsSrtpKeys` | Designed around a hypothetical exporter. |
| `DtlsState` | Five enum variants for a state machine that doesn't exist. |
| `Fingerprint` | SHA-1/SHA-256 cert fingerprint parse/format. |
| `FingerprintHash` | Two-variant enum used only by `Fingerprint`. |
| `SrtpProfile` | RFC 5764 protection profile IDs. |
| `extract_srtp_keys` | Splits a hypothetical exported buffer. |
| `parse_use_srtp_extension` | DTLS-SRTP wire-format reader. |
| `build_use_srtp_extension` | DTLS-SRTP wire-format writer. |

Module path `rsiprtp::srtp::dtls` itself disappears.

Surviving `rsiprtp::srtp` surface (unchanged):
`SrtpContext`, `SrtcpContext`, `CryptoSuite`, `SessionKeys`,
`SdesAttribute`.

No `Deprecated` step. The brief and the user are explicit: dev sandbox,
delete cleanly, no transition layer.

## Validation

```bash
cargo build --workspace
cargo clippy --workspace --all-targets -- -D warnings
cargo test --workspace
cargo public-api -p rsiprtp --diff-git-checkouts main HEAD
```

Acceptance:

- `cargo build` passes with no new warnings.
- `cargo clippy -D warnings` is clean.
- `cargo test --workspace` passes. The 30+ tests inside `dtls.rs::tests`
  go away with the file; that is expected and correct (you can't
  meaningfully test a stub that no longer exists). The
  `security.rs::test_dtls_srtp_simulation` integration test continues to
  pass — it never touched the `srtp::dtls` types.
- `cargo public-api --diff-git-checkouts` shows only `Removed` entries,
  all under `pub mod rsiprtp::srtp::dtls`, plus the file-level removal
  itself. No `Changed` entries.
- Grep for `dtls`, `Dtls`, `DTLS-SRTP`, `srtp::dtls` across `crates/`,
  `fuzz/`, `README.md`, and `CHANGELOG.md` returns only:
  - The CHANGELOG entry added by step 6.
  - The integration-test simulation comments in
    `tests/baresip_integration/scenarios/security.rs` (intentionally left).
  - The brief itself at `wrk_docs/2026.05.01 - Brief - DTLS-SRTP decision.md`
    (historical record).

## Risk

Low. There are no internal callers, no external published consumers
(rsiprtp 0.2.0 was published 2026-05-01), and the stub doesn't actually
do anything that needs preserving. The only foreseeable surprise is the
absent `cargo public-api` diff entry: a maintainer running it for the
first time after this change will see ~50 lines of `Removed`. The
CHANGELOG entry covers that.

## Estimate

~1–2 hours of work, including the public-api regeneration and a clean
`cargo test --workspace` run. Matches the brief's "1 day" upper bound
with margin.

## Open questions

None for the implementer. Arthur's "do not support the old code, no
feature flags, dev environment" guidance closes every ambiguity the brief
left open.
