### HLD — RFC 4475 fixtures: byte-perfect upgrade

**Date:** 2026-05-06
**Author:** Claude (with Arthur)
**Status:** Draft for review
**Related:** `crates/rsiprtp/tests/fixtures/rfc4475/README.md`,
`crates/rsiprtp/tests/parser_diff.rs`,
`crates/rsiprtp/tests/parser_roundtrip.rs`,
`crates/rsiprtp/tests/allocations_sip_parse.rs`

## Context

`crates/rsiprtp/tests/fixtures/rfc4475/` currently holds 9 SIP
messages that exercise RFC 4475 §3 ("Valid Messages") torture-test
categories. The fixtures' own README is explicit about what they
are:

> These fixtures are *representative* messages constructed per the
> categories in §3 — each fixture exercises the same parser corner
> the RFC §3.1 paragraph describes, not necessarily the byte-perfect
> example from the RFC body.

So today we say "we exercise the same parser corner RFC 4475 §3.1.X
describes". RFC 4475 Appendix A.1 publishes canonical wire bytes
for each §3 message as base64 blobs (the §3 prose shows readable
forms; the appendix is the authoritative wire form). Replacing our
representative fixtures with the byte-perfect A.1 bytes lets us say
something stronger and externally verifiable: "we accept the
RFC 4475 §3 corpus byte-for-byte".

The cost is a one-shot decode-and-replace plus a small amount of
fallout. The harness that consumes the fixtures (`parser_diff.rs`,
`parser_roundtrip.rs`, `allocations_sip_parse.rs`) does not embed
specific byte sequences — it asserts properties (both parsers
agree, parse/serialize is a fixed point, allocations stay within
budget). Most tests will pass unchanged.

## Goals

- Replace each fixture in `crates/rsiprtp/tests/fixtures/rfc4475/`
  with the byte-perfect RFC 4475 §A.1 bytes for the corresponding
  §3.1.X message.
- Update the fixture README to drop the "representative" caveat and
  state that the fixtures are now the canonical RFC 4475 §A.1
  messages.
- Pin each fixture to the RFC bytes by including a SHA-256 in the
  README, so a future reviewer can reproduce the decode.
- Keep all existing tests passing (after re-measuring any
  drift-sensitive numbers, e.g. the `longreq` allocation budget).

## Non-goals

- Adding new §3 fixtures. The RFC 4475 §3 sub-sections we
  intentionally skip (`intmeth`, `noreason`) stay skipped — our
  closed-set `Method` enum rejects exotic tokens by design and
  byte-perfect fixtures don't change that.
- Adding new §4 invalid fixtures. We have two (`badaspec_*`); RFC
  4475 §4 has many more (`ncl`, `scalar*`, `lwsruri`, `badinv01`,
  `regbadct`, …). Out of scope here; that's a separate corpus
  expansion.
- Changing the test harness. `parser_diff.rs`, `parser_roundtrip.rs`,
  and `allocations_sip_parse.rs` already accept the fixtures by
  filename and assert properties, not specific bytes.
- Adding any new feature flags or environment variables. Fixtures
  are the authoritative artifacts; checked in directly.

## What's already in place

- `crates/rsiprtp/tests/fixtures/rfc4475/{wsinv,esc01,escnull,esc02,lwsdisp,longreq,dblreq,semiuri,transports,unreason}.sip`
  — the 9 representative fixtures, with filenames already matching
  the RFC 4475 §A.1 names. We replace bytes in place; no rename.
- `parser_diff.rs:137-298` — one test per fixture; mostly calls
  `assert_equivalent(bytes)` (both parsers agree). Four fixtures
  have pinned-divergence tests against rsip 0.4 (`wsinv`, `dblreq`,
  `semiuri`, `transports`) — those tests assert the asymmetry
  itself, not specific byte content.
- `parser_roundtrip.rs:95-148` — `assert_roundtrip_fixed_point`
  on each fixture: parse → serialize → parse must reach a fixed
  point on the second pass.
- `allocations_sip_parse.rs:88-159` — `longreq.sip` is bound to a
  per-message allocation budget (`LONGREQ_INVITE_ALLOC_BUDGET = 55`,
  measured 47 + 15 % headroom on 2026-05-06). This is the one
  numeric assertion that may shift when we swap to byte-perfect.
- `crates/rsiprtp/tests/fixtures/rfc4475/README.md` — fixture
  catalog, RFC §-references, pinned-divergence catalog.

## Approach

One commit; no flags, no migration step.

1. Decode RFC 4475 Appendix A.1. **A.1 is a single base64 blob,
   not per-message blobs.** Decoding produces a gzip-compressed
   tar archive whose members are the per-message files (`wsinv`,
   `esc01`, …, plus the §4 invalid set and the §3.1.2.1 / §3.1.2.11
   ones we currently skip). The RFC publishes a Perl one-liner
   that does base64 → gunzip → untar in §A. This is the canonical
   procedure.
2. Extract the 10 tar members that correspond to the §3 fixtures
   we currently keep (`wsinv`, `esc01`, `escnull`, `esc02`,
   `lwsdisp`, `longreq`, `dblreq`, `semiuri`, `transports`,
   `unreason`) and overwrite each
   `crates/rsiprtp/tests/fixtures/rfc4475/*.sip` file with the
   tar member's bytes verbatim. Filenames already match tar
   member names, so this is purely content replacement.
3. **Add `.gitattributes` rules so the `.sip` bytes are preserved
   across platforms** — see the dedicated section below. Without
   this step the byte-perfect claim is false in practice.
4. Re-run the affected test families. Expect three classes of
   fallout:
   - **Most tests pass unchanged** — `assert_equivalent`,
     `assert_roundtrip_fixed_point`, and the rsip-rejection pins
     all assert properties, not byte sequences.
   - **`longreq` allocation budget may shift** — the byte-perfect
     `longreq` may be slightly larger or smaller than ours.
     Re-measure, update `LONGREQ_INVITE_ALLOC_BUDGET`, and update
     the comment that documents the measured baseline (currently
     `// Measured 2026-05-06: 47 allocations`).
   - **A pinned-divergence test may flip** if rsip 0.4 happens to
     accept/reject one of the byte-perfect inputs differently from
     our representative one. If so: update the assertion to match
     observed behavior, update the README divergence table, and
     keep the test (an asymmetry that exists with the canonical
     bytes is more credible than one we constructed).
5. Update `crates/rsiprtp/tests/fixtures/rfc4475/README.md`:
   - Replace the "representative messages constructed per…" lead
     paragraph with "canonical bytes decoded from RFC 4475 §A.1".
   - Drop the parenthetical "not necessarily the byte-perfect
     example" caveat.
   - Add a `SHA-256` column to the fixture table so a reviewer
     can reproduce the decode (`sha256sum *.sip`).
   - Adjust per-row descriptions if the byte-perfect content
     exposes a corner not described in the current row text.
6. Update inline test-doc comments where they say "representative"
   or "constructed per". The wsinv test header (`parser_diff.rs:115`)
   currently says "we are testing the same parser corner the RFC
   §3.1 paragraph describes, not asserting byte-perfect copies of
   the RFC's own example bodies" — flip that to byte-perfect.

## Line-endings: the prerequisite I almost missed

The current fixtures look fine on disk but **are not byte-stable
across platforms today**. Verified state:

- No `.gitattributes` anywhere in the repo.
- `git config core.autocrlf` → `true` on this Windows checkout.
- `git ls-files --eol crates/rsiprtp/tests/fixtures/rfc4475/wsinv.sip`
  → `i/lf  w/crlf` — git stores LF in the index, materializes CRLF
  on Windows, materializes LF on Linux.

That means the README's claim ("Each `.sip` file contains literal
CRLF-terminated bytes") is true on Windows working trees only.
`include_bytes!` reads from the working tree, so on Linux CI the
exact bytes our parser sees are LF-terminated, not CRLF-terminated.
Tests pass anyway because our parser also accepts LF (per the M2-A
permissive policy), but the round-trip and diff harnesses have
been silently exercising different bytes on Linux vs Windows.

For "byte-perfect" to mean anything — and for the SHA-256 column
in the README to be reproducible on any host — we must:

1. Add a top-level `.gitattributes` (or scoped to the fixtures
   tree) with:
   ```
   crates/rsiprtp/tests/fixtures/rfc4475/*.sip   -text
   crates/rsiprtp/tests/fixtures/rfc4475_invalid/*.sip   -text
   crates/rsiprtp/tests/fixtures/handcrafted/*.sip   -text
   ```
   The `-text` attribute disables all conversion (line-endings,
   working-tree encoding) so the bytes round-trip git verbatim.
2. After landing the attribute and the byte-perfect content,
   run `git add --renormalize .` to force the index to match the
   working-tree bytes. Verify with
   `git ls-files --eol …/wsinv.sip` showing `i/-text w/-text`.

This step is non-optional for this HLD; without it the SHA-256s
in the README would be wrong on Linux, and the byte-perfect claim
would silently fail on contributor machines using non-Windows
hosts.

Note: this also closes a latent inconsistency that exists in the
*current* (representative) fixtures, where `parser_roundtrip.rs`
has been comparing two slightly different byte streams depending
on host. So we're fixing two things at once.

## Implementation plan

Single PR. Steps in order:

1. **Check errata.** Open
   `https://www.rfc-editor.org/errata_search.php?rfc=4475` in a
   browser (the search is JS-driven so WebFetch can't see it).
   Note any errata that affect the encoded reference messages and
   apply them to the decoded bytes before locking fixtures in.
2. **Decode A.1.** Extract the base64 block between
   `-- BEGIN MESSAGE ARCHIVE --` and `-- END MESSAGE ARCHIVE --`
   from the RFC text; pipe through `base64 -d | gunzip | tar x`.
   This produces the per-message files including all of `wsinv`,
   `intmeth`, `esc01`, `escnull`, `esc02`, `lwsdisp`, `longreq`,
   `dblreq`, `semiuri`, `transports`, `unreason`, `noreason`, and
   the §4 invalid set. The decode commands themselves aren't
   committed; the decoded bytes are the artifact.
3. **Add `.gitattributes`** with the `*.sip -text` rules described
   above. Commit this *before* the content swap so the working
   tree starts taking bytes verbatim.
4. **Overwrite the 10 §3 fixture files** with the corresponding
   tar member bytes. Spot-check with `xxd | head` and `xxd | tail`
   to confirm CRLF terminators and the trailing `\r\n\r\n` body
   separator are intact.
5. **Renormalize the index:** `git add --renormalize
   crates/rsiprtp/tests/fixtures/`. Verify with
   `git ls-files --eol crates/rsiprtp/tests/fixtures/rfc4475/*.sip`
   showing `i/-text w/-text` for every file.
6. **Compute SHA-256** for each new `.sip` file. These go in the
   README table for reproducibility.
7. **Run the test bar.** `cargo test -p rsiprtp` covers all three
   fixture harnesses. Note any failures and classify into the three
   classes documented above.
8. **Fix fallout.** Re-measure the `longreq` allocation count if
   it drifted; update `LONGREQ_INVITE_ALLOC_BUDGET` and the dated
   `// Measured …` comment. If a pinned-divergence test flipped,
   update assertion + README pin table; if a previously-passing
   `assert_equivalent` now fails, investigate whether the
   byte-perfect fixture exposes a real divergence (in which case
   it joins the pin table) or a real parser bug (fix before
   merging).
9. **Update README + inline test comments.** "Representative" →
   "canonical RFC 4475 §A.1". Add the SHA-256 column. Note in the
   README that bytes are protected by `.gitattributes -text`.
10. **Run `cargo run --release -p full_test`** for the full bar
   (fmt, clippy, cargo-deny, build, tests, doc, coverage).

## Risks

- **A.1 decode mishap.** The base64 in the RFC text spans multiple
  lines and may be wrapped; trailing/leading whitespace must be
  stripped before decode but CRLFs *within* the message bytes
  must be preserved. Mitigation: validate by `xxd`-comparing the
  first/last 32 bytes of each decoded message against the §3.1.X
  prose form. Spot-check that every line ends `\r\n` and the
  trailing `\r\n\r\n` body separator is intact.
- **Hidden new divergence.** A byte-perfect fixture may surface
  a corner our representative version smoothed over. If our
  parser now rejects, that's a parser bug to fix; if rsip 0.4
  rejects, that's a new pinned divergence (our existing pin
  catalog is the right place for it). Either way: visible
  failure surfaced by tests, not silent.
- **Allocation budget drift.** Most likely failure mode. The fix
  is mechanical (re-measure, bump constant, bump dated comment).
  Already accounted for in the plan.
- **None of the above breaks `parser_diff_oracle`.** The fuzz
  oracle's known-asymmetry heuristics (M11 findings #6, #11,
  #12, #13, #14) are about wire shapes, not specific fixtures —
  they'll keep working unchanged.
- **Fuzz seed corpus is unaffected.** Verified by grep: nothing
  under `fuzz/` references `tests/fixtures/rfc4475` or the
  individual fixture filenames. The fuzz targets get their seeds
  elsewhere; swapping fixture bytes won't shift fuzz coverage.

## Decisions to confirm with Arthur before implementing

The A.1 tar contains more than the 10 fixtures we use today.
Once decoded, these are essentially free to add — the cost is a
few lines of test code and a row in the pin table per fixture.
Calling them out as explicit decisions rather than burying them
under "out of scope":

1. **Add `intmeth` (§3.1.2.1) and `noreason` (§3.1.2.11)?** We
   currently skip both because our `Method` enum is closed-set
   per RFC 3261 §7.1, so we reject exotic method tokens by
   design. The byte-perfect upgrade doesn't change that, but it
   does make a "documented rejection" assertion (we reject; rsip
   may accept) cheap to write. Recommend **yes** — they cost
   ~10 LOC each and they close the §3 conformance gap explicitly,
   which is the entire pitch of this change. They become both-
   reject or asymmetric-pin tests in `parser_diff.rs` rather than
   `assert_equivalent` tests.
2. **Pull more §4 fixtures from the same tar?** The tar contains
   the full §4 invalid corpus (`ncl`, `scalar02`, `scalarlg`,
   `lwsruri`, `badinv01`, `regbadct`, etc.). Recommend **no, not
   in this change** — the §4 set raises real questions per fixture
   (do both parsers reject for the right reason? does our parser
   reject at framing or at typed-form?) that deserve their own
   HLD. Worth a follow-up.
3. **Replace the existing `badaspec_*` fixtures with their tar
   counterparts?** Marginal value — the existing two are
   structurally identical to the tar versions for our framing
   tests. Recommend **no** unless step 2 is also a yes.

## Out of scope (explicitly)

- Pulling the broader §4 invalid corpus (decision 2 above).
- Test harness changes. The harness already does the right thing
  with whatever bytes the fixtures contain.
- Any rename or restructure of fixture filenames. They already
  match A.1 / tar member names.
- Reworking how `include_bytes!` reads fixtures. The
  `.gitattributes` change above is sufficient to make the bytes
  byte-stable; the harness keeps using `include_bytes!`
  unchanged.
