# SIP/RTP library review: mdsiprtp vs mdsiprtp2 vs mdsiprtp3

Date: 2026.01.13

## Executive summary

- mdsiprtp is the most complete and production leaning of the three. It is modular, has strong protocol coverage (SIP, SDP, RTP, SRTP, ICE), and includes extensive tests and integration scaffolding. The tradeoff is complexity and a heavier dependency footprint.
- mdsiprtp2 is a clean workspace with strong engineering hygiene (toolchain pinning, scripts, ADRs) and a reasonable SIP core, but it is mostly a skeleton for media, NAT, and services.
- mdsiprtp3 is a monolithic prototype with a decent amount of SIP and RTP logic, but it is incomplete in key areas (transport, NAT, SRTP, B2BUA, mixing) and docs overstate readiness.

## At a glance comparison

| Area | mdsiprtp | mdsiprtp2 | mdsiprtp3 |
| --- | --- | --- | --- |
| Architecture | Multi-crate, layered stack; Sans-IO transaction layer | Workspace with core and service crates; daemon style bins | Single crate, all modules in one tree |
| SIP core | Wraps rsip; dialogs, transactions, session management | Basic parser, transaction engine, dialog/session timers | Custom parser/headers/transactions; UAC/UAS |
| RTP/media | RTP/RTCP, jitter, DTMF, mixer, G.711/G.722, optional Opus | Media core is placeholder; RTP not implemented | RTP/RTCP, jitter, G.711, DTMF, mixer TODO |
| NAT/Security | ICE/STUN/TURN, SRTP, DTLS, TLS transport | NAT traversal placeholder; TLS/WS stubs | No ICE/SRTP; UDP only; TCP/TLS TODO |
| Testing | Unit + extensive integration (Asterisk, Baresip) | Unit and property tests, small integration tests | Unit tests only, no integration suite |
| Docs/ops | README, dockerized integration setup, examples | ADRs, scripts (SBOM, coverage), pinned toolchain | Status docs, mixed accuracy vs code |
| Maturity | Highest | Early framework | Mid prototype |

## mdsiprtp

Pros
- Strong modularity across crates: sip, sdp, rtp, srtp, ice, media, transport, session.
- Good protocol depth: SIP transactions and dialogs, RTP/RTCP, SRTP (SDES and DTLS), ICE/STUN/TURN, TLS transport.
- Sans-IO transaction logic makes core state machines deterministic and testable.
- Broad test surface with integration suites (Asterisk, Baresip) and targeted security and resilience tests.
- Practical examples and an app (Gabby) demonstrate end to end usage.

Cons
- Large dependency footprint and workspace complexity, especially with Gabby in the same repo.
- No pinned toolchain or workspace quality gates like the mdsiprtp2 setup.
- Some integration test scaffolding still has TODOs (media validation and auth challenge flow), suggesting remaining gaps.
- Heavier lift if you want only a thin SIP or RTP subset; modularity helps, but the repo is still big.

## mdsiprtp2

Pros
- Clean workspace layout with explicit crates for core, services, media, NAT traversal, and tooling.
- Good engineering hygiene: pinned toolchain, scripts for SBOM and coverage, ADRs for decisions.
- SIP core has a resolver with NAPTR/SRV, Happy Eyeballs ordering, session timers, and a virtual clock for deterministic tests.
- Property based tests for transaction timers and basic parser tests.

Cons
- Media core is a placeholder (dummy RTP packet), NAT traversal is unimplemented, and services are mostly stubs.
- SIP parsing is string based and limited compared to full RFC handling or typed headers.
- Transport layer is minimal (no pooling, TCP and TLS are stubs).
- No RTP/RTCP, SRTP, or ICE features yet, so end to end calling is not viable without major work.

## mdsiprtp3

Pros
- Single crate is simple to navigate and build.
- Implements a custom SIP parser, header model, dialog and transaction logic, plus basic UAC/UAS flows.
- RTP/RTCP, jitter buffer, G.711 codecs, and DTMF detection are present.
- Includes voicemail and B2BUA scaffolding to show intended higher level use cases.

Cons
- Many TODOs and stubs: mixer loop, TCP transport, RTP extension handling, B2BUA media forwarding, and a placeholder UserAgent.
- UDP only, fixed RTP port assumptions, no TLS, no SRTP, no ICE.
- Docs claim higher completion than the code shows, which makes status hard to trust.
- No integration test harness; only module level unit tests.

## Practical fit

- mdsiprtp is the closest to a production grade SIP/RTP stack and the best base if you want full feature coverage now.
- mdsiprtp2 is a good foundation if you want to build a clean, service oriented stack but expect to implement most media and NAT functionality yourself.
- mdsiprtp3 is useful as a compact learning or prototyping base, but it is not ready for secure or interoperable deployments without significant work.

## Consolidation roadmap

### Recommendation: use mdsiprtp as the base
It has the broadest protocol coverage and the strongest test and integration story. The other two codebases add useful hygiene and a few app level features.

### Merge map (what to take from each)

| Capability | Source | Target approach |
| --- | --- | --- |
| Toolchain pinning, ADRs, SBOM/coverage scripts | mdsiprtp2 | Port to mdsiprtp workspace; align scripts and CI gates |
| Config crate with reloadable settings | mdsiprtp2 | Replace or wrap mdsiprtp config to match sipd style |
| Voicemail server flows and WAV export | mdsiprtp3 | Move into mdsiprtp-session or a new app crate |
| B2BUA and agent handle API | mdsiprtp3 | Merge into mdsiprtp-session or a new b2bua crate |
| UAC/UAS call flow examples | mdsiprtp3 | Port into examples or integration tests |

### Phased plan

Phase 0: baseline
- Pick mdsiprtp as the canonical workspace.
- Create a compatibility branch to trial ports without disrupting main.

Phase 1: hygiene and tooling
- Add rust-toolchain pinning, ADRs, and scripts from mdsiprtp2.
- Normalize linting, formatting, and SBOM/coverage outputs.
- Align config and logging defaults across crates.

Phase 2: feature consolidation
- Port voicemail server module into a new crate (e.g., mdsiprtp-voicemail).
- Port B2BUA and agent APIs into a dedicated crate (e.g., mdsiprtp-b2bua).
- Rewire these features to use existing transport, SDP, RTP, SRTP, and ICE from mdsiprtp.
- Remove duplicate codecs or parsers to avoid split behavior.

Phase 3: interop and validation
- Extend mdsiprtp integration tests to cover voicemail and B2BUA flows.
- Add security and resilience tests for SRTP + ICE paths and media relay.
- Define a compatibility test matrix (Asterisk, Baresip, simple softphone).

Phase 4: cleanup and docs
- Remove deprecated code paths and align errors and logging formats.
- Update README and architecture docs to reflect the new structure.
- Freeze the legacy repos or archive with a migration note.
