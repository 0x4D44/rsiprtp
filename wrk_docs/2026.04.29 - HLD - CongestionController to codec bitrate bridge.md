# HLD — CongestionController to codec bitrate bridge

## Why

REMB parsing (`crates/mdsiprtp-rtp/src/rtcp.rs:923`) and the AIMD `CongestionController` (`crates/mdsiprtp-rtp/src/session.rs:81`) already exist, but no code applies the produced `target_bitrate` to any encoder. This task closes the loop: pump `CongestionController::target_bitrate()` updates into `OpusCodec::set_bitrate(...)` so REMB-driven adaptation actually changes the encoder rate in flight.

G.711 and G.722 are fixed-rate; pushing a target rate at them must be a non-event — a compile-time non-event, not a no-op method on a misleading common trait.

Assumes the sibling HLD (ropus 0.12 swap with typed runtime API) has landed: `OpusCodec::set_bitrate(Bitrate)`, `bitrate()`, `effective_bitrate_bps()` are available.

## Goal

The smallest correct path from a `u64` target bps to a typed `Bitrate::Bits(u32)` on `OpusCodec`, with light hysteresis to keep REMB jitter from churning the encoder. Adaptive codecs adapt; fixed-rate codecs are not in the picture.

## Non-goals

- Generating outbound REMB.
- Transport-wide congestion control (RFC 8888 / TCC).
- NB / MB / WB / SWB / FB bandwidth reselection at low bitrates. `Encoder::set_max_bandwidth` exists; a follow-up task can add a parallel pump using the same call site.
- Refactoring `MediaSession`'s hardcoded `G711Codec` (`crates/mdsiprtp-session/src/call.rs:218`) so it can hold an Opus encoder. Orthogonal; tracked separately.
- Wiring the inbound RTCP receive path that runs `Remb::parse` → `CongestionController::on_remb`. No such path exists today; this HLD does not build it. The bridge is exercised by the integration test below by calling `cc.on_remb(...)` directly with REMB bytes parsed via `Remb::parse`.

## Design

This is option #1 from the briefing: an `AdaptiveBitrate` trait implemented only by codecs that actually adapt, plus a tiny standalone hysteresis filter that takes a `u64` and a `&mut dyn AdaptiveBitrate`.

Why option #1 over the others: option #2 (unified `Codec` trait with no-op `set_target_bitrate` on fixed-rate codecs) would mislead readers into thinking they can adapt G.711. Option #3 (subscription channel) introduces async-vs-sync coupling the codec layer does not have today. Option #1 says exactly what is true: adaptation is a property only some codecs declare.

### `AdaptiveBitrate` trait — `mdsiprtp-media`

```rust
// crates/mdsiprtp-media/src/codec/mod.rs
pub trait AdaptiveBitrate {
    /// Apply a target bitrate in bits per second.
    /// Implementations clamp or reject according to codec rules.
    fn set_target_bitrate_bps(&mut self, bps: u32) -> Result<(), String>;
}
```

Single method. Telemetry (current encoder rate, FEC state, etc.) is read off the concrete `OpusCodec` (via `effective_bitrate_bps()` and friends from the sibling HLD); the trait does not duplicate it.

Implemented for `OpusCodec` only:

```rust
// crates/mdsiprtp-media/src/codec/opus.rs
impl AdaptiveBitrate for OpusCodec {
    fn set_target_bitrate_bps(&mut self, bps: u32) -> Result<(), String> {
        self.set_bitrate(Bitrate::Bits(bps))
    }
}
```

`G711Codec` and `G722Codec` do not implement the trait. A caller that does not have an `&mut dyn AdaptiveBitrate` simply skips the bridge.

### `BitrateBridge` — `mdsiprtp-session`

A standalone hysteresis filter. It does **not** take a `CongestionController` directly — that would couple `mdsiprtp-media` ↔ `mdsiprtp-rtp` for no gain. The bridge takes a `u64` (today supplied by `cc.target_bitrate()`, tomorrow potentially by something else).

```rust
// crates/mdsiprtp-session/src/bitrate_bridge.rs
pub struct BitrateBridge {
    last_applied_bps: Option<u32>,
    last_applied_at: Option<Instant>,
    min_interval: Duration, // default 200 ms
    rel_threshold: f32,     // default 0.05 (5 %)
}

impl BitrateBridge {
    pub fn new() -> Self { /* defaults above */ }

    /// Apply `target_bps` to `codec` if the change clears both the relative
    /// threshold and the minimum interval. Returns `Ok(true)` if applied.
    /// `now` is injected so tests can drive time deterministically (Sans-IO
    /// style, matching the rest of the stack).
    pub fn poll(
        &mut self,
        target_bps: u64,
        codec: &mut dyn AdaptiveBitrate,
        now: Instant,
    ) -> Result<bool, String>;
}
```

Behaviour, in order:

1. Saturating-cast `target_bps` to `u32`. The CC's own `min_bitrate` / `max_bitrate` keeps it in a sane range; the codec rejects values outside its valid range.
2. If `last_applied_bps.is_none()`, apply unconditionally (first call).
3. Else, with `last = last_applied_bps`:
   - If `last == 0` (defensive — CC clamping prevents this in practice), apply.
   - Else apply iff `|new − last| / last >= rel_threshold` **and** `now − last_applied_at >= min_interval`. Otherwise return `Ok(false)`.
4. On apply, call `codec.set_target_bitrate_bps(new)?`, record `last_applied_bps = new`, `last_applied_at = now`, return `Ok(true)`.

The "and" semantics deliberately suppress small changes even after long quiet — but small changes accumulate against the *last-applied* baseline, so slow drift will eventually cross the 5 % gate and apply. The encoder tracks drift in 5 % steps; that is the intent.

That is the whole component. No async, no channels, no pacing, no clamping of its own — just a filter.

### First-call behaviour

`CongestionController::default()` starts at `target_bitrate = 500 kbps`; `OpusConfig::default()` starts at 32 kbps. On the very first `poll`, the bridge will apply the CC's value to the encoder, regardless of whether any REMB has arrived yet. That is correct — once the bridge is installed, CC owns the bitrate — but it means the configured `OpusConfig.bitrate` is effectively a transient. Callers who care should construct the CC with `initial_bitrate` matching the codec's starting rate. Document at the call site; do not put cleverness in the bridge.

### Codec error handling

`set_target_bitrate_bps` returns `Result<(), String>` because ropus rejects out-of-range values. The bridge surfaces that error. **Production callers should log-and-continue** rather than fail the call: a transient bad value (e.g., a buggy CC tune-up that overshoots) should not tear down the media session. The bridge does not swallow the error itself — that is a policy decision belonging to the caller.

### Where it lives

- **Trait**: `mdsiprtp-media` (next to the codecs it abstracts over).
- **Filter component**: `mdsiprtp-session` (where the eventual call site sits — the call lifecycle, the `CongestionController`, and the encoder all meet here). One file. No bridge logic in any other crate.

### Clamping

Two layers, each with one job:

- **CC** owns the policy ceiling/floor (`min_bitrate`, `max_bitrate`), set at construction.
- **Codec** owns the standard's ceiling/floor (Opus accepts ~6–510 kbps; ropus rejects out-of-range values).

The bridge does no clamping. A codec rejection propagates as `Err(String)` — that signals CC misconfiguration, not a transient.

### Update cadence

Hysteresis defaults: 5 % relative change, 200 ms minimum interval. Constructor-only knobs, not env-driven, not a public field. If realistic REMB traces show jitter or laggy adaptation, retune the constants before exposing them.

## Implementation outline

Five touch points; no edits to `G711Codec`, `G722Codec`, `CongestionController`, RTCP code, or the existing `MediaSession`.

1. `crates/mdsiprtp-media/src/codec/mod.rs` — declare `AdaptiveBitrate`.
2. `crates/mdsiprtp-media/src/codec/opus.rs` — `impl AdaptiveBitrate for OpusCodec`.
3. `crates/mdsiprtp-media/src/lib.rs` — re-export `AdaptiveBitrate` at crate root.
4. `crates/mdsiprtp-session/src/bitrate_bridge.rs` — new file: `BitrateBridge`.
5. `crates/mdsiprtp-session/src/lib.rs` — `pub mod bitrate_bridge;` and re-export `BitrateBridge`.

## Tests

### Unit — bridge (`crates/mdsiprtp-session/src/bitrate_bridge.rs`)

Uses a tiny in-test `MockAdaptive` that records the last-set value, plus injected `Instant`s. Real codec exercise is reserved for the integration test below.

- `applies_first_update` — fresh bridge; `poll(100_000, mock, t0)` → mock sees 100 000, returns `Ok(true)`.
- `suppresses_subthreshold_change` — apply 100 kbps at `t0`, then `poll(102_000, …, t0 + 1 s)` (2 % < 5 %) → mock unchanged, returns `Ok(false)`.
- `applies_supra_threshold_change` — apply 100 kbps at `t0`, `poll(110_000, …, t0 + 1 s)` → mock at 110 000.
- `respects_min_interval` — apply 100 kbps at `t0`, `poll(200_000, …, t0 + 50 ms)` → suppressed; `poll(200_000, …, t0 + 250 ms)` → applied.
- `monotonic_through_repeated_updates` — feed a sequence of values with advancing `Instant`s; verify the mock ends at the final post-hysteresis value.
- `surfaces_codec_error` — `MockAdaptive` configured to return `Err`; `poll` returns the same `Err` and does not update the bridge's state (next `poll` still treats it as first-call).
- `slow_drift_eventually_applies` — sequence of small (~2 %) increases over many polls past `min_interval` each time; verify cumulative drift > 5 % vs last-applied value triggers an apply.

### Integration — REMB bytes → bridge → Opus (`crates/mdsiprtp-session/tests/bitrate_bridge_remb.rs`, new)

The codec-focused integration test the briefing asks for. No Asterisk, no baresip.

1. Build `CongestionController` (defaults: initial 500 kbps, min 50 kbps, max 5 Mbps), `BitrateBridge`, and `OpusCodec` with default config (32 kbps).
2. Encode one frame; record `codec.effective_bitrate_bps()`.
3. Build a `Remb` bytes-on-the-wire payload with `bitrate = 64_000` (chosen above the CC's 50 kbps floor so the clamp does not fire). Round-trip the bytes through `Remb::parse(...)`. Feed the parsed value into `cc.on_remb(...)`. Assert `cc.target_bitrate() == 64_000`.
4. `bridge.poll(cc.target_bitrate(), &mut codec, Instant::now())?` — assert returns `Ok(true)` and `codec.effective_bitrate_bps()` is now ≈ 64 000.
5. Encode another frame and assert `effective_bitrate_bps()` has dropped relative to step 2.

This is the meaningful test: real REMB bytes → CC → bridge → real Opus → smaller frames out. ropus is bit-exact against xiph; we do not re-verify the codec, only the bridge.

### What we do not test

- `CongestionController` state transitions — already covered in `mdsiprtp-rtp::session` tests.
- Opus encode/decode correctness — covered by ropus's own RFC 6716 vectors.

## Risks / open items

- **Hysteresis defaults (5 % / 200 ms)** are educated guesses. Retune under real REMB traces before exposing them as runtime config.
- **Production path is dark.** The bridge is shippable now and exercised by the integration test, but in `MediaSession` the encoder is hardcoded to `G711Codec` (`crates/mdsiprtp-session/src/call.rs:218`) and there is no inbound RTCP receive path that calls `Remb::parse` → `CongestionController::on_remb`. Lighting the bridge up end-to-end requires both, tracked separately. This HLD deliberately stops at the bridge so each piece lands cleanly.
- **First-call surprise.** As noted above, the first `poll` overrides `OpusConfig.bitrate` with whatever the CC has (default 500 kbps). The eventual call-site code should construct CC with `initial_bitrate` matching the codec's starting rate; otherwise the encoder ramps to 500 kbps before any REMB is ever seen.
- **NB / MB / WB / SWB / FB reselection** at low bitrates is out of scope. When a follow-up wants it, the entry point is `Encoder::set_max_bandwidth`, called from the same site as `set_bitrate`. No bridge changes; a parallel pump.
