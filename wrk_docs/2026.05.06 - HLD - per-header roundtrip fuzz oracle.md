## 1. Context and scope

We have **two complementary oracles** at the message and SDP level today
(`sip_message_roundtrip`, `sdp_session_roundtrip`), each implemented as a
small `tests/<oracle>/mod.rs` module re-included from `fuzz_targets/*.rs`
via `#[path]`. Both perform a parse–serialize–reparse fixed-point check.

Per-header fuzz targets, by contrast, are **crash-only**: ~8 lines each
that invoke the typed parser and discard the result
(`fuzz/fuzz_targets/sip_via_typed.rs`, `sip_contact.rs`, `sip_name_addr.rs`,
`sip_cseq.rs`, `sip_uri.rs`, `sip_via.rs`, `sip_digest.rs`).

Per-header **proptest** fixed-point properties live at
`crates/rsiprtp/tests/proptest_sip_typed.rs`; they assert
`H::parse(s)? == H::parse(&H::parse(s)?.to_string())?` on
**structurally-valid** generated inputs, with shrinking. They cover
`Via`, `From`, `To`, `Contact`, `CSeq`, and `SipUri`.

The post-mortem follow-up is to add a libfuzzer-driven round-trip oracle
on **byte-mutated arbitrary input**, not generator-shaped input. This is
strictly additive to both the proptest oracle and the existing crash
targets:

| Layer | Input shape | Asserts | Strength |
|-------|-------------|---------|----------|
| Crash target | arbitrary bytes (utf8-gated) | `parse` does not panic | trivial |
| **This work** | arbitrary bytes (utf8-gated) | parse-then-serialize-then-reparse fixed point | catches serializer asymmetries |
| Proptest fixed-point | grammar-valid generated strings | same fixed point | shrinks failures; explores deep paths |
| Tier-1 message oracle | arbitrary bytes (whole message) | same fixed point at message granularity | corpus-mutator hunts |

The new layer answers a question neither of the others does cheaply:
*can the corpus mutator, starting from the existing per-header crash-target
corpus, find a byte-string that the typed-header parser accepts but
re-emits in a form that the same parser then parses to a different AST?*

## 2. Goals

1. Promote at least one per-header crash target to a libfuzzer round-trip
   oracle. Recommendation: start with **one** target (`sip_via_typed`),
   prove the pattern, then fan out.
2. Reuse the established `tests/<oracle>/mod.rs` + `fuzz_targets/<bin>.rs`
   `#[path]` pattern so the oracle code is shared between an integration
   test driver (sanity) and the fuzz binary (mutation).
3. Add the new target to a `fuzz_overnight.ps1` profile so the inventory
   test is satisfied and the rotation budget remains healthy.
4. Land with documentation precise enough that fanning out (`sip_contact`,
   `sip_name_addr`, `sip_cseq`, `sip_uri`) is a near-mechanical follow-up.

## 3. Non-goals

- **Replacing the crash targets.** They keep their corpora and stay
  scheduled. The round-trip target is **separate** with its own corpus
  (libfuzzer-managed), not a swap-in. Rationale in §11.
- **Asserting wire-string equality (`s == s2`).** Parsers are lenient
  (whitespace trimming, scheme-case lowercase in `SipUri`, method
  upper-casing in `CSeq`, bare-addr-spec → angle-bracketed in `NameAddr`).
  Wire equality would falsely fire. AST equality after the first
  normalization round is the contract — same as the Tier-1 oracle
  (`assert_roundtrip_fixed_point`).
- **Differential testing against `rsip 0.4`.** Out of scope; lives at
  `tests/parser_diff.rs` with its own oracle.
- **Per-header proptest expansion.** Already shipped in
  `tests/proptest_sip_typed.rs`. We are not duplicating its work —
  byte-mutation is a different signal than grammar-shaped generation.
- **Headers without a `parse(&str) -> Result<Self, _>` and a
  `Display` impl.** `sip_digest` parses `DigestChallenge`; round-tripping
  it requires a serializer, which the current `DigestChallenge` does not
  expose (see §5.5). Out of scope for v1.

## 4. Choosing a starting target

We have seven crash targets; criteria for promotion:

| Target | Surface area | Mutation susceptibility | Wire frequency | Proptest covers? | Has `Display`? |
|--------|---|---|---|---|---|
| `sip_via_typed` | high (5+ params, IPv6 brackets, tri-state rport) | medium (grammar-gated) | every transaction | yes | yes |
| `sip_contact`   | high (wildcard branch, NameAddr branch, q/expires) | medium | every REGISTER + INVITE | yes | yes |
| `sip_name_addr` | high (display-name quoting, escapes, params) | low (`<...>` framing) | every From/To | yes (via From/To) | yes |
| `sip_cseq`      | small (`u32 SP METHOD`) | high (small grammar = mutator finds the shape) | every request | yes | yes |
| `sip_uri`       | very high (scheme, userinfo, hostport, params, headers, percent-enc) | medium | inside every header | yes | yes |
| `sip_via`       | (legacy untyped `Via` — superseded by `sip_via_typed`) | n/a | n/a | n/a | n/a |
| `sip_digest`    | medium (challenge params, quoted strings) | medium | only on auth | partly (no proptest yet) | **no** |

Recommendation: **`sip_via_typed`** first.

Why Via:

- **Highest density of round-trip risk.** Five parameters
  (`branch`/`received`/`rport`/`maddr`/`ttl`), tri-state `rport`,
  IPv6 bracketing in `sent-by`, quoted-string parameter values
  (`parse_params` honors quoted-string escapes — `via.rs:261-273`).
  Each axis is a place a serializer asymmetry could hide.
- **Real-world traffic frequency.** Every SIP transaction emits a
  Via header.
- **Existing corpus to seed from.** `fuzz/corpus/sip_via_typed/` already
  has months of mutator output. Copying it as the seed for the new
  target gives the new target's mutator a head start.
- **Tractable.** ~8-line oracle with one clear assertion shape.

Why **not** start with `sip_uri`, despite higher surface area:

- `SipUri` has known leniencies (`;;` collapses, `?` headers without
  `=`, scheme case lowercase) that the proptest oracle has already
  exercised at the structural level. The byte-mutation coverage gap is
  smaller than for Via, where the parameter parser has a quoted-string
  branch that a generator can't easily reach.

Why **not** `sip_cseq` despite the smallest grammar:

- The state space (`u32 × method`) is so small that the proptest run
  has effectively saturated it (256 cases × `any::<u32>()` × 14 methods).
  Byte-mutation is unlikely to find anything proptest hasn't.

Recommended fan-out order if v1 lands clean:

1. v1: `sip_via_typed_roundtrip`
2. v2 (next sprint): `sip_contact_roundtrip` (wildcard / Addr split is
   load-bearing for round-trip — the v2 known-bug at
   `proptest_sip_typed.rs:386-394` is exactly this kind of signal)
3. v3: `sip_uri_roundtrip` (needs a corpus dictionary for percent-encoded
   sequences; non-trivial setup — see §10 risks)
4. **defer:** `sip_name_addr` (covered transitively by Contact + From/To
   round-trips at the message level; standalone target adds little)
5. **defer:** `sip_cseq` (proptest saturates)
6. **defer:** `sip_digest` (needs a serializer first — see §5.5)

## 5. Oracle shape

### 5.1 Skeleton

Mirrors `assert_roundtrip_fixed_point` in
`crates/rsiprtp/tests/parser_roundtrip_oracle/mod.rs:30-66`, scoped to
one header:

```rust
pub fn assert_via_roundtrip_fixed_point(bytes: &[u8]) {
    // Phase 1: utf8 gate. Non-utf8 bytes are not a typed-header
    // failure mode — the framing layer rejects them upstream.
    let Ok(s) = std::str::from_utf8(bytes) else { return };

    // Phase 2: first parse. Reject means "out of scope"; round-trip
    // is undefined for inputs we reject. Returns silently.
    let Ok(v1) = Via::parse(s) else { return };

    // Phase 3: serialize.
    let s2 = v1.to_string();

    // Phase 4: reparse — MUST succeed. A serializer that emits bytes
    // its own parser rejects is the bug we're hunting.
    let v2 = Via::parse(&s2).unwrap_or_else(|e| panic!(
        "second parse failed: serializer produced text our parser \
         cannot accept.\nv1: {v1:#?}\ns2: {s2:?}\nerror: {e:?}",
    ));

    // Phase 5: re-serialize and assert fixed point at v2.
    let s3 = v2.to_string();
    let v3 = Via::parse(&s3).unwrap_or_else(|e| panic!(
        "third parse failed: round-trip not idempotent at parse step.\n\
         v2: {v2:#?}\ns3: {s3:?}\nerror: {e:?}",
    ));
    assert_eq!(v2, v3,
        "round-trip not a fixed point at v2 (AST inequality).\n\
         s2: {s2:?}\ns3: {s3:?}");
    assert_eq!(s2, s3,
        "round-trip not a fixed point at v2 (string inequality).\n\
         s2: {s2:?}\ns3: {s3:?}");
}
```

This is exactly the Tier-1 oracle shape, tightened for the per-header
case: one less `to_bytes`/`from_utf8` cycle because `Display` works on
`&str` directly.

### 5.2 Early-reject rationale (per supervisor question 2)

The two early-return branches (utf8 fail, parse fail) are the same
"undefined input" branches the Tier-1 oracle uses. We deliberately
**do not** assert anything in them, because:

- Asserting `Err` on parse failure would couple the oracle to the
  parser's reject set — which is exactly the surface the crash target
  already exercises and which we don't want to over-specify here.
- Asserting on the utf8 boundary is a framing concern; the message-level
  oracle already covers framing.

In other words: an oracle that accepts everything in early-reject is
**not** "a crash target in disguise" — it is a crash target *plus* a
property assertion on inputs the parser accepts. The crash-only signal
(no panic on rejected input) is preserved for free, because the function
still runs the parser on every input. We just defer the post-parse
assertion to the accepted-input branch.

The honest framing: this oracle subsumes the crash target on accepted
inputs and matches the crash target on rejected inputs. Keeping the
crash target alongside it is defensive — see §11.

### 5.3 What the oracle catches

The fixed-point assertion catches three classes of bug:

1. **Serializer emits unparseable text** (Phase 4 panic). E.g. a future
   change to `Via::Display` that forgets to bracket an IPv6 sent-by.
2. **Parse–serialize chain is non-idempotent** (Phase 5 string inequality
   on m2 → m3). E.g. a re-quoting bug that flips `"x"` ↔ `x` each round.
3. **Parse–serialize chain converges in bytes but not in the AST**
   (Phase 5 AST inequality). E.g. a parameter-merging bug where
   `;rport;rport=5060` collapses to a different shape than reparsing
   the collapsed form yields.

It does **not** catch:

- Bugs that depend on the *original* input's wire form. The first
  normalization is allowed to be lossy (lenient parser); the contract
  is only that the *normalized* form is a fixed point. (Same caveat as
  Tier-1.)
- Bugs in the rejection set — anything we panic on at parse stage is the
  crash target's job.

### 5.4 Why wire-string equality is not the contract

`Via::parse` accepts:

- Whitespace-trimmed input (`s.trim()` at `via.rs:63`).
- Embedded whitespace inside `;` chunks (`parse_params` trims each
  chunk at `via.rs:282`).
- Empty parameter chunks (`;;` between params produces no entry).

`Via::Display` (`via.rs:187-198`) emits a single canonical form. So a
mutator that produces `"SIP/2.0/UDP h ; branch=z9hG4bK1 ;;rport"` parses
fine, but `s2 = "SIP/2.0/UDP h;branch=z9hG4bK1;rport"`. Asserting
`s == s2` would fire on every such input. Asserting `v2 == v3` is the
right invariant: the *second* round must be stable.

### 5.5 `DigestChallenge` exclusion (per non-goals)

`DigestChallenge::parse` exists but `Display` does not (verified via
`Grep "impl fmt::Display for DigestChallenge"` returns no hits in
`crates/rsiprtp/src/sip/`). Round-trip oracle requires both. Promoting
`sip_digest` requires implementing `Display` first, which is a separate
unit of work. Listed in §4 fan-out as deferred for that reason.

## 6. Module placement and `#[path]` sharing

The Tier-1 pattern (`tests/parser_roundtrip_oracle/mod.rs` re-included
from both `tests/parser_roundtrip.rs` and
`fuzz/fuzz_targets/sip_message_roundtrip.rs` via `#[path]`) is the
right model. We adopt it verbatim.

### 6.1 Layout decision

**Recommended: per-header subdirectories under `tests/`, each with a
`mod.rs`.**

```
crates/rsiprtp/tests/
├── parser_roundtrip_oracle/
│   └── mod.rs                           ← existing, untouched
├── via_roundtrip_oracle/                ← NEW (v1)
│   └── mod.rs                           ← assert_via_roundtrip_fixed_point
├── via_roundtrip.rs                     ← NEW (v1) — test driver
├── (v2 fanout)
├── contact_roundtrip_oracle/mod.rs
├── contact_roundtrip.rs
├── ...
```

Each `<header>_roundtrip_oracle/mod.rs` exports one
`assert_<header>_roundtrip_fixed_point(bytes: &[u8])` function.

Each `<header>_roundtrip.rs` is the integration-test driver that
exercises the oracle on a small set of static fixtures (smoke; not the
mutator's job).

The fuzz target re-includes the oracle module via `#[path]`:
```rust
#![no_main]
use libfuzzer_sys::fuzz_target;

#[path = "../../crates/rsiprtp/tests/via_roundtrip_oracle/mod.rs"]
mod oracle;

fuzz_target!(|data: &[u8]| {
    oracle::assert_via_roundtrip_fixed_point(data);
});
```

### 6.2 Considered alternatives

**Alternative A: a single shared module**
`tests/per_header_roundtrip_oracle/mod.rs` exporting one assertion
function per header. Rejected: cargo-fuzz `#[path]` brings the *whole*
module into the binary, including unused branches. Each fuzz target
would compile assertions for all headers. Dead-code elimination would
likely strip them, but the source coupling makes the fan-out diff
larger and adds inter-header noise to per-header crash triage. The
Tier-1 oracle is a single function precisely *because* it is one
oracle; the per-header oracles are five different functions.

**Alternative B: put the oracle inside `parser_roundtrip_oracle/mod.rs`**
Rejected: that module's purpose is documented as the **whole-message**
oracle. Co-locating per-header oracles muddies its responsibility.

**Alternative C: in-source helpers (`#[cfg(test)]` in
`crates/rsiprtp/src/sip/parser/typed/via.rs`).**
Rejected: cargo-fuzz binaries cannot reach `cfg(test)` items. The
Tier-1 oracle deliberately lives in `tests/` for this reason.

**Alternative D: per-header `tests/<header>_roundtrip.rs` (no
subdirectory) — tests file *is* the oracle.**
Rejected: cargo discovers every `tests/*.rs` as an integration-test
binary, but `#[path = "../../crates/rsiprtp/tests/via_roundtrip.rs"]`
from the fuzz target would re-include the test's `#[test]` functions
in the fuzz binary. Existing pattern uses a subdirectory specifically
to opt out of cargo's auto-discovery (per `parser_roundtrip_oracle/mod.rs:11-16`
explicit comment). We follow the precedent.

## 7. `#[path]` reuse vs feature-flag — verdict

**Use `#[path]`. Identical to Tier-1.**

The fuzz crate is standalone (`fuzz/Cargo.toml:11-12` declares
`[workspace]`, opting out of the workspace) so a feature flag would
have to live on the `rsiprtp` crate, which is a higher-noise change
than `#[path]` on a single test module. `#[path]` is exactly what the
Tier-1 oracle does (`fuzz_targets/sip_message_roundtrip.rs:4`) and it
works cleanly. No reason to invent a different mechanism.

**Sharing path verified:** the test driver
`tests/via_roundtrip.rs` does `#[path = "via_roundtrip_oracle/mod.rs"]`
(test runs from `crates/rsiprtp/`), the fuzz target does
`#[path = "../../crates/rsiprtp/tests/via_roundtrip_oracle/mod.rs"]`
(fuzz crate at `fuzz/`, two levels up to repo root). Both pull the
same source. No duplication.

## 8. Wrapper integration

### 8.1 Profile placement

The new target joins the **`sip` profile** in `fuzz_overnight.ps1` (lines
40-50). After v1 lands, the profile becomes:

```powershell
"sip" = @(
  "sip_message_roundtrip",
  "sip_via_typed",
  "sip_via_typed_roundtrip",   # ← NEW
  "sip_contact",
  "sip_name_addr",
  "sip_cseq",
  "sdp_session_roundtrip",
  "sip_uri",
  "sip_digest"
)
```

This keeps the new target alongside its sibling crash target so they
share a corpus directory parent (different basenames, so libfuzzer
keeps them isolated, but they live near each other on disk for triage
convenience).

**Note on superseding:** I deliberately do **not** retire `sip_via_typed`
into `sip-legacy` when the round-trip target lands. The crash target is
strictly cheaper (no serialization + reparse) and keeps detecting
panics on rejected inputs that the round-trip oracle skips. They are
complementary; see §11.

### 8.2 Rotation-cost arithmetic

Current `sip` profile has 8 targets, 30-min slices (`SliceSeconds=1800`),
8h budget = 28800s. Cycle time = 8 × 1800 = 14400s = 4h, so each target
gets ~2 visits per overnight run.

Adding one target → 9 targets. Cycle time 9 × 1800 = 16200s = 4h30m.
Each target ~1.78 visits (rounding down to 1 full visit + half of a
second). The change is **not material**: we lose 22 % of one target's
second visit, distributed evenly across all targets. No reason to
adjust `SliceSeconds`.

If fan-out adds three more (Contact, URI, NameAddr roundtrip) →
12 targets → 6h cycle → 1.33 visits/target. At that point we'd want to
either bump the budget or split the profile (`sip-roundtrip` separate
from `sip-crash`). Out of scope for v1; flagged in §13.

### 8.3 Smoke-run command

Standalone smoke before adding to the profile (developer ergonomics):


In CI / overnight rotation, the wrapper drives it; no extra command.

## 10. Risk register

### R1 — Corpus seeding strategy

**Risk.** A fresh round-trip target with an empty corpus starts from
scratch — its first hour of fuzzing reproduces what the existing
`sip_via_typed` crash target found in its first hour months ago. The
mutator wastes time re-discovering known-good shapes.

**Mitigation.** Seed `fuzz/corpus/sip_via_typed_roundtrip/` from
`fuzz/corpus/sip_via_typed/` at S2 time:

```powershell
Copy-Item -Recurse fuzz/corpus/sip_via_typed/* fuzz/corpus/sip_via_typed_roundtrip/
```

These corpora *should not be merged in either direction afterward* —
the round-trip oracle's mutator will evolve toward inputs that survive
parse, serialize, and reparse, and its corpus accumulates that signal.
Cross-pollinating back to the crash target would dilute the crash
target's "interesting reject" inputs.

**Additional consideration.** The Tier-1 message-roundtrip corpus
(`fuzz/corpus/sip_message_roundtrip/`) almost certainly contains plenty
of full SIP messages with Via headers; we do **not** want to seed from
that, because per-header inputs are header *values*, not whole messages
with `Via:` prefix. Seeding from a whole-message corpus would mean every
seed is a parse failure, defeating the seeding effort.

### R2 — Parameter-order non-determinism

**Risk** (called out in supervisor prompt). Typed-header `to_string()`
having parameter-order non-determinism (HashMap iteration) would make
the round-trip non-idempotent.

**Mitigation: not applicable here.** Verified by reading the source:

- `Via::params` is `Vec<(String, Option<String>)>` (`via.rs:53`),
  iterated in storage order at `via.rs:190`.
- `NameAddr::parameters` is `Vec<...>` (`name_addr.rs:53`), iterated in
  storage order at `name_addr.rs:164`.
- `ContactAddr::params` is `Vec<...>` (`contact.rs:96`), forwarded
  through `NameAddr::Display` at `contact.rs:186-191`.
- `Contact::Wildcard.params` is `Vec<...>` (`contact.rs:76`), iterated
  in storage order at `contact.rs:175`.
- `SipUri::params` is `Vec<...>` (`uri.rs:86`), iterated in storage
  order at `uri.rs:389`.
- `SipUri::headers` is `Vec<...>` (`uri.rs:91`), iterated in storage
  order at `uri.rs:399`.

All five typed-header param fields are `Vec`-backed. None use HashMap.
Iteration order is deterministic. The risk does not apply to this
workstream. (The proptest HLD also flagged this and confirmed it; see
`2026.05.06 - HLD - proptest SIP typed headers.md` §"Open questions"
question 1.)

### R3 — `SipUri` percent-encoding round-trip

**Risk** (called out in supervisor prompt). `SipUri::parse` does not
percent-decode (verified at `uri.rs:131` doc comment: "We do not
attempt percent-decoding here — values stay as-on-the-wire"). So
`%41` stays as the three bytes `%41`, not the byte `A`. Display
re-emits them verbatim. Fixed-point should hold.

**However**, this risk is not v1's problem because v1 targets Via, not
URI. When fan-out reaches `sip_uri_roundtrip`, the risk re-emerges in a
specific form: the parser treats `?` as the URI-headers separator and
splits `&key=value` pairs, but does not validate percent-encoding. A
mutator that produces `sip:h?k=%` (incomplete percent-escape) parses
cleanly, Display re-emits `sip:h?k=%` cleanly, and the round-trip holds.
That's the right behavior; flagging it in advance so v3 implementer
doesn't get confused. Add a fixture for it in `tests/uri_roundtrip.rs`
when v3 lands.

### R4 — Quoted-string parameter values in Via

**Risk.** `Via::parse_params` (`via.rs:261-273`) honors a
quoted-string state-machine for parameter values. RFC 3261 §25.1 grammar
allows `;branch="z9hG4bK foo"` in principle; our parser stores the
quoted form verbatim (including the quotes) as the value string.

**Verification needed** (test fixture in S1): does
`v.params == [("branch", Some("\"z9hG4bK foo\""))]` round-trip?
The `Via::Display` path at `via.rs:192` emits `;branch="z9hG4bK foo"`
verbatim. Reparse should yield the same vec entry. The quoted-string
state machine inside `parse_params` is the load-bearing path here —
exactly the kind of place a mutator could produce
`;branch="z9\";rport"` and trigger an asymmetry. The round-trip oracle
catches this where the crash target cannot (it doesn't compare
re-parses).

### R5 — IPv6 bracket interactions

**Risk.** `split_sent_by_params` (`via.rs:207-240`) walks bytes
honoring `[`/`]` IPv6 framing. A mutator producing
`SIP/2.0/UDP [::1];branch=z9hG4bK1` parses cleanly; producing
`SIP/2.0/UDP [::1]:5060;branch=z9hG4bK1` likewise. Mutator producing
`SIP/2.0/UDP [];branch=z` is interesting: `SipUri` is not invoked on
the sent-by token at the Via level (Via stores `sent_by` as a raw
string per `via.rs:48-50`), so an empty IPv6 bracket pair just becomes
the literal sent-by `[]`, which Display re-emits verbatim. Round-trip
holds. No bug, but this is the kind of case the oracle should sanity-fixture
in S1 so future regressions are caught.

### R6 — Slow oracle ⇒ low ips

**Risk.** Round-trip is parse + serialize + parse + serialize + parse
(or 2× parse + 1× serialize, depending on accounting). Worst case ~5×
the work of the crash target per execution. Libfuzzer ips drops 5×;
fewer mutations per second.

**Mitigation: accept it.** The Tier-1 message-roundtrip target also
runs at lower ips than the crash target and finds bugs the crash target
cannot. Same trade. We are paying mutations-per-second for assertion
strength. If overnight ips dip below the rotation's noise floor we can
raise the slice budget for that one target — but unlikely to be
needed at one new target / nine total.

### R7 — Generator-shaped vs byte-mutated coverage drift

**Risk.** The proptest fixed-point at `proptest_sip_typed.rs` and the
new fuzz target both assert *exactly the same property*. If proptest
is finding zero failures, what extra signal does fuzzing add?

**Answer: different inputs.** Proptest covers "structurally valid by
generator construction." Fuzzing covers "byte-mutated from a corpus
seed." There are inputs in the parser's accept set that the generator
will never produce (deeply nested escapes, off-grammar but accepted
forms like `;;`, mid-token whitespace, NUL bytes inside quoted strings,
etc.), and there are inputs the generator easily produces that fuzzing
takes ages to find.

The two are orthogonal. Both are needed.

### R8 — Inventory test caching

**Risk.** `tests/fuzz_inventory.rs` reads `fuzz_overnight.ps1` and the
fuzz_targets directories at test time. If the new target is added to
the profile but the file isn't created (or vice versa), the inventory
test fails. This is *good* — the test catches drift — but the
implementation order matters.

**Mitigation.** Stage order in §12 has the file land (S2) before the
profile (S3). The inventory test is asymmetric: a target file without a
profile entry fails; a profile entry without a target file fails at
runtime in the wrapper, not at test time. The test is one-way.

### R9 — Crash artifacts on the new target need triage process

**Risk.** First overnight run on the new target hits a crash. Existing
triage process (`fuzz_overnight.ps1` lines 240-267) writes to
`wrk_journals/fuzz_triage/crash_NNN_<target>/` and pauses for
`RESUME` file. The supervisor needs to know what to do with a panic
from the round-trip oracle vs from the crash target.

**Mitigation.** Document at the top of
`tests/via_roundtrip_oracle/mod.rs` the same way Tier-1 does: a panic
from this oracle is a *parser/serializer asymmetry* finding, not a
crash, and the input that triggered it is the minimum reproducer the
mutator-shrinker has produced.

## 11. Why keep the crash target alongside

Two reasons to **not** retire `sip_via_typed` when
`sip_via_typed_roundtrip` lands:

1. **Different rejection behavior.** The crash target only checks "does
   `parse` panic." The round-trip oracle returns silently on parse
   failure. So if a future change makes `parse` panic on a specific
   input the round-trip oracle would have early-exit-skipped, only the
   crash target catches it.
2. **Cheaper per-mutation.** The crash target gets through ~5× more
   mutations per second. On byte mutations of the corpus, it explores
   the reject surface much faster. Removing it would slow down crash
   discovery by the same factor.

This dual-target arrangement matches the Tier-1 pattern: we kept
`sip_message` (legacy crash target on rsip 0.4) in the `sip-legacy`
profile alongside `sip_message_roundtrip` in the `sip` profile.

## 13. Open questions for supervisor sign-off

### Q1 — One header or fan-out in v1?

**Recommendation:** start with **`sip_via_typed`** alone. Land the
pattern, prove the smoke + inventory + integration-test scaffold works,
then fan out in a follow-up sprint. Fanning out four headers in v1
makes the diff hard to review.

**Counter-argument:** the per-header fan-out is mechanical once the
pattern lands, and doing them together amortizes the wrapper-update
overhead. Three commits vs. one.

**Need supervisor verdict.**

### Q2 — Corpus seeding from the crash target?

**Recommendation:** yes, copy `fuzz/corpus/sip_via_typed/*` into
`fuzz/corpus/sip_via_typed_roundtrip/` before the smoke run. This
saves the new target hours of cold-start time. Caveat: do not merge
back; the corpora evolve toward different signals.

**Counter-argument:** the crash-target corpus is heavy on
parse-rejection inputs, which the round-trip oracle skips. Half the
seed value may be wasted. Cleaner to start from empty and let the
mutator find its own corpus shape.

**Need supervisor verdict.**

### Q3 — Retire the crash target on supersession?

**Recommendation:** keep both per §11. Two reasons (different
rejection behavior + cheaper per-mutation).

**Counter-argument:** the round-trip oracle subsumes the crash target
on accepted inputs and matches it on rejected inputs (no panic ⇒ both
pass; panic ⇒ both fire). One target is simpler.

**Recommend keeping both** but want explicit supervisor sign-off
before the implementation agent starts.

### Q4 — `proptest-regressions/` for the per-header proptest is
gitignored today. Should the fuzz target's corpus go in git?

**Not v1's question.** Libfuzzer-managed corpora live under
`fuzz/corpus/<target>/` and the existing `.gitignore` (verified)
excludes `fuzz/target/` but not `fuzz/corpus/`. Existing crash-target
corpora are committed selectively — the supervisor's decision call,
not this HLD's. Flag for awareness.

### Q5 — Should the `tests/via_roundtrip.rs` driver also exercise the
RFC 4475 fixture corpus's Via headers (extracted)?

**Recommendation: no in v1.** RFC 4475 fixtures are whole-message;
extracting individual Via lines is a copy-paste-prone task. The Tier-1
oracle already runs RFC 4475 messages end-to-end at
`tests/parser_roundtrip.rs:94-156`. Per-header drivers stick to a
small, hand-curated fixture set.

**Counter:** if a Via header from RFC 4475 §3.1.1.1 (`wsinv`) trips
the per-header oracle but not Tier-1, that's a real signal and the
drivers should overlap. Adds ~30 LOC of fixture extraction for v1.

**Need supervisor verdict.**

### Q6 — Per-header oracle module name vs Tier-1 module name?

`parser_roundtrip_oracle/mod.rs` uses the term "parser" for the
whole-message oracle. The proposed `via_roundtrip_oracle/mod.rs` drops
the "parser" prefix because the level is unambiguous from the header
name. The proptest module is `proptest_sip_typed.rs`. Naming is
inconsistent across the three.

**Recommendation: leave it.** Each module's name is locally
unambiguous. Renaming the existing module for consistency is a
bigger change than warranted.
