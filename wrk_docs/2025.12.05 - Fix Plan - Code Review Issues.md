## Issues to Fix

### Issue #1: Time-Based PRNG (Medium Severity)

**Problem:** Multiple files use time-based pseudo-random number generation which is predictable.

**Files Affected:**
- `crates/mdsiprtp-rtp/src/session.rs:217-233` - `rand_u16()` and `rand_u32()`
- `crates/mdsiprtp-sdp/src/builder.rs:359-366` - `rand_u64()`
- `crates/mdsiprtp-session/src/call.rs:497-503` - `rand_u32()`

**Solution:**
1. Add `rand` crate to workspace dependencies
2. Create centralized random utilities in `mdsiprtp-core/src/util.rs`
3. Replace all time-based PRNG calls with proper random functions
4. Update all affected crates to use the centralized utilities

**Implementation:**
```rust
// mdsiprtp-core/src/util.rs
use rand::Rng;

pub fn random_u16() -> u16 {
    rand::thread_rng().gen()
}

pub fn random_u32() -> u32 {
    rand::thread_rng().gen()
}

pub fn random_u64() -> u64 {
    rand::thread_rng().gen()
}
```

### Issue #3: Potential Integer Overflow in Jitter Calculation (Low Severity)

**Problem:** Cast from `i32` to `u32` could produce unexpected results.

**File:** `crates/mdsiprtp-media/src/jitter.rs:182-183`

**Current Code:**
```rust
self.receiver_state.jitter =
    self.receiver_state.jitter + ((d as i32 - self.receiver_state.jitter as i32) >> 4) as u32;
```

**Solution:** Use saturating arithmetic:
```rust
let adjustment = (d as i32 - self.receiver_state.jitter as i32) >> 4;
if adjustment >= 0 {
    self.receiver_state.jitter = self.receiver_state.jitter.saturating_add(adjustment as u32);
} else {
    self.receiver_state.jitter = self.receiver_state.jitter.saturating_sub((-adjustment) as u32);
}
```
