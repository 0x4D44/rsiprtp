## 2. Architecture Review

### 2.1 Strengths

**Layered Design (Excellent)**
The codebase follows a clean layered architecture that mirrors the SIP protocol stack:
```
Session Layer  → Call management, media sessions
Dialog Layer   → INVITE dialog lifecycle
Transaction    → RFC 3261 client/server transactions
SIP/SDP/RTP    → Protocol parsing and building
Transport      → UDP network I/O
```

**Sans-IO Pattern (Excellent)**
The transaction and dialog state machines are implemented using the Sans-IO pattern, making them:
- Highly testable without network mocking
- Deterministic and reproducible
- Easy to drive from any async runtime

**Typestate Considerations**
Transaction states are modeled with explicit state enums (`ClientState`, `ServerState`) enabling compile-time validation of valid transitions.

**Proper Separation of Concerns**
Each crate has a single responsibility:
- `mdsiprtp-core`: Only errors and config
- `mdsiprtp-sip`: Only SIP message handling
- `mdsiprtp-rtp`: Only RTP packets
- etc.

### 2.2 Areas for Improvement

**Dialog Layer Duplication**
The `session::call::Dialog` struct duplicates functionality from `dialog::DialogId`. While the session layer needs a lightweight representation, consider using composition:

```rust
// Current (crates/mdsiprtp-session/src/call.rs:18-28)
pub struct Dialog {
    id: DialogId,  // From dialog crate
    local_uri: String,
    remote_uri: String,
    local_cseq: u32,
}
```
This is acceptable for Phase 1, but could be refactored to avoid confusion between the two `Dialog` types.

### 3.2 Issues Found

#### Issue #1: Time-Based PRNG (Medium Severity)
**Location:** Multiple files
**Files:**
- `crates/mdsiprtp-rtp/src/session.rs:217-233`
- `crates/mdsiprtp-sdp/src/builder.rs:359-366`
- `crates/mdsiprtp-session/src/call.rs:497-503`

```rust
fn rand_u32() -> u32 {
    use std::time::{SystemTime, UNIX_EPOCH};
    let duration = SystemTime::now()
        .duration_since(UNIX_EPOCH)
        .unwrap_or_default();
    (duration.as_nanos() & 0xFFFFFFFF) as u32
}
```

**Problem:** This is not cryptographically secure random. For SSRC, session IDs, and other SIP identifiers, RFC 3261 recommends cryptographic randomness to prevent:
- Session hijacking via predictable Call-IDs
- SSRC collision in multiparty scenarios

**Recommendation:** Use `rand` crate with `rand::random::<u32>()` or `uuid::Uuid::new_v4()` (already a dependency).

---

#### Issue #2: Duplicate Code (Low Severity)
**Location:** Random number generation functions are duplicated across 3 files

**Recommendation:** Move to `mdsiprtp-core` as utility functions:

```rust
pub fn random_u16() -> u16 { ... }

```

#### Issue #3: Potential Integer Overflow in Jitter Calculation (Low Severity)
**Location:** `crates/mdsiprtp-media/src/jitter.rs:182-183`

```rust
self.receiver_state.jitter =
    self.receiver_state.jitter + ((d as i32 - self.receiver_state.jitter as i32) >> 4) as u32;
```

**Problem:** If jitter is very high, the cast from `i32` to `u32` could produce unexpected results on negative values.

**Recommendation:** Use saturating arithmetic:
```rust
let adjustment = (d as i32 - self.receiver_state.jitter as i32) >> 4;
self.receiver_state.jitter = self.receiver_state.jitter
    .saturating_add_signed(adjustment);
```

---

#### Issue #4: Missing Input Validation (Low Severity)
**Location:** `crates/mdsiprtp-sip/src/message.rs`

The SIP request builder doesn't validate URI formats before building:
```rust
pub fn uri(mut self, uri: &str) -> Self {
    self.uri = Some(uri.to_string());  // No validation

}
```

**Recommendation:** Add basic SIP URI validation or at least document that invalid URIs will fail at runtime.

## 7. Performance Considerations

### 7.1 Memory Usage

- **Efficient:** Uses `bytes::Bytes` for zero-copy packet handling
- **Efficient:** `BTreeMap` for ordered jitter buffer
- **Note:** Each call allocates ~50KB for buffers

### 7.2 CPU Usage

- **G.711 codec:** O(n) encoding/decoding, very fast
- **SDP parsing:** Single-pass, O(n)
- **Jitter buffer:** O(log n) operations (BTreeMap)

### 7.3 Potential Bottlenecks

1. **Jitter buffer PLC:** Linear fade creates new Vec each time
2. **SIP parsing:** String allocations could be reduced
3. **HashMap lookups:** Consider `FxHashMap` for hot paths
