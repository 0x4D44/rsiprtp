## Supervisor protocol

Per `feedback_supervised_subagent_waits.md` (anti-pattern memory): the supervisor
drives the long wait directly; sub-agents only handle bounded code work.

**Wait driver:** `Monitor` running `pwsh -File fuzz_overnight.ps1` with stdout
filtered to event lines (`START|HEARTBEAT|CLEAN|CRASH|RESUME|DONE`). Each line
becomes a chat notification.

**On HEARTBEAT (every 5 min):** record cumulative stats periodically — every
~30 min — to keep the journal cadence sane. Don't journal every heartbeat.

**On CLEAN:** one-line note in journal that target completed its slice.

**On CRASH:** rotation pauses (whole rotation, not just the crashed target — see
the comment in `fuzz_overnight.ps1:184-186`). Then:
1. Read the triage `fuzz.log` directly (small file).
2. Dispatch **Explore** agent: scope = read the crash log + locate the failing
   parser/serializer site, report shape and reproducer bytes. Bounded.
3. Dispatch **Plan** (or general-purpose) agent: scope = propose a minimal fix +
   regression test, given the failure shape. Bounded.
4. Dispatch **general-purpose** agent: scope = implement the fix + add the unit
   test. Bounded.
5. Read the diff inline, decide if `code-reviewer` skill is warranted (yes for
   non-trivial fixes; no for one-line CTL/UTF-8-class tightening).
6. Markup this journal with the crash subsection.
7. Touch `wrk_journals/fuzz_triage/<slot>/RESUME` to release the wrapper.
