## Test 1 — `registration::tests::test_needs_refresh` (Windows panic)

### Symptom

CI on Windows panicked inside `std::time::Instant::sub` (rustc
`library/std/src/time.rs:445:33`).

### Hypothesis

The Windows monotonic `Instant` is anchored to system boot. Subtracting more
than the current uptime from any `Instant` panics with `overflow when
subtracting duration from instant`. The test/production code computes
`expires_at - Duration::from_secs(self.config.expires / 5)` where the divisor
is 720s (3600 / 5). The test sets `expires_at = Instant::now() + 10s`, so
`refresh_at = now + 10s - 720s = now - 710s`. If Windows uptime is < 710s
when the test runs (fresh boot, fresh CI runner image), the subtraction
panics.

### Evidence

- `crates/rsiprtp/src/session/registration.rs:142-150` shows the bare
  `expires_at - Duration::from_secs(...)` arithmetic.
- `crates/rsiprtp/src/session/registration.rs:1053-1063` is the failing test;
  `expires_at` is computed as `now + 10s` so the subtraction is guaranteed to
  reach into negative territory relative to the platform epoch.
- The CI panic stack pointed at `Instant::sub`, not `checked_sub`, confirming
  the unchecked arithmetic was the panic site.
- Linux/macOS happen to use a different `Instant` representation that allows
  this subtraction; Windows alone enforces the post-boot lower bound, which
  matches the Windows-only failure mode.

### Fix

Replace the unchecked subtraction in `needs_refresh` with `checked_sub`:

```rust
let now = Instant::now();
let window = Duration::from_secs((self.config.expires as u64) / 5);
match expires_at.checked_sub(window) {
    Some(refresh_at) => now >= refresh_at,
    None => true,
}
```

If the subtraction would underflow, the refresh threshold is logically before
the platform epoch, which is necessarily in the past — so the manager *does*
need to refresh. Returning `true` preserves the documented invariant.

### Why it works

- Removes the only panic site in the function.
- Both unit tests (`test_needs_refresh` and `test_needs_refresh_far_future`)
  still exercise the intended invariant. The "far future" case (`+ 7200s`)
  produces `refresh_at = now + 6480s`, which `checked_sub` returns as `Some`
  on every platform; the comparison `now >= now + 6480s` is `false`, so the
  test still asserts the no-refresh path. The "needs refresh" case takes the
  `None` arm on a freshly booted Windows runner and the `Some` arm
  elsewhere — both branches return `true`, so the assertion holds
  unconditionally.
- This is a real production fix: the bug was latent on Windows whenever a
  process registers within ~12 minutes of boot, which is plausible for kiosk
  or appliance deployments.
- No `#[cfg]` gating required; behaviour is the same on all platforms.

## Test 2 — `tests::rfc3261::builder_validation::test_generate_tag_unique` (macOS flake)

### Symptom

Test generates 100 tags via `generate_tag()` and asserts at least 91 are
unique. Fails probabilistically on macOS only.

### Hypothesis

Birthday-paradox math rules out the obvious explanation: with 100 tags from
a 64-bit space, P(collision) ≈ 100²/(2·2⁶⁴) ≈ 2.7e-16 — utterly negligible.
The brief suggested possibly checking for a 32-bit entropy source, but the
actual root cause is even simpler: `rand_u64` derived its entropy *only*
from `SystemTime::now()`. The implementation was

```rust
let duration = SystemTime::now().duration_since(UNIX_EPOCH).unwrap_or_default();
duration.as_nanos() as u64 ^ (duration.as_secs() << 32)
```

When two consecutive calls observe the same `SystemTime::now()` value (i.e.
the wall clock didn't tick between calls), they produce identical tags.
macOS `SystemTime` resolution is coarser than Linux's nanosecond-grained
`clock_gettime(CLOCK_MONOTONIC)`, and 100 calls in a tight loop can easily
fit inside a single tick.

### Evidence

- `crates/rsiprtp/src/sip/message.rs:1058-1065` showed the time-only
  entropy. No randomness source was actually consulted.
- The XOR with `(secs << 32)` doesn't help: when `now()` returns identical
  values on adjacent calls, both operands collide together, so the XOR
  produces an identical result.
- `generate_branch` and `generate_call_id` both use `uuid::Uuid::new_v4()`
  which *does* draw from OS randomness — those functions never flake.
- `rand = "0.8"` is already a workspace dependency
  (`crates/rsiprtp/Cargo.toml:45`) and is used elsewhere in the crate
  (e.g. `ice/agent.rs`, `ice/stun.rs`, `srtp/sdes.rs`), so adding a call
  introduces no new dependency.
- The companion test at `crates/rsiprtp/src/sip/message.rs:2084-2089`
  worked around the bug with `thread::sleep(Duration::from_millis(1))`,
  which is itself diagnostic evidence the original author was aware of
  the timing-collision issue.

### Fix

Replace `rand_u64()` with `rand::thread_rng().next_u64()`:

```rust
pub fn generate_tag() -> String {
    use rand::RngCore;
    format!("{:x}", rand::thread_rng().next_u64())
}
```

The `rand_u64()` helper is deleted entirely (it had only one caller).

### Why it works

- `rand::thread_rng()` is a CSPRNG seeded from OS entropy on first use, then
  reseeded periodically. It does not depend on the wall clock and produces
  uncorrelated 64-bit outputs at any call rate.
- Birthday-paradox math: with proper 64-bit randomness, P(collision over
  100 tags) ≈ 2.7e-16. P(collision over the entire CI fleet over years)
  remains negligible.
- Behaviour is identical on Linux, Windows, and macOS — the macOS-specific
  flake disappears because the root cause (clock-resolution dependence) is
  removed.
- Aligns with how `generate_branch`/`generate_call_id` already work
  (cryptographic randomness via `uuid` v4).

- `cargo test --workspace --exclude gabby -- --test-threads=1`: 14 binaries,
  all green, 0 failed across ~2820 tests.
- Targeted: `registration::tests::test_needs_refresh`,
  `test_needs_refresh_far_future`, both `test_generate_tag_unique` instances,
  and `test_generate_tag` all pass repeatedly on Windows.
- `cargo clippy --workspace --exclude gabby --all-targets -- -D warnings`:
  exits 0.

## Risks / follow-ups

- The `thread::sleep(1ms)` workaround in
  `crates/rsiprtp/src/sip/message.rs:2086` is now redundant. Left in place
  rather than touching unrelated code — the test still passes and removing
  the sleep is purely cosmetic. Worth tidying in a future cleanup pass if
  ever revisited.
- No `Cargo.toml` changes were needed; `rand` was already a dependency.
- No public API changes: `generate_tag`'s signature is unchanged.
