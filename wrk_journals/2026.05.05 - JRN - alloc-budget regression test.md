# JRN — Allocation-budget regression test for SIP INVITE parsing

**Date opened:** 2026-05-05
**Goal:** Lock in allocation gains from the SIP message parser rewrite by adding an allocation-count regression test. "Parsing a typical INVITE allocates ≤ N times."

## Why
A peer agent flagged that without an allocation oracle the rewrite's perf gains can silently regress. A counted-allocation test is a cheap, stable oracle: it converts a perf property into a deterministic assertion machines can check at scale.

## Plan / decisions
- Supervisor mode: agents do all reading, writing, reviewing.
- TDD-ish: pick a representative INVITE fixture, measure the actual allocation count + bytes, then assert with a small headroom. Encode N once and let regressions fail it.
- Methodology candidates:
  - `stats_alloc` — small, stable, lets us scope counters with a guard. Good for unit tests.
  - `dhat` — heavier, adds a feature flag, profile-grade reporting. Overkill for a per-PR regression test.
  - Custom counting global allocator wrapping `System` — zero new deps, full control, but more code.
- Lean toward `stats_alloc` if we can keep it dev-dep + cfg(test). If feature gating becomes ugly, fall back to a tiny custom allocator inside the test crate.

## Workstream
1. **Explore** — map SIP parser entry, existing tests, decide which crate hosts the test.
2. **Design** — pin allocator approach + INVITE fixture + budget.
3. **Implement** — TDD: write test that fails with sentinel, then set real budget.
4. **Review** — code-reviewer subagent.
5. **Apply feedback** — code-review-apply-feedback subagent.
6. **Verify** — independent agent confirms test compiles, runs, locks budget. Iterate to 100%.
