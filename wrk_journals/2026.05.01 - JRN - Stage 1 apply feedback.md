## Required-fix outcomes

### 1. README architecture section (R1 in review)

**What was wrong:** the Architecture section described `rsiprtp` as "a
workspace of focused crates layered from foundations (`rsiprtp-core`)" —
which is the pre-collapse story. Anyone reading the published README on
crates.io would see a description that doesn't match what they're depending
on. The "Transport and security" feature bullets also overstated DTLS-SRTP
support.

**What was changed:**
- Architecture section rewritten to describe a single crate with module
  layering rooted at `rsiprtp::core`, with the prelude offering a flat
  import surface and every module also reachable directly.
- "Architecture" feature bullet updated from "Layered workspace of focused
  crates, re-exported through one facade" to "Single crate organized into
  focused modules with a flat `prelude` import surface".
- "Transport and security" bullet reworded from "SRTP encryption and
  DTLS-SRTP key exchange" to "SRTP encryption with SDES key exchange
  (DTLS-SRTP framing types are present, but the DTLS handshake itself is
  not yet implemented)" — see DTLS investigation below for why this is the
  truthful framing.
- "ICE / STUN / TURN crates" → "ICE / STUN / TURN modules".

**Why:** ship documentation that matches the published artifact.

### 2. ARCHITECTURE.md Mermaid diagram and prose (R2)

**What was wrong:** ~23 references to deleted crate names
(`rsiprtp-core`, `rsiprtp-sip`, …, `rsiprtp-session`), a Mermaid crate-graph
diagram with one node per former crate, and an "Application Layer →
Facade → …" structure that no longer exists.

**What was changed:**
- Mermaid diagram rewritten: the "Facade" subgraph is now a single
  "Crate" subgraph with one `rsiprtp` node; every former-crate node is
  relabelled to its module form (`rsiprtp::session`, `rsiprtp::dialog`,
  etc.). The graph topology is unchanged — only labels.
- Prose section "Crate layering" → "Module layering"; the bulleted
  responsibility list now uses `rsiprtp::<module>` throughout. The
  closing bullet about the facade crate was removed (no facade to
  describe).
- Sans-IO description ("`rsiprtp-session` layer happens to use Tokio")
  updated to `rsiprtp::session`.
- The trailing "transaction crate can drive a UDP UA…" updated to
  "transaction module".
- "Further reading" pointer changed from `crates/<name>/src/lib.rs` to
  `crates/rsiprtp/src/<module>/mod.rs`.

**Why:** ARCHITECTURE.md is reachable from the README via a relative link
and is the canonical architecture reference for contributors. Verified
clean post-edit: `Grep rsiprtp- ARCHITECTURE.md` returns no matches.

### 3. deny.toml stale rationale (R3)

**What was wrong:** the audit-trail comments and `[bans]` rationale
described a "single-facade publishing plan with path-only workspace deps"
that no longer applies. The published `rsiprtp` crate has zero path-only
deps now — every dependency is a versioned crates.io ref.

**What was changed:**
- Top-level "Advisory ignores" comment for RUSTSEC-2023-0071 updated:
  "rsiprtp-transport tests" → "the transport-layer tests".
- Top-level "Bans" rationale rewritten: removed the
  "single-facade with path-only deps" explanation; replaced with "the
  published `rsiprtp` crate has no path-only workspace deps … `cargo
  publish` already enforces correct version specifiers; we keep `'warn'`
  as a conservative fallback so accidental `*` in dev/test deps surface
  as a warning rather than a hard build break."
- Inline ignore-entry comment for RUSTSEC-2023-0071 (line ~122) updated
  the same way.
- Inline `wildcards = "warn"` comment block (line ~142–152) rewritten to
  match the new rationale; `allow-wildcard-paths = true` is kept as

**Behaviour unchanged:** `wildcards = "warn"`, license allow list,
advisory ignores all retained. `cargo deny check` still reports
`advisories ok, bans ok, licenses ok, sources ok`.

**Why:** the audit trail is only valuable if it's accurate. Per task
brief: "keep the audit trail accurate."

### 4. DTLS feature investigation — **Case A** confirmed

**Investigation evidence:**
- `Grep "use openssl|openssl::|extern crate openssl" crates/rsiprtp/src/`
  returned **zero hits in source code**. Only matches were two doc-comment
  prose mentions in `crates/rsiprtp/src/srtp/dtls.rs:10` and `:32`
  ("Implementation Options: rustls + webrtc-rs / openssl / ring"), both
  describing what *could* be done in the future.
- `Grep 'feature\s*=\s*"dtls"' crates/rsiprtp/src/` returned **zero hits**.
- `git show 0e9478b:crates/rsiprtp-srtp/src/dtls.rs` (the original
  pre-collapse file) confirmed: no `use openssl` anywhere, ever. The dtls
  module has always been pure-Rust framing/parsing using `sha2` and
  `sha1` only.
- `git show 0e9478b:crates/rsiprtp-srtp/Cargo.toml` shows `openssl =
  { version = "0.10", optional = true }` and `dtls = ["openssl"]` —
  **declared but never imported**. The feature was a placeholder for
  future DTLS handshake work that never landed.

**Decision: Case A.** The DTLS module never directly used openssl. The
feature flipped a heavy native C dep on for nothing. Per task brief:
"drop the `dtls` feature and the `openssl` optional dep entirely … be
honest."

**Action taken:**
- `crates/rsiprtp/Cargo.toml`: removed `dtls = ["openssl"]` from
  `[features]`. Removed the `openssl = { version = "0.10", optional =
  true }` line and its `# DTLS (optional, gates openssl native dep)`
  comment.
- `crates/rsiprtp/src/lib.rs:39`: changed
  `[`srtp`]: SRTP encryption (DTLS-SRTP gated by `dtls` feature)`
  to a two-line bullet that says SRTP/SDES is implemented and DTLS-SRTP
  framing types are present but the handshake is not yet implemented.
- `crates/rsiprtp/src/srtp/dtls.rs:53`: doc snippet `use crate::srtp…`
  updated to `use rsiprtp::srtp…` (also addresses S3 below for this
  file).
- README "Transport and security" bullet rewritten to match (see R1).

**Impact:**
- No more "opt-in cost with zero opt-in benefit" feature shipping to
  crates.io.
- `cargo build --features dtls` now correctly errors with `the package
  'rsiprtp' does not contain this feature: dtls` rather than spending
  build time compiling openssl-sys for nothing.
- `package.metadata.docs.rs` still has `all-features = true`; with no
  features declared this is now equivalent to default. Verified
  `cargo doc -p rsiprtp --no-deps --all-features` builds cleanly.
- The validation gate "build with `--features dtls` either succeeds OR
  fails with a clean openssl-unavailable error" no longer applies — the
  feature is gone. The remaining gates (1, 3, 4, 5) all pass.
