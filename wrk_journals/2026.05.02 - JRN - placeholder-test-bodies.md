# Implement placeholder test bodies in baresip integration suite

**Started:** 2026-05-02
**Author:** Claude (Opus 4.7) with Arthur

## Background

In the previous TODO audit we found that ~10 tests across `crates/rsiprtp/tests/baresip_integration/scenarios/registration.rs` and `registration_advanced.rs` are placeholders — bodies are either empty, just `println!("...placeholder")`, or set up framework state without exercising it. They are `#[ignore]`'d, so they don't contaminate `cargo test`, but with `--include-ignored` they pass without asserting anything.

A green test that asserts nothing is worse than a missing test. Arthur asked to implement them properly so the periphery is verified before publishing. Honest scope on what we ship matters.

### Recalibration vs the earlier audit

I originally claimed 16 placeholders. After re-reading more carefully:

- The `println!("...test completed")` lines in `error_recovery.rs` and `call_transfer.rs` are **end-of-test markers in tests that DO have real assertions**. Not placeholders.
- The actual placeholders (empty body / `println!("...placeholder")`) are concentrated in `registration.rs` (4) and `registration_advanced.rs` (6).

So the real work scope is **10 tests across 2 files**.

## Surveyed primitives

### What's already there

- `crates/rsiprtp/src/sip/auth.rs` — full RFC 2617/3261/7616 digest helpers:
  - `DigestChallenge::parse` — parses `WWW-Authenticate` header values
  - `DigestResponse::from_challenge` — generates digest hash from credentials
  - `DigestCredentials::new(user, pass)`
  - All publicly re-exported via `rsiprtp::sip::{DigestChallenge, DigestResponse, DigestCredentials}` (and `prelude`)
- `crates/rsiprtp/tests/baresip_integration/framework/sip_endpoint.rs::TestEndpoint`:
  - `send_raw(msg, dest)` — fires raw bytes
  - **No `recv_raw`** — only `wait_for_incoming`/`wait_for_answer` which expect specific message types
  - Has `sip_socket: Arc<UdpSocket>` — could expose
- `crates/rsiprtp/tests/baresip_integration/framework/asterisk.rs::AsteriskInstance`:
  - Spawns local `asterisk` binary (not Docker) with generated `pjsip.conf` containing test users 1001/1002/1003 with passwords `test1001`/`test1002`/`test1003`
  - `is_available()` returns true if `asterisk -V` runs
  - AMI control via `connect_ami` / `send_ami_action`

### What's missing

- A `recv_raw(timeout)` method on `TestEndpoint` so tests can wait for arbitrary responses (we need this to read 401 challenges and 200 OK to REGISTER)
- The `registration.rs` placeholders are gated on `is_baresip_available()` — that's wrong, those tests don't use baresip. Need to remove the gate and the `#[ignore]`.

## Strategy

### `registration.rs` (4 placeholders) — local-pair approach, NO external deps

Run a minimal in-test "registrar" as a second `tokio::net::UdpSocket` bound to a free port. The `TestEndpoint` builds and sends raw REGISTERs; the registrar parses and responds. This means these tests can run on any dev box, on any CI, with no Asterisk install — and they actually assert.

Tests:
1. `test_register_message_format` — Send REGISTER, registrar verifies required headers (Via, From, To, Call-ID, CSeq, Contact, Expires), responds 200 OK. Endpoint verifies 200 OK received.
2. `test_register_with_authentication` — Send REGISTER, registrar issues 401 with WWW-Authenticate, endpoint uses `DigestResponse::from_challenge` to build authenticated REGISTER, registrar verifies digest, responds 200 OK.
3. `test_register_refresh` — Register, then send a second REGISTER on the same Call-ID with incremented CSeq. Registrar verifies CSeq, responds 200 OK.
4. `test_unregister` — Register normally, then send REGISTER with `Expires: 0`. Registrar verifies and responds 200 OK with `Expires: 0`.

These should:
- **Drop the `#[ignore]` and `is_baresip_available()` gate** (they're misattributed)
- Run as part of `cargo test --workspace` (they're in `tests/` so they will)

### `registration_advanced.rs` (6 placeholders) — Asterisk-gated, real PBX

Keep Asterisk gating (these test against real registrar behavior that the local shim wouldn't model). Change is in test bodies, not test gating:

1. `test_register_with_asterisk` — wire the auth round-trip using `DigestChallenge::parse` + `DigestResponse::from_challenge`, verify final 200 OK
2. `test_register_with_expiry` — register with custom `Expires`, verify Asterisk's reply Expires
3. `test_registration_refresh` — register, refresh, query AMI for `PJSIPShowEndpoint` to confirm contact still present
4. `test_unregister_expires_zero` — register, then unregister, query AMI to confirm contact removed
5. `test_register_multiple_contacts` — register from two endpoints (different ports) as same user, query AMI to confirm both contacts
6. `test_register_contact_params` — REGISTER with `Contact: <...>;expires=N`, verify accepted

These remain `#[ignore = "requires asterisk to be installed"]` but the bodies will actually assert when run.

### Framework additions

Add `TestEndpoint::recv_raw(timeout) -> Result<(Vec<u8>, SocketAddr)>` so tests can read arbitrary SIP responses without the canned `wait_for_*` flows.

### 2026-05-02 — `registration.rs` complete

Added `TestEndpoint::recv_raw(timeout)` — small parity addition with the existing `send_raw`.

Rewrote `registration.rs` from 4 placeholder bodies to 4 fully-asserting tests + 4 unit tests on local helpers. Each scenario test spins up a `tokio::net::UdpSocket` as a minimal in-test registrar and exchanges real SIP messages.

- **`test_register_message_format`** — registrar verifies the REGISTER carries Via/From/To/Call-ID/CSeq(REGISTER)/Contact/Expires; client verifies the 200 OK echoes the Call-ID and includes `expires=` on the Contact.
- **`test_register_with_authentication`** — client receives a real 401 challenge, parses it via `DigestChallenge::parse`, computes the Authorization via `DigestResponse::from_challenge`, sends authenticated REGISTER. Registrar extracts the client's `cnonce`/`nc`/`response`, recomputes the MD5 digest with the known password, and asserts they match byte-for-byte. This is the meat — it would have caught any breakage in the digest helper invocation.
- **`test_register_refresh`** — same Call-ID, incrementing CSeq across two REGISTERs.
- **`test_unregister`** — REGISTER with `Expires: 0`; registrar asserts the value, response echoes `Expires: 0`.

Dropped `#[ignore]` and the `is_baresip_available()` gate from these — they were misattributed; these tests don't use baresip.

Added two dev-dependencies (`md5`, `hex`, both already in the workspace tree) so the test registrar can recompute the expected digest. Cleaner than reaching into private helpers.

### 2026-05-02 — `registration_advanced.rs` complete

Rewrote all 6 placeholder bodies. Built a shared `authenticated_register(...)` helper inside the file that drives the full unauth-REGISTER → 401 → digest → authenticated-REGISTER → 200 flow against Asterisk and returns the final 200 OK as a string for per-test assertions.

Per test:

- **`test_register_with_asterisk`** — completes the auth round-trip. Asserts 200 OK and that the response carries an `expires=` parameter on Contact or an Expires header (registrar must indicate the binding lifetime).
- **`test_register_with_expiry`** — registers with `Expires: 120`, asserts the granted lifetime is in `(0, 120]`. Catches both "ignored expiry" and "silently dropped to default 3600".
- **`test_registration_refresh`** — registers once, then again. Asserts both succeed with 200 OK.
- **`test_unregister_expires_zero`** — registers, then re-registers with `Expires: 0`. Asserts the 200 OK signals removal (no Contact, or `expires=0`).
- **`test_register_multiple_contacts`** — registers two endpoints (different SIP ports) under the same AOR `1001`. Both must return 200 OK; pjsip.conf has `max_contacts=5` so neither should be evicted.
- **`test_register_contact_params`** — registers with `Contact: <...>;expires=90;q=0.7` and a contradicting header-level `Expires: 3600`. Asserts the granted expiry follows the per-Contact value (RFC 3261 §10.2.4).

Plus 4 unit tests on the local helpers (`build_register`, `find_header`, asterisk-availability check).

#### Skip semantics

The 6 main tests stay `#[ignore = "requires asterisk to be installed"]`. With `--include-ignored` on a box with no Asterisk, they hit the early-return (`spawn_asterisk_and_endpoint` returns `None`) and "pass" without asserting. **This is the same `#[ignore]` skip behaviour the rest of the suite uses.** It's a known-and-accepted limitation of cargo's test gating; the bodies are real and *will* assert when an Asterisk is reachable. Documented this in the file's module doc.

When run on a box with Asterisk (verified locally not yet — see Verification below), the test names will print real pass/fail.

## Risks / unknowns

- Asterisk is probably not installed on this Windows dev box. I will write the bodies, run `cargo check`/`clippy`, and run only the non-Asterisk tests. The Asterisk-gated ones will be exercised the next time someone has Asterisk available.
- `pjsip.conf` test users use MD5 (Asterisk default). If pjsip happens to challenge with SHA-256, our digest helpers handle that too — no issue.
- Loopback UDP timing: real Asterisk takes 100–500ms to start; for the local-pair tests we don't have that latency, so timeouts can be tight (e.g. 1s).
