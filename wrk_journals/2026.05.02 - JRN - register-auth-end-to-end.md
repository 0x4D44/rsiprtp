# 2026-05-02 — Wire REGISTER + digest auth end-to-end

## Goal

Close the most embarrassing gap: REGISTER with digest auth is unwired.
Each caller (the `basic_call` example, `asterisk_integration.rs`) hand-rolls the
bind / send / recv / 401 / retry / 200 loop. Build a reusable async driver
on top of the sans-IO `RegistrationManager` and prove it works.

  `RegistrationManager` with `create_register`, `create_register_with_auth`,
  `create_unregister`, `handle_response`. Looks complete; 401/407 paths covered

- Digest auth: `crates/rsiprtp/src/sip/auth.rs` — MD5 + SHA-256, qop auth /

- Transport: `UdpTransport::bind / send_to / recv` is async-friendly.
- Existing live test: `crates/rsiprtp/tests/asterisk_integration.rs`. The
  `test_register_with_asterisk` test re-implements the loop inline. Skips

So the gap is **the glue layer** — a reusable async driver — and the **proof
that the wire bytes actually work** (recompute the MD5 server-side, not just
trust that the manager built a valid header).

## Design

Add async driver methods directly on `RegistrationManager`. Keeps the sans-IO
core intact (the existing methods stay) and lets callers do
`reg.register(&transport, server_addr).await?`. Symmetric `unregister` method.

Auth retry policy: at most **one** digest round-trip per call. Asterisk
(and any RFC-3261 registrar) issues exactly one 401 → expects retry → 200.
A second 401 means the credentials are wrong, not a stale-nonce loop —
return an error, don't pile on retries. Provisional 100-class responses are
swallowed by `handle_response`, so the driver just keeps reading until a
final response arrives.

Timeouts: per-recv timeout of 5s (matches existing test). Configurable via a
`RegistrarTimeouts` struct so tests can use shorter values.

## Implementation log

### Step 1 — driver

Added two async methods on `RegistrationManager`:

- `register(&transport, registrar_addr, recv_timeout, total_timeout)` — sends
  the initial REGISTER, waits for a response, dispatches it through the
  existing sans-IO `handle_response`, and on `Some(retry)` sends the auth
  retry. At most one digest retry; a second 401/407 returns
  `RegistrationError::AuthRejected` (almost always wrong credentials, never
  a useful retry).
- `unregister(...)` — symmetric for `Expires: 0`.

Shared loop body in private `run_request()`. Internals:
- `1xx` responses → `handle_response` returns `Ok(None)` and `status < 200`,
  so the loop re-reads.
- `2xx` → `Ok(())`, manager state already set to `Registered` /
  `Unregistered` by `handle_response`.
- `4xx/5xx/6xx` (non-401/407) → `handle_response` returns
  `Err(Failed(code, reason))`, propagated.
- Network/parse failures get their own variants (`Transport`, `Malformed`).

`total_timeout` bounds wall time; `recv_timeout` bounds individual `recv`
calls. The driver picks `min(recv_timeout, remaining_total)` so a
`total_timeout` smaller than `recv_timeout` still works.

### Step 2 — proof against an Asterisk-shaped registrar

`crates/rsiprtp/tests/register_auth_e2e.rs` runs an in-process mock UDP
registrar that:
- Issues a 401 with `WWW-Authenticate: Digest realm="rsiprtp-mock",
  nonce="<fixed>", algorithm=MD5[, qop="auth"]`.
- Recomputes the expected MD5 digest server-side (HA1/HA2/response per
  RFC 2617 §3.2.2.1) using the known password.
- Accepts with 200 OK only if the client's `response=` field matches
  byte-exactly. Mismatch → 403 Forbidden.

Five scenarios:
1. `register_with_digest_md5_no_qop` — basic flow.
2. `register_with_digest_md5_qop_auth` — exercises cnonce / nc.
3. `register_wrong_password_is_rejected` — drives the 403 path; verifies
   `RegistrationError::Failed(403, _)`.
4. `register_times_out_when_registrar_silent` — silent server,
   `total_timeout=400ms` → must return `Timeout`, not hang.
5. `register_then_unregister` — full lifecycle; verifies the unREGISTER
   reuses the stashed challenge (single round-trip) and carries
   `Authorization` + `Expires: 0`.

All 5 pass in 0.22s on my machine.

## Risks & follow-ups

- **Stale-nonce loop**: a registrar that issues `stale=true` on its 401
  expects the client to retry with a fresh `nc` against a new nonce. My
  driver treats a second 401 as `AuthRejected`. If we hit a registrar
  that legitimately rotates nonces under load, this would surface as
  spurious auth failures. Asterisk doesn't do this for REGISTER in
  practice — but real-world SIP trunks might. Defer until we see it.
- **Refresh timer**: `RegistrationManager::needs_refresh()` exists but
  nothing drives it from the new helpers. A long-running registrar would
  expire after `expires` seconds. Out of scope here; the driver is
  one-shot.
- **TCP/TLS transport**: `register()` is hard-bound to `UdpTransport`.
  Generalizing to TCP/TLS is non-trivial because the connection model
  differs. Punt until someone needs it.
