## Required Changes

### 1. `D:\language\rsiprtp\README.md:108-114` — Architecture section is now factually wrong

The README still reads:
> `rsiprtp` is a workspace of focused crates layered from foundations
> (`rsiprtp-core`) up through transport, media, transactions, dialogs, and
> finally a session layer ... The top-level `rsiprtp` crate re-exports the pieces most consumers want via `rsiprtp::prelude::*`.

After the collapse, `rsiprtp` is a **single crate with internal modules** — there is no workspace of focused crates from a consumer's perspective, and there is no `rsiprtp-core`. This text ships in the published tarball; users opening the README on crates.io will see a description that doesn't match what they're depending on.

**Suggest:** rewrite to "`rsiprtp` is a single crate organized into modules layered from foundations (`rsiprtp::core`) up through ..." — the rest of the paragraph carries over with `rsiprtp-X` swapped for `rsiprtp::X`.

The HLD's step 12 only said "Trim README" for the ICE bullet; it did not call out the architecture paragraph. This is a reviewer-found regression, not a missed HLD step.

### 2. `D:\language\rsiprtp\ARCHITECTURE.md:1-100+` — 23 references to deleted crate names

The README links to `[ARCHITECTURE.md](ARCHITECTURE.md)`. ARCHITECTURE.md has a Mermaid crate-graph diagram and prose with **23 references to `rsiprtp-{core,sip,...}`** crate names that no longer exist. Lines 27, 28, 32, 36, 37, 41, 42, 43, 47, 48, 52, etc. all describe a layered workspace structure.

This file isn't shipped in the published tarball, but it is the canonical architecture documentation reachable from the README via a relative link, and it will be the first thing contributors and curious users see. Leaving it stale undermines the quality of what we just shipped.

**Suggest:** rewrite the `subgraph` blocks to use module names (`rsiprtp::session`, `rsiprtp::dialog`, etc.) and the surrounding prose to describe modules, not crates. The diagram structure can stay the same — only labels change.

### 3. `D:\language\rsiprtp\deny.toml:32-37, 122, 142-152` — stale "single-facade with path-only deps" rationale

deny.toml has two now-incorrect comment blocks:
- Line 32-37 / 142-152 explain why `wildcards = "warn"` rather than `"deny"` because the "single-facade publishing plan uses path-only workspace deps". After the collapse, the published crate has zero path deps — it's a real single crate with versioned workspace deps. The justification is obsolete; `wildcards = "deny"` should now be safe (and arguably preferred for a publishable crate).
- Line 27, 122 reference "rsiprtp-transport tests" by the old crate name.

The HLD called this out explicitly at line 348-349: "Expect to simplify the `wildcards = 'warn'` rationale comment in `deny.toml` since the 'single-facade with path-only deps' no longer applies." The implementer didn't action this.

**Suggest:** flip `wildcards = "deny"` and rewrite the rationale, OR keep `"warn"` with a one-line "we're conservative" note. Update the line-27/122 comments to say "rsiprtp transport-layer tests" or similar.

## Suggestions

### S1. `D:\language\rsiprtp\crates\rsiprtp\Cargo.toml:69` — drop unused `crossbeam` and `dasp` deps

`crossbeam.workspace = true` and `dasp.workspace = true` were faithfully migrated from the old `rsiprtp-media/Cargo.toml` but **neither is actually `use`d anywhere in the merged source tree** (`grep -r "crossbeam\|dasp" crates/rsiprtp/src` returns nothing). They were dead in `rsiprtp-media` too — pre-existing waste, not introduced by Stage 1, but the consolidation is the natural moment to clean it up. Dropping them shrinks the published dep graph.

If you'd rather keep the diff minimal in Stage 1, leave it for a follow-up — but flag it now so it doesn't get forgotten.

### S2. `D:\language\rsiprtp\crates\rsiprtp\src\sip\message.rs:3` — redundant `#![allow(unexpected_cfgs)]`

Now that `[workspace.lints.rust] unexpected_cfgs = { check-cfg = ["cfg(coverage)", "cfg(coverage_nightly)"] }` is in the workspace root, this module-level `#![allow(unexpected_cfgs)]` is redundant. It still works (it's an inner attribute scoping to this file) but is dead noise. Removing it makes the file consistent with the rest of the tree (no other module has the equivalent).

### S3. Inner-module doctests use `crate::` instead of `rsiprtp::`

Three `rust,ignore` doctest snippets were rewritten from `rsiprtp_X::` to `crate::X::`:
- `D:\language\rsiprtp\crates\rsiprtp\src\session\bitrate_bridge.rs:27`: `use crate::session::BitrateBridge;`
- `D:\language\rsiprtp\crates\rsiprtp\src\session\hold.rs:18`: `use crate::session::{HoldManager, ...};`
- `D:\language\rsiprtp\crates\rsiprtp\src\transport\resolver.rs:11`: `use crate::transport::resolver::{...};`

Doctests run as external binaries — `crate::` does not refer to `rsiprtp` in that context. These would fail to compile if not `,ignore`d. They're harmless today but mislead readers of the rendered docs (the example shown to a user importing the crate uses the wrong path). Compare to e.g. `session/mod.rs:11` which correctly uses `use rsiprtp::session::{CallManager, ManagerConfig};`.

**Suggest:** swap `crate::` for `rsiprtp::` in these three snippets so the docs show consumers the correct import.

### S4. Stale comments referencing old crate names in source

`grep -n "rsiprtp-" crates/` finds stale comments in:
- `crates/rsiprtp/src/sip/tests/mod.rs:1` — "Unit tests for rsiprtp-sip crate."
- `crates/rsiprtp/src/sdp/tests/mod.rs:1` — "Unit tests for rsiprtp-sdp crate."

- `crates/gabby/src/server.rs:118` — "// Parse using rsiprtp-sip"

None affect compilation. Trivial polish; address with a single sweep.

### S5. Optional `openssl` dep is declared but unused

`crates/rsiprtp/Cargo.toml:84` declares `openssl = { version = "0.10", optional = true }` and `[features] dtls = ["openssl"]`, but no source file actually has `use openssl::...` anywhere — the `srtp::dtls` module is currently pure-Rust framing/parsing with no openssl call sites. Pre-existing from `rsiprtp-srtp`; not introduced by Stage 1. The feature exists today as a placeholder for future DTLS handshake work. Worth flagging because the Validation-2 gate "build with `--features dtls`" only proves the openssl crate compiles, not that anything actually uses it.

Not a blocker — out of Stage 1 scope. Just noting it so future stages don't assume gate-2 means anything substantive.

### S6. `tokio` features

The old `rsiprtp-transport/Cargo.toml` requested `tokio = { workspace = true, features = ["net", "sync", "rt", "macros", "io-util"] }`. The merged crate just says `tokio.workspace = true`, which inherits the workspace `["full"]`. Since `["full"]` is a strict superset of those features, this is correct — but worth double-checking with the supervisor that we're OK shipping the published crate with `tokio = { features = ["full"] }` rather than a tighter feature set. Tighter features = smaller compile time for downstream users. Out of Stage 1 scope unless you want to optimize now.
