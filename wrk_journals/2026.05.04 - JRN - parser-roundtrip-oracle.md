# Journal — SIP parser round-trip oracle

**Date started:** 2026-05-04
**Author:** Claude (with Arthur)
**Design:** `wrk_docs/2026.05.04 - HLD - SIP parser round-trip oracle.md`

## Plan summary

Build a round-trip oracle that asserts a fixed-point invariant on
the parse/serialize cycle:

```
m1 = Message::parse(input)?
m2 = Message::parse(&m1.to_bytes())?
m3 = Message::parse(&m2.to_bytes())?
assert_eq!(m2, m3);
assert_eq!(m2.to_bytes(), m3.to_bytes());
```

If first parse fails, return silently — round-trip is undefined.

Three new files, no changes to existing code:

1. `crates/rsiprtp/tests/parser_roundtrip_oracle/mod.rs` — the oracle.
2. `crates/rsiprtp/tests/parser_roundtrip.rs` — fixture driver.
3. `fuzz/fuzz_targets/sip_message_roundtrip.rs` — libfuzzer target.

Plus one `fuzz/Cargo.toml` entry.

### 11:56 UTC — first finding (round-trip #1)

After 6,787,263 executions (~5 minutes wall), the first non-fixed
point surfaced. Failure mode is **second parse failed** —
serializer-emits-bytes-our-parser-cannot-accept.

**Shape:**

`m1` is a `Response` whose reason-phrase contains bare LF +
control bytes (NUL, `\u{13}`, `\u{17}`, `\u{18}`, `\u{c}`,
`\u{16}`) along with embedded `\r` runs. RFC 3261 §25.1 grammar:

```
Reason-Phrase = *(reserved / unreserved / escaped /
                  UTF8-NONASCII / UTF8-CONT / SP / HTAB)
```

CR, LF, NUL, and other CTL bytes are NOT in this production. Our
parser accepts them anyway; the serializer faithfully emits them
verbatim onto the start line; the re-parse stops at the first LF,
treats the bytes after as headers, and surfaces
`InvalidHeader("missing ':' in header: ...")`.

**Status:** the previous M11 session pinned bare-LF-in-reason-
phrase as finding #13 (`cd5911d`) — but the pin was a *skip in the
differential oracle*, not a parser tightening. The round-trip
oracle exposes that as long as the parser still accepts these
bytes, the serializer emits them, and re-parse breaks. The
differential oracle's structural blind spot exactly as predicted
in the design doc.

**Triage options to surface to user:**

1. **Tighten `parse_status_line`** to reject the full set of
   non-Reason-Phrase chars per §25.1 (CR, LF, NUL, all CTL bytes
   except SP/HTAB). This is the RFC-correct fix and makes the
   parser strictly more secure (anti-smuggling, anti-injection).
   May require updating the differential pin #13 since both
   parsers will then reject. Estimated diff: small.
2. **Pin in the round-trip oracle** — add a skip path for
   reason-phrases containing CR/LF/CTL. Cheaper but accepts a
   known parser-leniency.
3. **Tighten the serializer** to re-validate before emit. Worse
   than (1) — papers over a parser bug.

(1) is the right answer per the project's RFC-strict trajectory
(see commit `45c595f`: "tighten parse_status_line per RFC 3261
§7.2"). Surfacing to user before I implement.

### 17:30 UTC — fix landed (option 1)

User picked option 1: tighten `parse_status_line` per RFC 3261
§25.1. Implementation:

- `crates/rsiprtp/src/sip/parser/framing.rs` — after the existing
  status-code range check, reject any byte in the reason phrase
  that is `0x7F` (DEL) or `< 0x20` and not `0x09` (HTAB). Error
  message: `"reason phrase contains forbidden control byte 0xNN"`.
- 6 new unit tests in the framing `#[cfg(test)]` module: bare CR,
  bare LF, NUL, DEL all reject; HTAB and high-bit (UTF8-NONASCII)
  accept.
- New differential pin
  `status_line_reason_ctl_byte_rsip_accepts_we_reject` in
  `tests/parser_diff.rs` documenting the new asymmetry (rsip 0.4
  still absorbs CTL bytes into the reason).
- Differential oracle's `(Ok, Err)` skip extended with
  `"reason phrase contains forbidden control byte"` so libfuzzer
  doesn't rediscover the divergence on every mutation.

**Pin #13 unchanged.** Its specific test bytes
(`b"SIP/2.0 202 \nNotAHeader\r\n\r\n\xac "`) consume the bare LF
via `framing::split_first_line`'s LF fallback as the start-line
terminator, so the LF never reaches `parse_status_line`. The
header parser still surfaces `"missing ':'"` on `NotAHeader`. Pin
behavior is exactly as before.
