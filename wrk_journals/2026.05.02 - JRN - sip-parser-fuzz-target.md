# 2026-05-02 — JRN — Add cargo-fuzz target for SIP parser

## Goal
Add **one** cargo-fuzz target that drives the SIP message parser exposed at
`rsiprtp::sip::SipMessage::parse(&[u8])`. This is the broadest external
attack-surface entry: every byte slice arriving on UDP/TCP/TLS goes through
this function before any session-layer logic runs, so any panic/UB here is a
remote denial-of-service primitive.

## Plan
1. Confirm cargo-fuzz tooling is present (`cargo fuzz --version`).
2. Confirm the parser entry point is reachable from outside the crate
   (`rsiprtp::sip::SipMessage::parse`).
3. Create the standard `crates/rsiprtp/fuzz/` skeleton manually (not via
   `cargo fuzz init`) so we don't perturb the workspace beyond what's needed:
   - `Cargo.toml` (its own workspace; `path = ".."` dep on rsiprtp)
   - `.gitignore` for `target/`, `corpus/`, `artifacts/`
   - `fuzz_targets/sip_message_parse.rs`
4. Make the harness call `SipMessage::parse(data)`. When parsing succeeds,
   exercise the cheap accessors (`method`, `uri`, `cseq`, `via_branch`,
   `from_tag`, `to_tag`, `call_id`, `body`, `content_type`) so that the
   fuzzer also covers our wrapper code, not just `rsip`.
5. Seed the corpus with a few valid SIP messages so libFuzzer has structure
   to mutate from.
6. Verify the harness compiles via `cargo +nightly fuzz build sip_message_parse`.
7. Document run instructions in a short `fuzz/README.md`.

## Notes / decisions

- **Why one target, not several.** The user asked for one. The natural
  choice is the message-level parser because it dominates attack surface;
  individual header parsers (`Via::parse`, `Contact::parse`, ...) and the
  URI parser (`SipUri::parse`) are reachable transitively from the message
  parser since `SipMessage::parse` triggers `rsip` to tokenize the headers,
  and our follow-on accessor calls hit our own typed-header code paths.
- **Why exercise accessors.** Just calling `parse` would mostly exercise
  the third-party `rsip` crate. The accessor calls walk our `headers.rs`
  parsers (`Via::typed`, `From::typed`, etc.) so the fuzzer covers
  `crates/rsiprtp/src/sip/{message,headers,uri}.rs` too.
- **Workspace isolation.** The root `Cargo.toml` uses explicit `members =
  [...]` (not glob), so `crates/rsiprtp/fuzz/` is *not* picked up by the
  main workspace. The fuzz crate gets its own `[workspace] members = ["."]`
  block to make it self-contained.
- **Toolchain.** cargo-fuzz needs nightly. `nightly-x86_64-pc-windows-msvc`
  is already installed on this box. On Windows libFuzzer ships with the
  rustc nightly distribution, so `cargo +nightly fuzz build` should work.

## Risks / open questions

- The accessor calls return `Result` for missing headers — this is a
  normal control-flow path and we deliberately ignore the result to keep
  the fuzzer focused on panics, not error variants.

## Issues hit during smoke run

1. **`STATUS_DLL_NOT_FOUND` on first run.** cargo-fuzz on Windows builds
   the binary with `-Zsanitizer=address` by default; the resulting `.exe`
   dynamically links `clang_rt.asan_dynamic-x86_64.dll` which lives under
   the MSVC toolchain rather than on the default `PATH`. Workaround:
   prepend
   `C:\Program Files\Microsoft Visual Studio\18\Enterprise\VC\Tools\MSVC\14.50.35717\bin\Hostx64\x64`
   to `PATH` for the run. Documented this in `fuzz/README.md` so future
   runs aren't tripped up.
2. **Linker errors with `-s none`.** Tried `-s none` to skip ASan and got
   `LNK2001: unresolved external symbol __stop___sancov_pcs` from
   sancov-instrumented dependency rlibs. Conclusion: don't switch
   sanitizers between builds without `cargo +nightly fuzz cargo clean`.
   Stayed with the default ASan build.
