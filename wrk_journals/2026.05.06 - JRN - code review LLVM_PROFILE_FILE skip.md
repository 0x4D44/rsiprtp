## Change under review

```rust
#[test]
fn invite_allocation_budgets() {
    // Skip under `cargo llvm-cov`: coverage instrumentation injects heap
    // activity for `__llvm_profile_*` counters on every covered branch,
    // which inflates allocation counts ~5x and busts these tight budgets
    // without indicating a real parser regression. `LLVM_PROFILE_FILE` is
    // set by cargo-llvm-cov for every test process it spawns.
    if std::env::var_os("LLVM_PROFILE_FILE").is_some() {
        eprintln!("skipping allocation budget test under llvm-cov instrumentation");
        return;
    }
    // ... rest of test ...
}
```

## Summary

A 9-line runtime guard at the top of `invite_allocation_budgets` that early-returns
when `LLVM_PROFILE_FILE` is set, preventing the allocation budget assertion from
firing under `cargo llvm-cov` instrumentation.

## Overall Assessment

**Looks good - needs minor consideration.** The change is small, focused, well-commented,
and solves a real problem (coverage instrumentation poisoning a tight allocation oracle).
The trade-off (oracle silently disabled under coverage) is acknowledged with a tracked
follow-up (`*_BUDGET_UNDER_COVERAGE` env override).

## Strengths

- **Excellent comment** (lines 193-197): explains the why, not the what — names the
  instrumentation symbols (`__llvm_profile_*`), quantifies the impact (~5x), and
  identifies the reliable signal (`LLVM_PROFILE_FILE`).
- **Smallest blast radius**: runtime check vs. cfg/feature gates means no Cargo.toml or
  CI coordination, no conditional compilation drift.
- **`var_os` over `var`**: correct choice — avoids UTF-8 decode failure on a path env
  var. Good Rust hygiene.
- **`eprintln!` skip notice**: visible in test output (with `--nocapture`), so silent
  skip won't surprise an operator running `cargo llvm-cov` and wondering why budgets
  passed.
- **Early return placement**: before any work is done, including `Vec` allocation. Clean.

## Required Changes

**None.** The fix is correct, minimal, and matches the stated rationale.

## Suggestions

### Suggested - Test harness visibility
`eprintln!` in a `#[test]` is captured by libtest and only printed on failure or with
`--nocapture`. Under a passing `cargo llvm-cov` run the skip notice will be invisible.
Options:
- Accept invisibility (the comment in source is the durable record).
- Future: pair with the planned `*_BUDGET_UNDER_COVERAGE` override so the test does
  *something* observable under coverage rather than degenerating to a no-op.

### Suggested - Inline TODO marker
Oracle is silently disabled under coverage; this is already on the TODO list
(`*_BUDGET_UNDER_COVERAGE` env override). Worth a `// TODO:` marker inline next to the
early-return so future readers see it without grepping a journal. One line, zero risk.

### Nit - Skip mechanism semantics
Stable Rust has no `#[ignore_if]`, so an early-return is the idiomatic approach. The
test still reports as `ok` rather than `ignored`, which slightly misrepresents what
happened. Not worth fixing without a richer harness.

## Questions

### Q1 - Future-proofing the env-var sentinel
Is `LLVM_PROFILE_FILE` guaranteed to be set by *every* version of `cargo-llvm-cov` we
care about, or just current? If a future version changes the env-var name, the guard
silently stops working and the budget oracle starts firing under coverage again. A
defensive secondary check (e.g., also looking for `CARGO_LLVM_COV` or similar) would
be belt-and-braces, but probably over-engineering for a 9-line patch.

### Q2 - Stale env-var leakage
Could a non-coverage caller accidentally have `LLVM_PROFILE_FILE` set in their
environment (e.g., from a prior coverage run that leaked the var into a shell)? The
test would silently skip in that case. Probability is low; impact is "oracle disabled
until shell is cleaned" — same severity as the acknowledged trade-off, so not a
blocker.

## Verdict

**Ship it.** The 9 lines do exactly what they claim, the comment is durable, and the
known trade-off has a tracked follow-up.
