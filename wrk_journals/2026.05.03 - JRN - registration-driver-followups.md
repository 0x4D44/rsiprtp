# Deferred items

## D1 — De-duplicate the 407 handler (registration.rs:356-403)

**File:** `crates/rsiprtp/src/session/registration.rs`
**Symptom:** the 407 (Proxy Authentication Required) handling block in
`handle_response` inlines the same digest-builder pattern as
`create_register_with_auth` (lines 213-254). Two places must be kept in
sync if digest computation changes (e.g., when adding SHA-512/256 support,
or threading a new `userhash` parameter).
**Why it wasn't fixed tonight:** real refactor; needs care because the 407
path doesn't flip `self.state` to `Refreshing` (deliberately — the state was
already set by the prior `create_register`), and the 401 path uses
`Authorization` while 407 uses `Proxy-Authorization`. A naive helper
extraction loses both nuances.
**Suggested shape:**

```rust
fn build_authenticated_register(
    &mut self,
    challenge: &Challenge,
    header: AuthHeaderKind,  // Authorization | ProxyAuthorization
    advance_state: bool,     // false in the 407 path
) -> Result<SipRequest, RegistrationError>
```

Then both `create_register_with_auth` and the 407 branch call it.

## D2 — Defer cseq/nc increments until after `builder.build()` succeeds

**Files:** `crates/rsiprtp/src/session/registration.rs:188, 217, 259, 368`
**Symptom:** `self.cseq += 1` (and similarly `self.nc += 1` after tonight's
fix) happen before `builder.build()` may fail (e.g. on an invalid registrar
URI). On failure, the manager is left with an advanced counter and a
non-trivially-recoverable state — caller has to know to call `reset()`.
**Why it wasn't fixed tonight:** edge case. The `register` driver only
exposes pre-validated URIs to the manager; the failure mode is essentially
"someone constructed `RegistrationManager` with bogus data." Real-world
callers don't trip it.
**Suggested shape:** compute next values into locals, build with them,
commit (`self.cseq = next; self.nc = next_nc`) only on `Ok(...)`.

## D3 — Drop the redundant `check_asterisk_available()` probe under `#[ignore]`

**File:** `crates/rsiprtp/tests/asterisk_integration.rs:76-79, 138-141, ...`
**Symptom:** with `#[ignore]` (added tonight by F4), the test only runs via
explicit `--include-ignored`. The 2-second OPTIONS probe at the top is now
double-protection; harmless but adds dead time.
**Why it wasn't fixed tonight:** R1 marked it optional and the cost of the
probe is real but small (2s × 3 tests on a manual run).
**Suggested action:** delete the early-return blocks; let the test fail
loud with the timeout if Asterisk isn't actually up.

# Lower-priority follow-ups (R1 questions, not findings)

## Q1 — `dyn Transport` in the registration driver

**Context:** the new `RegistrationManager::register` / `unregister` driver
takes `&UdpTransport`. To support TCP/TLS REGISTER, the driver would need
to abstract over the transport. The codebase already has a `Transport` enum;
threading it through is a small enhancement with a clear shape but
non-trivial async ergonomics (object-safe trait vs enum dispatch).
**Status:** future enhancement, not urgent. Most SIP REGISTER traffic is
UDP in practice; TLS-REGISTER comes up for some corporate deployments and
WebRTC bridges.

## Q2 — `cseq` starts at 1, first request goes out as CSeq=2

**Symptom:** `manager.cseq` is initialised to 1 and incremented to 2 before
the first request is built (registration.rs:188). So packet captures show
the very first REGISTER as `CSeq: 2 REGISTER`, which is unusual.
**Status:** RFC 3261 §10.2 doesn't mandate a specific starting value, only
monotonicity within the registration's call leg. Strictly compliant.
Cosmetic only — but if a future debugger spends time wondering "where did
CSeq=1 go," that's the answer. Could be cleaned up to start at 0 and
post-increment, so the first wire request is CSeq=1.
