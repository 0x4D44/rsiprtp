## Launcher design (`fuzz_overnight_parallel.ps1`)

- Spawns one `pwsh -File fuzz_overnight.ps1 -Profile <p> ...` child per
  profile via `Start-Process` with per-profile stdout/stderr redirected
  to `wrk_journals/fuzz_logs/parallel_<profile>.events.log`/`.err`.
- Tail-and-tag loop reads new bytes from each child's log every 500ms
  and re-emits each line on the launcher's stdout prefixed with
  `[<profile>] ` plus appends to a combined event log at
  `wrk_journals/fuzz_logs/parallel_combined.events.log`.
- Exits when all children exit, emitting `[parallel] DONE_ALL ...`.
- Sets the MSVC ASAN PATH prefix once so children inherit it (children
  also self-prepend, harmless).

The 5 children share `fuzz/corpus/<target>/`, `fuzz/artifacts/<target>/`,
and `wrk_journals/fuzz_{logs,triage}/`. **Active-profile targets do not
overlap** (verified — sip-legacy holds the only conflicts and is excluded),
so no contention on shared dirs. Per-run log filenames are
`run_NNN_<target>.log` — uniqueness preserved by target name across
profiles.

## Supervisor protocol

Per `feedback_supervised_subagent_waits.md`: supervisor drives the long
wait directly; sub-agents only handle bounded code work.

**Wait driver:** `Monitor` running `tail -F` on the combined event log
with a regex filter for terminal/significant events. Each line becomes a
chat notification.

**On HEARTBEAT (every 5 min × 5 children = ~1/min):** record cumulative
stats every ~30 min. Don't journal every heartbeat.

**On CLEAN:** one-line note in journal that target completed its slice.

**On CRASH:** the crashing profile's child wrapper pauses **its own
rotation** and waits for `RESUME` in the triage slot. The other 4
profiles continue uninterrupted. Then:
1. Read the triage `fuzz.log` directly (small file).
2. Dispatch **Explore** agent: scope = read the crash log + locate the
   failing parser/codec site, report shape and reproducer bytes.
3. Dispatch **Plan** (or general-purpose) agent: scope = propose a
   minimal fix + regression test, given the failure shape.
4. Dispatch **general-purpose** agent: scope = implement the fix + add
   the unit test.
5. Read the diff inline, decide if `code-reviewer` skill is warranted.
6. Markup this journal with the crash subsection.
7. Touch `wrk_journals/fuzz_triage/<slot>/RESUME` to release that
   wrapper.

### 22:30 UTC (T+34 min) — CRASH on sip-diff (run 2)

```
[sip-diff] CRASH run=2 target=sip_message_parse_diff exit=1 queue=crash_002
```

**Different divergence kind:** `ours accepted but rsip rejected` (panic
site `parser_diff_oracle/mod.rs:1278`, the `(Err, Ok)` arm). Confirms
the rebuild picked up my source change for crash-1 (line numbers shifted
by my +13 lines of skip 6 doc/rule).

**Shape:** rsip's tokenizer choked on `SIP/2.0 223 523223 523 524\n\n
524\n\n` as the head section. Our parser surfaced
`status=223, headers=[], body=[' ', '5', '2', '4', \r, \n, \r, \n]`
(8 bytes ending in CRLFCRLF). Family: same LFLF-leniency asymmetry as
existing skip #2 (M11 finding #17 / `lflf_separator_only_rsip_rejects_we_accept`).

**Why existing skip #2 missed it:** existing condition was
`!has_crlfcrlf && has_lflf && rsip_err.contains("Tokenizer error")`.
But this input has CRLFCRLF in the body region (the trailing `\r\n\r\n`
after ` 524`), so `!has_crlfcrlf` is false. The asymmetry root cause is
LFLF-in-head, not absence-of-CRLFCRLF — the `!has_crlfcrlf` was an
over-narrow proxy.

**First fix attempt:** scoped LFLF check to `bytes[..idx]` where `idx =
find_header_separator(bytes)` — i.e. "LFLF in the head section only".
Self-defeating: when our framer's LFLF-fallback PICKS an LFLF as the
boundary, that LFLF is removed from the head, so the scoped check
returns false on exactly the inputs it should mask.

### 22:33 UTC (T+37 min) — CRASH on sip-diff (run 4)

```
[sip-diff] CRASH run=4 target=sip_message_parse_diff exit=1 queue=crash_004
```

Same family, but LFLF-relaxation **didn't fire** — input has bare LFs
that aren't contiguous (e.g. `\n\r\n` or `\nX\n`), so
`bytes.windows(2).any(|w| w == b"\n\n")` returned false. rsip's
tokenizer chokes on ANY bare LF, not just LFLF — the LFLF check was
still too narrow.

**Final-final fix:** broadened the skip from "LFLF anywhere" to "any
bare LF anywhere" (LF not preceded by CR):

```rust
let has_bare_lf = bytes.iter().enumerate().any(|(i, &b)| {
    b == 0x0A && (i == 0 || bytes[i - 1] != 0x0D)
});
if has_bare_lf && rsip_err.contains("Tokenizer error") {
    return;
}
```

This is the root-cause condition: rsip's tokenizer is built for strict
CRLF input; a bare LF anywhere breaks it. Our parser tolerates bare LF
in multiple framing positions. The asymmetry family is "bare-LF
leniency", not specifically "LFLF leniency".

`cargo test -p rsiprtp --test parser_diff` → still 62 passed.

**Resume:** 22:34:26 UTC.

**Lessons learned (3 successive skip refinements in 5 min):**
1. The original `!has_crlfcrlf` guard was an over-narrow proxy for
   "LFLF was used as the separator".
2. Scoping the LFLF check to the head section is self-defeating when
   our framer's LFLF-fallback consumes the LFLF as the boundary.
3. The bare-LF check is the principled invariant — rsip's tokenizer
   narrowness is "no bare LF tolerated", not "no LFLF tolerated".

I should have started with #3, but the (Err, Ok) arm's two existing
skips both used substring checks (`\n\n` and unusual-byte), which
biased my first attempt toward a substring extension.

## Verdict

Eight hours of saturated parallel fuzzing across **27 of 28 active
targets** (sip-diff killed early). **Zero crashes** on the 4 remaining
profiles across all 64 slices. The differential target (sip-diff)
surfaced **6 distinct (Err, Ok) and (Ok, Err) divergences** in its
first 43 minutes, all in known M11-asymmetry families:

1. **Header/body separator `\n\r\n`** — RFC 3261 §7.1 strict
   CRLFCRLF; rsip lenient. Pinned + skipped.
2. **Bare LF anywhere in head** — rsip's tokenizer is CRLF-strict;
   our framer's LFLF fallback makes us tolerant. Skip refined twice
   from the original LFLF-only check (run-#6 finding) to a
   bare-LF-anywhere check after fuzz iterated through three adjacent
   shapes within ~7 min.
3. **Invalid UTF-8 in header values** — rsip enforces UTF-8 at parse
   time; we don't (RFC 3261 §7.3 specifies UTF-8 by default but
   real-world deployments carry non-UTF-8 freely). New skip added.
4. **4-digit status code `0233`** — rsip rejects per §7.2 `3DIGIT`;
   our parser accepts as 233. **This is a real parser leniency** that
   may want fixing rather than skipping. Documented as follow-up #1.

The remaining 4 profiles (sip / rtp / media / ice) ran to budget
without a single panic. RSS held under 1 GB per child throughout.
Total compute: ~32h fuzz-time in 8h wall-clock — clean 4× speedup
from the inter-profile fan-out.

This locks in the post-`09e18a3` SDP normalization, the post-`1c4583a`
SIP CTL fix, the recent S1–S3 Via roundtrip oracle work, and all
non-SIP fuzz targets at the 8h fuzz bar — including
**rtp_packet, rtcp_*, srtp_unprotect, srtcp_unprotect, ice_candidate,
g711/g722/opus_decode, jitter_push, dtmf_event, srtp_sdes** which had
**not been campaigned in any of the previous three overnight runs**.
This is the broadest non-SIP coverage to date.

## Follow-ups (no action overnight; for daylight review)

1. **Investigate parser leniency for multi-digit status codes**
   (`crates/rsiprtp/src/sip/parser/`). RFC 3261 §7.2 mandates exactly
   3 digits. Reproducer: `b"SIP/2.0 0233 25%\r\n\r\n"` → ours
   accepts as 233; rsip rejects (Tokenizer error). Decide: tighten
   the parser to reject 4+-digit codes, OR pin as documented
   asymmetry.

2. **Review the 6 sip-diff triage slots** under
   `wrk_journals/fuzz_triage/crash_00{1..6}_sip_message_parse_diff/`
   for any panic shape NOT covered by the 3 oracle skips I added
   tonight. If all 6 fit the documented families (LF-CR-LF, bare-LF,
   invalid-UTF-8, 4-digit-status), the slots can be deleted.

3. **Consider shortening slice cap for saturated targets** in
   `fuzz_overnight.ps1` — `rtcp_header` (cov 38, 261k exec/s),

   exec/s) and similar. A `-SliceSeconds 600` for a "saturated"
