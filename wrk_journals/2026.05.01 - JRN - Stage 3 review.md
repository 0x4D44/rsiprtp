## Fix 1 — `RegistrationManager::needs_refresh`: SOUND

### Semantics on `None`

`expires_at.checked_sub(window)` returns `None` only when the refresh
threshold would land before the platform epoch. On Windows the `Instant`
epoch is boot, so underflow means *the refresh threshold is earlier than
boot*. By definition `Instant::now() >= boot`, therefore `now >= refresh_at`
is true. Returning `true` is semantically correct — refresh now.

The implementer's reasoning is right: when underflow happens, the registration
window is so wide that the refresh point is logically already in the past,
hence "yes, refresh."

### Other panic sites in the chain

Grep for `Instant - Duration` in the session crate and call chain:

- `crates/rsiprtp/src/session/registration.rs:320` — `Instant::now() + Duration`
  (addition, no underflow). Safe.
- `crates/rsiprtp/src/rtp/session.rs:855, 1028, 1274` — three
  `Instant::now() - Duration::from_millis(20|200|200)` calls, all under
  `#[test]`. Latent only on machines with sub-200ms uptime, which is not
  a realistic CI scenario. Not in scope for this stage; flag for a future
  audit pass if test flakes appear.
- `crates/rsiprtp/src/session/bitrate_bridge.rs` uses `t0 + Duration` and
  `Instant::checked_duration_since` correctly throughout — no bare
  subtractions.

No other production-path `Instant - Duration` panics remain in the
registration call chain.

### Test coverage of the new code path

The existing `test_needs_refresh` constructs `expires_at = Instant::now() +
10s`. On a machine with > 710s uptime (anything past ~12 minutes after
boot), `checked_sub` returns `Some` and the test exercises the comparison
arm, not the `None` arm. So:

- The fix is correct.
- The test does not *enforce* the underflow path on developer machines.
- The test passes on a freshly booted Windows runner (the `None` arm) and
  on long-uptime machines (the `Some` arm) — both arms return `true`, so
  the assertion holds unconditionally. That's why it works on CI.

**Suggested follow-up:** add a regression test that constructs an
`expires_at` *equal to* `Instant::now()` (or close to it) so that
`expires_at.checked_sub(Duration::from_secs(720))` is guaranteed to take
the `None` arm regardless of uptime. Not blocking; the production fix
is sound either way.

### Clippy / atomicity

Clippy with `-D warnings` is clean on `cargo clippy -p rsiprtp --all-targets`.
The targeted tests `test_needs_refresh` and `test_needs_refresh_far_future`
both pass.

## Fix 2 — `generate_tag()`: SOUND (Yes-but)

### XOR doesn't help — confirmed

Original:

```rust
let duration = SystemTime::now().duration_since(UNIX_EPOCH).unwrap_or_default();
duration.as_nanos() as u64 ^ (duration.as_secs() << 32)
```

Both operands derive from a single `SystemTime::now()`. If two adjacent
calls observe identical `now()` values (as happens on macOS where
`SystemTime` ticks coarser than a tight loop), `nanos` and `secs << 32`
both collide together → XOR collides identically. Implementer's
diagnosis is correct.

### `rand::thread_rng()` correctness

`rand 0.8::thread_rng()` returns a thread-local `ThreadRng`, which is a
`ReseedingRng<ChaCha12Core, OsRng>`: seeded from `OsRng` (which on
Windows uses `BCryptGenRandom`, on Linux `getrandom(2)`, on macOS
`getentropy`) and reseeded periodically. Output is uncorrelated across
calls; not dependent on the wall clock. `rand` is already a workspace
dep (`Cargo.toml:57` and `crates/rsiprtp/Cargo.toml:45`) — used in
`sip/auth.rs:381`, `ice/agent.rs`, `ice/stun.rs`, `srtp/sdes.rs`. No new
dependency.

Birthday-paradox math: P(collision over 100 tags from 64-bit space) ≈
2.7e-16. Test threshold is "more than 90 unique" out of 100 — overwhelmingly
satisfied.

### Cross-process collisions

`thread_rng()` is per-thread but seeded from OS entropy. Two concurrent
`rsiprtp` processes on the same machine each get an independent seed
from the OS CSPRNG. Cross-process collision probability is identical to
the single-process birthday math (~2^-32 over 2^32 tags between two
processes). RFC 3261 dialog-matching requires uniqueness within a
dialog, which is well within tolerance.

### Companion test cleanup — incomplete

`crates/rsiprtp/src/sip/message.rs:2084` still has
`std::thread::sleep(std::time::Duration::from_millis(1))` between two
`generate_tag()` calls. The implementer notes this in their "Risks /
follow-ups" — they consciously left it. The sleep is now misleading: a
future maintainer reading `test_generate_tag_unique` will assume the
sleep matters. This is the "Yes-but":

**Suggested follow-up (not blocking):** delete the `thread::sleep` line.
It's three lines of cleanup and removes a tripwire. The companion
`tests/rfc3261.rs:765` `test_generate_tag_unique` (the 100-tag birthday
test) doesn't have this issue.

### Other ID-generation functions — not affected

- `generate_branch` (`message.rs:1044`) → `uuid::Uuid::new_v4()`. UUID v4
  with the `v4` feature uses `getrandom`, which is OS CSPRNG. Sound.
- `generate_call_id` (`message.rs:1061`) → same `Uuid::new_v4()`. Sound.
- `generate_cnonce` (`auth.rs:380`) → `rand::thread_rng().gen()` for 16
  bytes. Sound.
- `ice/stun.rs:296` and `ice/turn.rs:965` `generate_transaction_id` —
  worth a separate audit but out of scope for Stage 3.

The bug was confined to `generate_tag()` and the now-deleted `rand_u64`
helper. Fix scope is correct.
