# Background

Previous session merged 29 commits onto `main` (parser rewrite + 15
pinned rsip-side spec deficiencies). The wrap-up flagged two run-#6
findings as needing fresh attention rather than another autonomous loop:

1. **#2 — rsip accepts CRLF/controls inside header values.**
   Sample: `("6.~", "5\r-\n!S-*\u{1}\u{c}~~~*\u{b}* S54")`.
   Per RFC 3261 §7.3 CRLF terminates a header, so our rejection is
   correct. Likely pin candidate (rsip narrowness — but the principled
   heuristic from pin #14 should already cover non-printables in the
   header section unless this is a different path).
2. **#3 — ours accepts body with sustained 0xb8/0xf8 bytes; rsip rejects.**
   Body-side, principled heuristic doesn't scope here. Could be our bug
   (body framing too lax) or rsip rejecting for header reasons. Needs
   investigation.

The bigger strategic question (mentioned at session start): is
differential fuzzing bottoming out? If both #2 and #3 turn out to be
rsip-narrowness pins, the differential signal is decaying; the next
high-leverage move is a second oracle (round-trip parse→serialize→parse,
structural invariants) rather than another long cycle of the same one.

## 2026-05-04 — triage outcomes

### Finding #2 (CRLF/controls in header value)

Reproduced with wire `b"SIP/2.0 200 OK\r\n6.~: 5\r-\n!S-*\x01\x0c~~~*\x0b* S54\r\n\r\n"`:

- rsip ACCEPTS: parses one header `("6.~", "5\r-\n!S-*\u{1}\u{c}~~~*\u{b}* S54")`
  — its tokenizer absorbs the bare CR and bare LF mid-value.
- ours REJECTS: `Invalid header: missing ':' in header: !S-*\u{1}\u{c}~~~*\u{b}* S54`
  — `str::lines()` splits at the bare `\n`, the orphan continuation
  starts with `!` (no leading whitespace, so not a fold), and parsing
  fails when no `:` appears on that line.

Mechanism = identical to pin #13 (`header_missing_colon_rsip_accepts_we_reject`):
rsip's tokenizer absorbs bare LF, ours surfaces a downstream
"missing ':'" error. Already covered by the oracle's existing
`"missing ':'"` skip in the `(Ok, Err)` arm — the fuzzer panic is
*not* what triggered this finding being backlogged; it was tagged as
a "pin candidate" meaning a parser_diff regression pin would be
worthwhile but no oracle work is required.

**Action:** add pin in `parser_diff.rs` for the header-value position
(distinct from #13's reason-phrase position). Future-proof against
rsip fixing one position but not the other.

### Finding #3 (body high-bit bytes — actually LFLF-separator asymmetry)

Initial four probes all showed both parsers accepting. Broader
probing revealed the real trigger: input with `\n\n` separator (no
`\r\n\r\n` anywhere), e.g. `b"SIP/2.0 200 OK\n\n\xb8\xb8\xb8"`:

- ours ACCEPTS — `find_separator` falls back to LFLF (deliberate M2-A
  leniency, pinned by `framing::test_split_message_lf_only_fallback`),
  splits the message, surfaces the high-bit bytes as body.
- rsip REJECTS with Tokenizer error — rsip doesn't recognize LFLF as
  a separator, so it treats the whole input as a header section and
  tokenization fails on the high-bit bytes.

The journal's hypothesis ("rsip looks back at headers for some
reason") was off. The actual mechanism is **rsip lacking our LFLF
fallback**, not anything about body bytes specifically. The high-bit
bytes are incidental — any body bytes that don't tokenize as header
syntax (so non-token, non-LWS) would trigger the same failure mode.

The principled tokenizer-narrowness heuristic (pin #14) misses this
case because:
- It scopes `header_section` via `find_header_separator` which
  *also* falls back to LFLF.
- Result: header_section = `b"SIP/2.0 200 OK"` (pure ASCII).
- `has_unusual_byte == false` → skip declines → panic.

This is the symmetric `(Err, Ok)` analogue of pin #6
(`body_starts_with_header_like_line_rsip_misinterprets`), which
covers the `(Ok, Ok)` arm of the same LFLF asymmetry. Pin #6 fires
when both parsers extract *some* structure but disagree on what's
header vs body; this case fires when rsip's structure-extraction
fails entirely.

**Action:**
1. New skip in oracle `(Err, Ok)` arm: input has `\n\n` but lacks
   `\r\n\r\n`, AND rsip's error is Tokenizer-class.
2. New pin test `lflf_separator_only_rsip_rejects_we_accept` in
   `parser_diff.rs`.

### Strategic note on bottoming-out

Both findings turned out to be **rsip-narrowness pins**, not bugs in
our parser. After 17 pinned divergences, the differential signal is
heavily skewed toward rsip's quirks. Worth seriously considering a
second oracle (round-trip parse → serialize → parse must be a fixed
point) before committing more wall-clock to the same fuzz target.

### What's next

Two pinned findings landed. The strategic question — whether to
launch another long differential-fuzz run or pivot to a second
oracle (round-trip / structural invariants) — is still open and
deserves discussion. My recommendation: pivot. Reasoning:

1. Pin #14 introduced a principled tokenizer-narrowness skip that
   collapses a whole class of `(Err, Ok)` divergences. After that
   skip plus tonight's #17 LFLF skip, the remaining `(Err, Ok)`
   surface area against rsip is small.
2. `(Ok, Err)` similarly is mostly enumerated by the four
   error-string skips (invalid SIP version, status code out of
   range, invalid Request-URI, missing ':').
3. The differential oracle structurally cannot find bugs that
   *both* parsers agree on wrongly. A round-trip oracle (parse →
   serialize → parse must equal the first parse) would.

If we run the 8h campaign anyway, my prediction is that the
yield is mostly more rsip-side narrowness pins or `(Ok, Ok)`
DiffMessage divergences caused by neutral-representation gaps,
not bugs in our parser. Worth doing once for due diligence; but
not before adding a round-trip oracle.
