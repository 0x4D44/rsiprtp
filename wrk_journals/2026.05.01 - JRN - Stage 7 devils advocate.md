# Stage 7 — Visibility tightening: devil's advocate review

**Date:** 2026-05-01
**Author:** Claude (devil's advocate, separate from implementer)
**Reviewing:** `wrk_journals/2026.05.01 - JRN - Stage 7 visibility.md` and uncommitted Stage 7 diff

## Methodology

Walked the full removed-items list from the public-api.txt diff, classified
each as either:

- **Duplicate** — the type is still reachable via a parent-module
  `pub use`, so removing the secondary `module::sub::Item` path costs
  nothing real (e.g., `dialog::invite::InviteDialog` removed but
  `dialog::InviteDialog` still public).
- **Genuine loss** — the symbol is gone from the API entirely, or the
  only remaining path is one no consumer would naturally find.

After cross-checking every removed `pub struct`, `pub enum`, `pub fn`,
and `pub const` against the new `mod.rs` re-exports, the vast majority
(>95%) are duplicates. The implementer was correct that the
shrink came mostly from collapsing the trait-impl listing duplication
under `media::codec::{g711,g722,opus}`.

## Genuinely removed items

These are the ONLY items where a removed-line in `public-api.txt` does
NOT have a corresponding parent-module re-export:

1. **`core::random_u16`/`u32`/`u64`** — generic RNG helpers.
   No callers outside the crate. Trivially reimplementable. **Correct
   to demote.**
2. **`media::codec::g722::silence_frame()`** — gated to `#[cfg(test)]
   pub(crate)`. Only the G.711 silence helpers are exposed at
   `media::silence_byte`/`silence_frame`. The G.722 version was a
   one-liner returning `vec![0u8; 160]`. **Asymmetric but harmless** —
   if a caller needs G.722 silence they can `vec![0u8; 160]`
   themselves. Worth a follow-up: either expose it for symmetry with
   G.711, or remove the G.711 helpers too. Not a blocker.
3. **`media::codec::opus::OPUS_FRAME_MS`** — deleted (zero callers).
   **Correct.**
4. **`media::codec::opus::OPUS_MAX_PACKET_SIZE`** — `pub(crate)`. This
   IS a useful sizing constant for anyone allocating decode buffers
   when calling `OpusCodec::decode` with custom packet handling. But
   the codec API takes `&[u8]` so the consumer never needs to size
   their own buffer for normal use. **Borderline — leave demoted.**
5. **`session::hold::sdp_direction_attribute` / `parse_sdp_direction`** —
   `#[cfg(test)] pub(crate)`. SDP `a=sendrecv`/`a=recvonly` round-trip
   helpers. Trivially reimplementable. Correct to demote, BUT:
   the analogous logic also lives inside `MediaDirection::to_sdp` /
   `from_sdp` (or wherever it is on `MediaDirection` — not verified).
   If `MediaDirection` already has those methods, the free helpers
   were redundant and demotion is fine.
6. **`session::transfer::build_refer_to_blind`/`build_refer_to_attended`/
   `build_referred_by`** — `#[cfg(test)] pub(crate)`. These
   build raw `Refer-To: <…>` and `Referred-By: <…>` header strings.
   Trivially reimplementable from the underlying `ReferTo` /
   `ReplacesHeader` types (which remain `pub`). **Correct to demote.**

## Three "deliberately flagged leaks"

### a. `rtp::ReceiverState` with all-pub fields

Returned by `RtpSession::receiver_stats(&self) -> &ReceiverState`. The
fields (`base_seq`, `expected_packets`, `jitter`, `packets_lost`,
`packets_received`, `last_arrival`, etc.) are exactly the RFC 3550 §A.3
RTCP RR receiver state machine. Any consumer constructing custom RR
packets or rendering call-quality dashboards needs them. The fact
that they're public fields rather than accessor methods is a real
semver risk for the next refactor, but exposing them is correct —
this is RFC 3550 telemetry. **Implementer's call to leave `pub` is
correct.** Future tightening (private fields + accessor methods) is
a real API change deferred to a later release.

### b. `rtp::session::CongestionState`/`CongestionController`

These are AIMD primitives documented in the bitrate-bridge HLD as the
hand-rolled congestion-control core. Only directly exercised by tests
externally, but they're documented building blocks. The path is
unfortunate (`rtp::session::CongestionController`, with
`rtp::session` still `pub mod`) — a non-trivial WebRTC-style
consumer who wants to drive congestion control into a non-Opus codec
will find them. **Leave `pub`.**

### c. `session::call` kept `pub`

Implementer says "could be tightened by re-exporting `MediaSession`
through `session/mod.rs`." Verified: `session/mod.rs` already does
`pub use call::{Call, CallConfig, CallDirection, CallEndReason,
CallEvent, CallId, CallState, Dialog, MediaSession}`. So
`session::MediaSession` is reachable; `session::call::MediaSession` is
the only path the integration test in `tests/rtcp_drives_bridge.rs`
uses. The implementer's bikeshedding note is correct: this can be
tightened in a one-line follow-up by editing the test to use
`session::MediaSession`. **Worth doing as a clean-up; not a blocker.**

## Verdict

**Ship it as-is.**

The 25% reduction is almost entirely path-deduplication: collapsing
`media::codec::g711::G711Codec` (plus its 30 trait-impl methods) when
the same type is exposed at `media::G711Codec` is pure noise removal.
The same applies to the dialog/transaction/sip/transport/ice
demotions — every type a consumer would reasonably want has a parent
re-export.

Genuine losses are six items: three RNG helpers, two SDP-string
builders, three Refer-To header builders, one Opus constant, and one
G.722 silence helper. None of these are RFC-defined surfaces; all
are trivially reimplementable; none are mentioned in the README,
prelude, or any example.

### Required before commit: nothing.

### Optional follow-ups (do later, not now):

- **Symmetry**: either expose `G722Codec::silence_frame()` for parity
  with G.711, or move both G.711 helpers to `pub(crate)` (they're a
  one-liner). My preference: leave G.711 silence helpers pub and
  accept the asymmetry — gabby uses them.
- **`session::call`**: change
  `tests/rtcp_drives_bridge.rs` to import via `session::MediaSession`
  and demote `pub mod call` to `pub(crate) mod call`. One-line
  cleanup to remove the implementer's flagged bikeshed.
- **`ReceiverState` field privacy**: real semver-affecting change,
  rightly deferred. File a tracking issue for the next minor bump.

### Items the implementer left `pub` that I would also leave pub:

- `CongestionState`/`CongestionController` — documented bitrate-
  bridge primitives.
- `ReceiverState` with `pub` fields — RFC 3550 telemetry shape.
- `srtp::dtls::*` — fuzz target depends on it; module-doc lists
  `build_use_srtp_extension`/`extract_srtp_keys` as documented API.
- `transaction::client`, `transaction::server`, `transaction::timer`
  — these are RFC 3261 state-machine primitives that a serious
  Sans-IO consumer will want. The crate's headline value-add.
- `rtp::rtcp`, `rtp::session` — RFC 3550 primitives.
- `media::jitter`, `media::mixer`, `media::wav` — well-defined
  audio-processing primitives.

## Appendix — what the implementer got right that's not obvious

1. **`core::util` demotion**: `random_u*` were trivially reimplementable
   and pulled in nothing. The crate's `core::Error`/`Result`/
   `StackConfig`/`CodecConfig` are still `pub use`'d from
   `core::config` and `core::error`. No leak.
2. **Transport doc fix**: `transport::resolver` doc-comment was
   updated from `use rsiprtp::transport::resolver::{...}` to
   `use rsiprtp::transport::{...}`. Without this fix the demotion
   would have broken doc-tests.
3. **Codec re-export deduplication**: Removing the redundant
   `pub use g711::*` from `codec/mod.rs` (which `media/mod.rs` already
   forwards through) was the single biggest line-count win in
   `public-api.txt` and was free from a consumer perspective —
   `media::G711Codec` etc. all still resolve.
