## 1. The `dtls` feature is a placeholder shell — wired to nothing

**File evidence:**
- `crates/rsiprtp/Cargo.toml:28-30,84` declares `[features] dtls = ["openssl"]`
  with `openssl = { version = "0.10", optional = true }`.
- `rg openssl crates/rsiprtp/src/` returns **two doc-comment hits**, both inside
  `crates/rsiprtp/src/srtp/dtls.rs:10` and `:32`. No `use openssl::…`, no
  `extern crate openssl`, nothing.
- `rg 'feature\s*=\s*"dtls"' crates/rsiprtp/src/` returns **zero**. There is
  not a single `#[cfg(feature = "dtls")]` anywhere.

The implementer was *correct* that `parse_use_srtp_extension` is not gated and
the fuzz target compiles fine without the feature. They were correct to drop
`features = ["dtls"]` from `fuzz/Cargo.toml`. But the **deeper** issue they
walked past is this: the `dtls` feature *only* flips the `openssl` optional
dep on. With openssl installed, `cargo build --features dtls` will compile
openssl-sys, link `libcrypto`/`libssl`, and produce **dead native code**. The
feature does not make any real DTLS code reachable.

What this means in practice:

- HLD point 3 ("textbook legitimate use of a feature flag") is false against
  the current source. The flag gates a heavyweight C native dep that is
  imported nowhere. It is opt-in cost with zero opt-in benefit.
- README and the `lib.rs` doc claim "DTLS-SRTP gated by `dtls` feature" — at
  `crates/rsiprtp/src/lib.rs:39` — is **misleading**. The dtls feature does
  not gate any DTLS-SRTP code; it gates an unused dep. A user who reads "I'll
  enable `dtls` to get DTLS-SRTP" will get nothing extra at runtime and a
  build-time openssl requirement.
- `package.metadata.docs.rs` sets `all-features = true`. docs.rs Linux
  builders have system openssl, so this won't fail — but it will compile
  openssl-sys for nothing on the public docs build.

**What likely happens in CI on Linux in 8 hours:** `cargo build --features
dtls` will succeed (Ubuntu has libssl-dev). It will also compile openssl-sys
unnecessarily and produce a `unused-but-not-quite-unused` artifact. CI will
not catch the structural emptiness because there's no test or assertion that
exercises `--features dtls`.

**What likely happens at publish:** `cargo publish` ships a feature flag whose
public meaning is "pull in openssl for no reason." Once shipped this is
public API and adding/removing the flag becomes a semver decision.

**Concrete recommendation:** either delete the `dtls` feature entirely until
real DTLS code lands, or add a smoke `#[cfg(feature = "dtls")]` test that
verifies the feature is *meant* to be empty (so the absence is documented and
intentional).

## 2. CI does not validate gate 2 of the HLD validation matrix

`.github/workflows/ci.yml` runs:

- fmt
- clippy `--workspace --exclude gabby --all-targets`
- build/test on three OSes
- doc build with `RUSTDOCFLAGS=-D warnings`
- MSRV `cargo build --workspace --exclude gabby --locked` on Rust 1.75
- audit, deny

It **does not** run:
- `cargo build --workspace --exclude gabby --features dtls` (HLD gate 2)
- `cargo build --manifest-path fuzz/Cargo.toml` (HLD gate 9)
- `cargo publish --dry-run -p rsiprtp` (HLD gate 10 — the real exit gate)

So the implementer's note "CI Linux runners will pass this gate" is **assumed,
not enforced**. The `dtls` feature plumbing could break tomorrow and CI would
go green. The fuzz workspace could rot the same way.

The journal explicitly admits gates 2 and 4 didn't fully complete on Windows
and that they're treated as environmental. The implementer is right that they
*might* pass — but no one will discover it if they don't, because nothing
runs them.

**Concrete recommendation:** add three jobs to `.github/workflows/ci.yml`
before the publish PR, even if some are allowed to soft-fail.

## 3. README and `deny.toml` still reference deleted crate names

- `README.md:111` says `rsiprtp is a workspace of focused crates layered from
  foundations (\`rsiprtp-core\`) up through transport, media, transactions,
  dialogs, and finally a session layer`. This is the *old* architecture
  described as if still true. The collapse has been done; the README has not
  been updated.
- `deny.toml:27,30` references `rsiprtp-transport tests` (twice) in the
  audit-trail comments for the RSA Marvin Attack ignore. That crate no longer
  exists.
- `deny.toml:33-37` says "the project's single-facade publishing plan uses
  path-only workspace deps in the `rsiprtp` facade crate, which cargo-deny
  flags as wildcards" — this rationale is **stale**. After the collapse the
  `rsiprtp` crate has zero path deps; it has only versioned crates.io deps.
  `wildcards = "warn"` could go back to `"deny"` and `allow-wildcard-paths`
  is now meaningless.

These are documentation rot, not build breakage, but they ship in the
published tarball (README is shipped via `readme = "../../README.md"`). New
users reading the published README will be confused.

**Concrete recommendation:** README rewrite to describe single-crate
architecture with module structure; `deny.toml` audit-trail edits.

## 4. The empty `crates/rsiprtp-transport/` directory is in the working tree

`ls -la crates/` shows `rsiprtp-transport` still present. Git's index is
clean (the deletes are staged). The journal says "Functionally inert: Cargo
doesn't reference it, git doesn't track it." This is **mostly true** but with
caveats:

- `cargo` walks the workspace based on `[workspace] members`, which now
  excludes `rsiprtp-transport`. Cargo will not pick it up.
- However, `cargo package --list -p rsiprtp` was run with `--allow-dirty` and
  produces a clean list (verified — see finding 5). So this won't ship.
- The *actual* risk: a future automation that `git mv`s files or
  hand-crafts paths could trip over the directory's existence and a
  developer pulling fresh on Windows might find their `cargo build` blocked
  by the same Windows file lock. Annoyance, not bug.

The journal says "Will retry from a clean shell post-commit; if persistent,
schedule for the next reboot." This is fine, but the commit should not be
made with this dangling — it gives the working tree two states: "git says
clean," "filesystem says transport-still-here." Confusion magnet.

**Concrete recommendation:** reboot, `rmdir`, then commit. Don't ship the
divergence.

## 6. `cargo public-api` baseline regenerated, but the diff hides regressions

`crates/rsiprtp/public-api.txt` went from 86 lines (old facade re-exports
only) to **25,061 lines** (every internal item walked). The HLD predicted
this and called it "a large but mechanical diff." It is *both* mechanical
and a hiding place.

What's confirmed preserved (sampled): all prelude items
(`CallManager`, `RegistrationManager`, `RtpSession`, `JitterBuffer`,
`UdpTransport`, `Method`, `SipMessage`, etc.) are present in the new
baseline.

What's *added* (not necessarily a regression, just a surprise): the entire
internals of every former-internal-crate is now public API surface. Items
that were `pub` only to cross former crate boundaries are now visible to
downstream users. This is the visibility tightening problem the HLD flagged
as item 20 — and the implementer correctly flagged it as Stage 8 work.

What's *gone* relative to the old facade baseline: the line
`pub use rsiprtp::transport`, `pub use rsiprtp::media`, etc. were facade
re-exports. They have been replaced with `pub mod rsiprtp::transport`. From
a downstream user's perspective `use rsiprtp::transport::UdpTransport` still
works. But anyone who had relied on rustdoc's "Re-exports" section seeing the
modules listed will see a different layout.

**Real risk:** if someone previously imported `rsiprtp::sip::tests::…`
expecting it to be inaccessible (since `tests/` modules under each former
crate were typically `pub(crate)`-by-convention), the merged crate may now
expose them. The new baseline has 25k lines; nobody is going to read it.

**Concrete recommendation:** a quick `grep "pub.*tests::" public-api.txt`
audit before publish; flag any inadvertently exposed test modules.

**Concrete recommendation:** none in this stage; visibility tightening is
correctly scoped to Stage 8.

## 10. The discarded SIP fuzz seed corpus was never in git

The journal claims the corpus is "Reconstructable". Verified:
- `git log --all --full-history -- "**/corpus/**"` returns **nothing**.
- `git ls-tree -r 729261026f -- "crates/rsiprtp-sip/fuzz/"` (the original
  commit that added the SIP fuzz crate) lists `.gitignore`, `Cargo.toml`,
  `README.md`, `fuzz_targets/parse_sip_message.rs` — no `corpus/` directory.
- The corpus was **never tracked**. So losing it via `git rm -r` was
  impossible; what was lost was an untracked working-tree set of files.

This means the journal's framing — "ten hand-crafted RFC-3261 messages …
flagged for follow-up" — is asymmetric: those files existed only on the
implementer's disk, presumably from manual fuzzer runs. They are gone unless
they live in another working copy.

**Real impact:** very low. Fuzzers can rebuild a corpus quickly from
empty-corpus runs against the SIP parser. The journal note is correct that
this is "minor."

**Concrete recommendation:** when re-establishing the corpus, **commit it to
the repo this time** so future moves don't have the same risk.

## Severity ranking — which findings will bite us?

In descending order of how likely they are to bite us in CI or in production:

1. **The `dtls` feature is a placeholder shell** (finding 1). User-visible:
   anyone who sets `features = ["dtls"]` in their `Cargo.toml` will pull in
   openssl-sys for nothing. This becomes locked-in public API at publish.
   **High** likelihood of confusion; **medium** likelihood of CI surprise
   if any downstream consumer wires the feature.

2. **CI doesn't run gate 2 or gate 9** (finding 2). Any structural bug in
   the `dtls` feature wiring or the fuzz Cargo.toml will go undetected by
   CI. **Medium** likelihood of a silent rot.

3. **README and deny.toml audit-trail still describe the old workspace**
   (finding 3). The README ships in the tarball; users reading
   docs.rs/crates.io will see a description that doesn't match the actual
   crate. **High** likelihood of user confusion at first contact.

4. **`crates/rsiprtp-transport/` empty dir lingering on Windows** (finding
   4). Low impact but committing in this state is a code smell — we'd be
   shipping a working tree that disagrees with its own git index. **Low**
   likelihood of a build break, **medium** of contributor confusion.

5. **25k-line public API baseline hides accidental exposures** (finding 6).
   Nobody will diff it line-by-line; if `tests/` modules or internal types
   leaked into the public surface, we won't notice until Stage 8. **Low**
   likelihood of breakage, but if it does break it's a semver-affecting
   issue that we missed at the right time to catch it.

The implementer's gate 1 (`cargo build --workspace --exclude gabby`), gate 3
(test pass on Windows), `cargo package --list`, fuzz `cargo check`, gabby
`cargo check`, and `cargo deny` all pass cleanly. The collapse mechanics
themselves are correct. The risks are at the *edges* — features that are
nominally on but structurally empty, CI that doesn't run all gates,
documentation that hasn't caught up.
